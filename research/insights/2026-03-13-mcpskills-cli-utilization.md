## Candidate Systems Assessed

### 1. `/plugin-creator:skill-creator` (Local System)

**Path**: `plugins/plugin-creator/skills/skill-creator/SKILL.md`

**What it does**: Provides guidance for creating effective skills from scratch or from requirements. Documents the skill creation process, best practices, directory structure, YAML frontmatter, bundled resources (scripts, references, assets).

**Assessment**: Weak fit — No integration opportunity.

**Reason**: skill-creator is an educational guide to the skill creation process, not a tool orchestrator. It teaches manual steps (plan resources, determine location, initialize, edit, package, iterate). mcpskills-cli is an automation tool that requires:
1. A running MCP server accessible via HTTP
2. Bearer token authentication
3. One CLI invocation to generate the skill

The skill-creator workflow is process-oriented; mcpskills-cli would be a tool that *replaces* part of that process. But the research entry shows mcpskills-cli targets users who already have running MCP servers — a context the skill-creator guide doesn't address or assume.

**Gap**: skill-creator does not document MCP-based skill generation as an option. Adding mcpskills-cli would require new documentation sections covering MCP server prerequisites, endpoint discovery, credential management — outside the scope of this guide.

### 2. `/fastmcp-creator` Plugin (Local System)

**Path**: `plugins/fastmcp-creator/`

**What it does**: Teaches Claude how to build production-ready FastMCP v3 MCP servers with tools, resources, auth, deployment, testing.

**Assessment**: Weak fit — No integration opportunity.

**Reason**: fastmcp-creator outputs MCP servers (the resource mcpskills-cli *consumes*). The workflow would be:
1. User builds MCP server via `/fastmcp-creator`
2. User runs `mcpskills-cli` separately to convert server → skill

While this is a logical sequence, integration would require:
- fastmcp-creator to provision/run the MCP server (not its responsibility)
- mcpskills-cli invocation within the skill (out of scope for MCP server builder)

The two tools are **consecutive manual steps**, not integrated. The plugin correctly focuses on server creation; converting to skills is a follow-up the user initiates separately.

**Candidates examined**:

**Reason**: Research agents gather documentation and context *about* tools/frameworks. They produce reference files, not executable artifacts. mcpskills-cli is an artifact *generator* that requires:

Research agents could theoretically invoke mcpskills-cli if the user had provided an MCP server endpoint during research initialization, but:
1. None of the research processes assume or require MCP servers as input
2. No pattern exists for provisioning/managing running services during research
3. The output (SKILL.md) is generated by code, not researched from docs

## Workflow Context

mcpskills-cli sits at the intersection of two workflows:

```
User Workflow:
1. User builds MCP server via /fastmcp-creator
2. User runs: mcpskills-cli --url <endpoint> --token <token>
3. User gets SKILL.md + call script
4. User loads skill into Claude Code
```

**Current state**: Steps 1 and 3-4 are documented in the codebase. Step 2 (mcpskills-cli invocation) is a manual, external step users perform when they have a running MCP server.

**Why integration is weak**: Automating step 2 would require:
- Documentation change: fastmcp-creator would need to teach running the server and capturing its endpoint
- MCP server lifecycle management: fastmcp-creator would need to provision/start the server
- Responsibility creep: MCP builder becomes responsible for skill generation (out of scope)

None of these are natural extensions of the fastmcp-creator plugin's current responsibility (teach MCP server building).

## Conclusion

**mcpskills-cli is a well-designed, focused tool** for a specific use case: "I have an MCP server running; convert its tools to a Claude Code skill automatically."

**The codebase teaches adjacent activities**:
- How to build MCP servers (fastmcp-creator)
- How to create skills manually (skill-creator)
- How to research and document tools (research processes)

**But none of these are callers for mcpskills-cli** because:
1. They don't provision/manage running MCP servers
2. They don't have a documented need to automate the MCP→skill conversion
3. Integration would violate responsibility boundaries and require new prerequisites

**Suitable use case** (outside current codebase): A user who has built an MCP server with fastmcp-creator, started the server manually, and wants to convert it to a skill would run mcpskills-cli as a separate, intentional step. This is not an internal automation opportunity.
