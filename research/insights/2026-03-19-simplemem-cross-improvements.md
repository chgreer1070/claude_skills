## Improvement 1: Cross-session context recovery for agent workflows

**Source pattern**: "Multi-turn agent workflows: Agents can retain context across independent skill invocations or backlog-linked tasks without manual re-seeding" (Relevance section, item 1)
**Local system**: `plugins/python3-development/skills/implementation-manager/scripts/task_status_hook.py`
**Confidence**: Low
**Impact**: Medium
**Backlog**: Deferred -- confidence low: already tracked as #775 "Persistent structured session metadata for cross-session context recovery"

### Current state

The task_status_hook.py writes `LastActivity` timestamps and `Completed` timestamps per task. The `.claude/context/active-task-{session_id}.json` file tracks the currently active task but is deleted on SubagentStop. No mechanism persists session context (decisions made, files modified, discoveries) across independent skill invocations or across agent restarts.

### Target state

A session finalization step would persist structured observations (decisions, discoveries, key file paths) from the completed session into a queryable store. On subsequent session start, relevant prior observations would be injected into the agent's context, token-budgeted to avoid context window pressure.

### Measurable signal

A new file or database entry exists after session stop containing at least `source_session_id`, `observation_type`, and `content` fields. On session start, a `context` field in the start response contains retrieved prior observations.

## Improvement 2: Heuristic observation extraction from agent session events

**Source pattern**: "Observations are generated by heuristic analysis of the session event log: Decisions (agent choices extracted from assistant messages containing 'decided', 'chosen', 'will use'), Discoveries (facts from tool output that appear novel), Learnings (insights from assistant messages)" (Key Features > Observation Extraction section)
**Local system**: `plugins/python3-development/skills/implementation-manager/scripts/task_status_hook.py`
**Confidence**: Medium
**Impact**: Medium
**Backlog**: Deferred -- confidence medium: the local hook system records timestamps only, not event content. SimpleMem-Cross's extraction is concrete (keyword-based heuristic on event logs), but implementing it in this repo would require recording event content first, which is a prerequisite not yet in place. The gap is real but the path to implementation has an unstated dependency.

### Current state

The PostToolUse hook in task_status_hook.py updates only the `LastActivity` timestamp field. It does not record the content of tool calls, agent messages, or decisions made during a task. When a task completes, no structured observations are extracted or persisted. The hook receives `tool_name` and could receive `tool_input`/`tool_output` from the hook JSON, but discards everything except the event type and timestamp.

### Target state

The PostToolUse hook (or a companion hook) would record selected event data (tool name, abbreviated input/output) to a session event log. On SubagentStop, a finalization step would scan the event log and extract structured observations tagged as decision, discovery, or learning based on keyword heuristics matching patterns like "decided", "chosen", "will use", "learned", "realized".

### Measurable signal

After a task completes via SubagentStop, a file at `.claude/context/session-observations-{session_id}.json` exists containing an array of objects with fields `type` (decision|discovery|learning), `content`, `source_event_id`, and `timestamp`. At least one observation is present for sessions where the agent made explicit decisions.

## Improvement 3: Research curator cross-session memory for reducing redundant research

**Source pattern**: "The research-curator agent could leverage SimpleMem-Cross to retain findings about libraries, tools, and patterns across multiple research sessions, reducing redundant research and enabling pattern-based recommendations" (Relevance section, specific use case paragraph)
**Local system**: `.claude/agents/research-curator.md`, `.claude/skills/research-curator/SKILL.md`
**Confidence**: Medium
**Impact**: Low
**Backlog**: Deferred -- confidence medium: the research-curator agent currently has no cross-session memory, but it operates on persistent research files in `./research/` which serve as a durable knowledge base. The cross-reference system (`@research-cross-referencer`) already links related entries. The gap exists (no automatic recall of prior findings at session start) but the existing file-based research directory partially compensates, making the net improvement uncertain without measurement.

### Current state

The research-curator agent (`.claude/agents/research-curator.md`) starts each invocation with no memory of prior research sessions. It receives a URL or `--rerun` target and performs fresh research from primary sources. The research directory `./research/` contains all prior entries as markdown files, and the `@research-cross-referencer` agent adds cross-reference sections, but neither mechanism automatically surfaces relevant prior findings when starting a new research session. If the agent researches a tool similar to one already documented, it does not know about the prior entry unless explicitly told.

### Target state

When the research-curator agent starts a new research session, a pre-step would search the existing `./research/` directory (or a vector index over it) for entries semantically similar to the new target. The top N relevant prior entries would be summarized and injected into the agent's prompt, enabling it to note relationships, avoid redundant coverage, and identify patterns across tools.

### Measurable signal

The research-curator agent's prompt includes a `## Prior Related Research` section containing titles and key findings from 1-3 related existing entries when starting a new research session. The agent's output references at least one prior entry when a related entry exists in `./research/`.

## Improvement 4: Token-budgeted context injection at skill invocation

**Source pattern**: "Skill integration: Memory can be injected via lifecycle hooks into skill contexts, reducing context window pressure" and "Token counter budgets entries by their serialized length, starting with highest-relevance matches. Stop adding entries when token budget (default 2,000) is exhausted" (Relevance section item 2; Key Features > Automatic Context Injection)
**Local system**: `.claude/skills/research-curator/SKILL.md` (and skill system broadly)
**Confidence**: Low
**Impact**: Low
**Backlog**: Deferred -- confidence low: the local skill system does not have a context injection mechanism at skill load time. SimpleMem-Cross's token-budgeted injection is a concrete mechanism, but the local architecture would need a new hook type (pre-skill-load) or modification to how skills are activated. Whether Claude Code's skill activation supports such hooks is not confirmed from the files read. The gap requires architectural investigation before it can be considered actionable.

### Current state

Skills are loaded via `Skill(skill="name")` which reads the SKILL.md file and applies it as context. There is no pre-load hook that injects additional context from prior sessions. The skill system does not have a token budget mechanism for injected context. Each skill invocation starts with a clean slate of only the skill's own instructions.

### Target state

A pre-skill-load mechanism would query a memory store for observations relevant to the skill being invoked and inject them as a preamble, token-budgeted to a configurable limit (e.g., 2,000 tokens). The skill's SKILL.md frontmatter could declare `max_context_tokens: N` to control injection size.

### Measurable signal

A skill's frontmatter contains a `max_context_tokens` field. When the skill is loaded, a `## Injected Context` section appears in the agent's context containing prior observations relevant to the current task, with total token count not exceeding the configured budget.
