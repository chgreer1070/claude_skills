## Overview

A first-person engineering account by Ryan Lopopolo (OpenAI Member of Technical Staff) describing how a small team of three to seven engineers built a production internal beta product with approximately one million lines of code entirely generated by Codex agents over five months, with zero manually-written code. The article defines "harness engineering" as the discipline of designing environments, feedback loops, scaffolding, and control systems that allow AI coding agents to do reliable, high-throughput work. It documents specific architectural decisions, tooling patterns, and workflow philosophies that produced 3.5 pull requests per engineer per day.

---

## Problem Addressed

| Problem | Solution |
|---------|----------|
| AI agents underperform in underspecified environments, lacking tools and abstractions needed for complex tasks | Harness engineering: systematically building environment scaffolding depth-first, adding capabilities agents need before expecting complex output |
| Monolithic AGENTS.md files become stale, crowd out context, and provide non-guidance when everything is "important" | Treat AGENTS.md as table of contents (~100 lines) pointing to structured `docs/` directory as system of record, enforced by linters and CI |
| Agent-generated code drifts architecturally over time, replicating both good and bad patterns | Encode "golden principles" as custom linters with agent-readable error messages, enforce mechanically, run background "garbage collection" Codex tasks on a recurring cadence |
| Human QA capacity becomes the bottleneck as code throughput scales | Make application state (UI, logs, metrics, traces) directly legible to agents via Chrome DevTools Protocol, LogQL, PromQL, and ephemeral per-worktree observability stacks |
| Context management fails at scale when agents cannot access relevant information during task execution | Push all knowledge into the repository as versioned artifacts; knowledge in Slack threads or docs outside the repo is effectively invisible to agents |
| Conventional blocking merge gates become counterproductive at high agent throughput | Operate with minimal blocking gates; treat corrections as cheap and waiting as expensive; reserve human time for judgment and validation |

## Key Features

### Repository Structure as Agent Context System

- AGENTS.md kept at ~100 lines as table of contents only, not encyclopedic instruction set
- `docs/` directory structured as system of record: `design-docs/`, `exec-plans/`, `product-specs/`, `references/`, with index files
- `references/` subdirectory stores LLM-oriented documentation files (e.g., `uv-llms.txt`, `nixpacks-llms.txt`) directly in-repo
- Architecture documentation provides domain and package layering maps
- Quality document grades each product domain and architectural layer, tracking gaps over time
- Plans checked in as first-class artifacts: ephemeral lightweight plans for small changes, full execution plans with progress/decision logs for complex work
- Dedicated linters and CI jobs validate knowledge base is current, cross-linked, and correctly structured
- Recurring "doc-gardening" Codex task scans for stale documentation and opens fix-up pull requests

### Application Legibility for Agents

- Application bootable per git worktree, allowing one isolated instance per in-progress change
- Chrome DevTools Protocol wired into agent runtime, providing DOM snapshots, screenshots, and navigation capabilities
- Ephemeral per-worktree observability stack: logs queryable via LogQL, metrics queryable via PromQL, traces exposed, torn down when task completes
- Enables prompts like "ensure service startup completes in under 800ms" to be directly executable and verifiable by the agent

### Architectural Enforcement via Mechanical Constraints

- Rigid domain-based layered architecture: Types → Config → Repo → Service → Runtime → UI, with validated dependency directions
- Cross-cutting concerns (auth, connectors, telemetry, feature flags) enter through single explicit Providers interface
- All architectural rules enforced by Codex-generated custom linters with error messages written to inject remediation instructions into agent context
- "Taste invariants" enforced statically: structured logging, naming conventions for schemas and types, file size limits, platform-specific reliability requirements
- Agent-readable error messages transform linting from friction into guidance

### Agent-to-Agent Review Loop

- Pull request lifecycle: agent opens PR, requests additional agent reviews locally and in cloud, responds to feedback, iterates until all agent reviewers are satisfied
- Human review optional, not required
- Agents use standard development tools directly: `gh`, local scripts, repository-embedded skills
- Agents pull review feedback inline, push updates, squash, and merge their own pull requests
- Described as "Ralph Wiggum Loop" -- agent reviews its own changes and iterates until satisfied

### Entropy Management and Garbage Collection

- "Golden principles" encoded as mechanical rules in the repository
- Examples: prefer shared utility packages over hand-rolled helpers; validate data at boundaries rather than YOLO-probing shapes
- Recurring background Codex tasks scan for deviations, update quality grades, open targeted refactoring PRs
- Most cleanup PRs reviewable in under a minute, eligible for automerge
- Prevents "AI slop" accumulation that previously required 20% of engineering time (every Friday) to clean up

### Full End-to-End Agent Autonomy

- Given a single prompt, Codex can now: validate codebase state, reproduce a bug, record video of failure, implement fix, validate via application driving, record resolution video, open PR, respond to feedback, detect/remediate build failures, merge change
- Escalates to human only when judgment is required

## Technical Architecture

The harness engineering model operates on a single core principle: agents can only act on information available in context during a task run. The architecture is built around making relevant information discoverable, current, and structurally enforced.

```text
Repository Layout (System of Record)
├── AGENTS.md          (100 lines max, table of contents)
├── ARCHITECTURE.md    (domain and package layering map)
├── docs/
│   ├── design-docs/   (catalogued, indexed, verification-tracked)
│   ├── exec-plans/    (active/, completed/, tech-debt-tracker.md)
│   ├── generated/     (e.g., db-schema.md -- auto-generated)
│   ├── product-specs/ (indexed, feature specs)
│   ├── references/    (LLM-oriented: uv-llms.txt, nixpacks-llms.txt)
│   ├── DESIGN.md
│   ├── FRONTEND.md
│   ├── PLANS.md
│   ├── PRODUCT_SENSE.md
│   ├── QUALITY_SCORE.md
│   ├── RELIABILITY.md
│   └── SECURITY.md
└── [application code]
```

Agent task execution model:

1. Agent receives prompt from human engineer
2. Agent reads AGENTS.md for entry point and pointers
3. Agent navigates docs/ for relevant design docs, exec plans, architectural constraints
4. Agent implements changes using standard tools (gh, local scripts, repository-embedded skills)
5. Agent boots isolated worktree instance of application
6. Agent validates via browser (CDP), logs (LogQL), and metrics (PromQL)
7. Agent runs self-review loop and requests cloud-based agent reviews
8. Agent responds to feedback, iterates, merges
9. Background garbage-collection tasks scan for drift on recurring cadence

Technology choices favor agent-legibility: "boring" technologies with stable APIs and strong training-set representation are preferred. Some third-party library functionality is reimplemented in-repo where the reimplementation improves agent inspectability and test coverage (e.g., custom `map-with-concurrency` with 100% test coverage and OpenTelemetry instrumentation rather than `p-limit`).

## Installation & Usage

This is an article documenting practices and principles, not an installable tool. Key actionable patterns extracted:

### AGENTS.md Pattern

Keep AGENTS.md short (~100 lines) as a table of contents. Use it to point to deeper sources of truth. Never treat it as an encyclopedia.

```text
AGENTS.md structure:
- Brief repository overview (what this is, who it's for)
- Pointer to ARCHITECTURE.md
- Pointer to docs/design-docs/index.md
- Pointer to docs/exec-plans/active/
- Pointer to relevant FRONTEND.md, RELIABILITY.md, etc.
- Key invariants (3-5 non-negotiable rules)
```

### Structured docs/ Directory

Treat the docs directory as an owned, versioned, linted artifact. Include:
- Design documents with index and verification status
- Execution plans (active and completed) checked in as first-class artifacts
- Quality scores per domain, tracked over time
- LLM-format reference files for key dependencies (fetch vendor docs in `llms.txt` format)

### Custom Linter Error Messages for Agent Guidance

Write linter error messages as remediation instructions, not just error descriptions. When a lint rule fires, the error message should tell the agent exactly what to do.

```text
Instead of: "Error: missing boundary validation"
Write:      "Error: Missing boundary validation at data ingress point. Add Zod schema
             parse() at the boundary per docs/design-docs/core-beliefs.md#data-safety.
             See docs/references/validation-patterns.md for canonical examples."
```

### Progressive Disclosure Architecture

Structure agent-facing documentation so agents start with a small stable entry point and are guided to deeper sources only when needed. Avoid front-loading all context.

### Ephemeral Per-Worktree Environments

Make applications bootable per git worktree so agents can validate changes in full isolation without polluting other in-progress work.

### Recurring Garbage Collection

Schedule recurring agent tasks to:
- Scan for deviations from golden principles
- Update quality grade documents
- Open targeted refactoring PRs for identified drift

Design these tasks so their output PRs are reviewable in under one minute and eligible for automerge.
