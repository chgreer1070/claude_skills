# Kaizen Dashboard -- DuckDB Schema

Companion to [kaizen-dashboard-redesign-spec.md](./kaizen-dashboard-redesign-spec.md).

---

## Table Definitions

### sessions

Session-level metadata. One row per Claude Code session. Populated by the ingestion script from JSONL file headers and aggregated statistics.

```sql
CREATE TABLE IF NOT EXISTS sessions (
    session_id       TEXT PRIMARY KEY,
    project_path     TEXT,                    -- slug from ~/.claude/projects/{slug}/
    project_name     TEXT,                    -- human-readable (last segment)
    first_timestamp  TIMESTAMP WITH TIME ZONE,
    last_timestamp   TIMESTAMP WITH TIME ZONE,
    total_turns      INTEGER DEFAULT 0,
    tool_call_count  INTEGER DEFAULT 0,
    user_turn_count  INTEGER DEFAULT 0,
    error_count      INTEGER DEFAULT 0,
    intervention_count INTEGER DEFAULT 0,
    model            TEXT,                    -- primary model used (from assistant records)
    git_branch       TEXT,
    slug             TEXT,                    -- session slug (e.g. "squishy-discovering-breeze")
    ingested_at      TIMESTAMP WITH TIME ZONE DEFAULT now()
);
```

### findings

Detected anti-patterns from the 10-dimension analysis pipeline. Each finding represents a pattern observed across one or more sessions.

```sql
CREATE TABLE IF NOT EXISTS findings (
    finding_id       TEXT PRIMARY KEY,        -- UUID generated by ingestion script
    dimension        TEXT NOT NULL,           -- one of the 10 FindingDimension values
    severity         TEXT NOT NULL,           -- critical | warning | info
    title            TEXT NOT NULL,           -- short description
    description      TEXT,                    -- detailed explanation
    frequency        INTEGER DEFAULT 1,      -- number of sessions exhibiting this pattern
    first_seen       TIMESTAMP WITH TIME ZONE,
    last_seen        TIMESTAMP WITH TIME ZONE,
    recommendation   TEXT,                    -- hook | skill_patch | agent_prompt | claude_md | script
    status           TEXT DEFAULT 'open',     -- open | fixed | wont_fix | duplicate
    fix_id           TEXT,                    -- FK to fixes.fix_id (NULL if unfixed)
    ingested_at      TIMESTAMP WITH TIME ZONE DEFAULT now(),
    source_file      TEXT                     -- .planning/kaizen/ markdown file this came from
);

CREATE INDEX IF NOT EXISTS idx_findings_dimension ON findings(dimension);
CREATE INDEX IF NOT EXISTS idx_findings_severity ON findings(severity);
CREATE INDEX IF NOT EXISTS idx_findings_status ON findings(status);
```

### finding_evidence

Specific evidence instances supporting each finding. Links a finding to the exact session, timestamp, and transcript excerpt where the pattern was observed.

```sql
CREATE TABLE IF NOT EXISTS finding_evidence (
    evidence_id      TEXT PRIMARY KEY,        -- UUID
    finding_id       TEXT NOT NULL,           -- FK to findings.finding_id
    session_id       TEXT NOT NULL,           -- FK to sessions.session_id
    timestamp        TIMESTAMP WITH TIME ZONE,
    record_uuid      TEXT,                    -- UUID of the specific JSONL record
    tool_name        TEXT,                    -- tool involved (Bash, Read, Task, etc.)
    excerpt          TEXT,                    -- truncated transcript excerpt (max 500 chars)
    context_summary  TEXT,                    -- what was happening when this occurred
    ingested_at      TIMESTAMP WITH TIME ZONE DEFAULT now()
);

CREATE INDEX IF NOT EXISTS idx_evidence_finding ON finding_evidence(finding_id);
CREATE INDEX IF NOT EXISTS idx_evidence_session ON finding_evidence(session_id);
```

### interventions

Human interruptions or corrections during sessions. An intervention is any point where the human stopped the normal agent flow and redirected it.

```sql
CREATE TABLE IF NOT EXISTS interventions (
    intervention_id   TEXT PRIMARY KEY,       -- UUID
    session_id        TEXT NOT NULL,          -- FK to sessions.session_id
    timestamp         TIMESTAMP WITH TIME ZONE,
    intervention_type TEXT NOT NULL,          -- interrupt | correction | denial | abort | redirect
    trigger_text      TEXT,                   -- the user's message that constitutes the intervention
    trigger_preview   TEXT,                   -- truncated to 200 chars for display
    preceding_action  TEXT,                   -- what the agent was doing when interrupted
    preceding_tool    TEXT,                   -- tool name of the preceding agent action
    outcome           TEXT,                   -- what happened after the intervention
    related_finding_id TEXT,                  -- FK to findings.finding_id (NULL if not linked)
    ingested_at       TIMESTAMP WITH TIME ZONE DEFAULT now()
);

CREATE INDEX IF NOT EXISTS idx_interventions_session ON interventions(session_id);
CREATE INDEX IF NOT EXISTS idx_interventions_type ON interventions(intervention_type);
```

### state_transitions

The core state machine data. Each row represents one edge in the `context -> action -> outcome` flow. Context is derived from the preceding state (user turn, file state, active tools). Action is the agent's response (tool call or text). Outcome is determined by what follows (success, error, intervention).

```sql
CREATE TABLE IF NOT EXISTS state_transitions (
    transition_id     TEXT PRIMARY KEY,       -- UUID
    session_id        TEXT NOT NULL,          -- FK to sessions.session_id
    sequence_index    INTEGER NOT NULL,       -- position in the session flow (0-based)
    timestamp         TIMESTAMP WITH TIME ZONE,

    -- Context (input state)
    context_type      TEXT,                   -- user_message | tool_result | system_event
    context_summary   TEXT,                   -- short description of the input state
    context_files     TEXT,                   -- comma-separated file paths in context
    context_tokens    INTEGER,               -- input token count (from usage)

    -- Action (agent response)
    action_type       TEXT NOT NULL,          -- tool_use | text_response | delegation
    action_tool       TEXT,                   -- tool name (NULL for text_response)
    action_summary    TEXT,                   -- short description of the action
    action_model      TEXT,                   -- model used for this turn

    -- Outcome (result)
    outcome           TEXT NOT NULL,          -- success | error | intervention | abandoned
    outcome_detail    TEXT,                   -- error message, intervention type, etc.
    intervention_id   TEXT,                   -- FK to interventions (NULL if outcome != intervention)

    -- Linkage
    finding_id        TEXT,                   -- FK to findings (NULL if no associated finding)
    ingested_at       TIMESTAMP WITH TIME ZONE DEFAULT now()
);

CREATE INDEX IF NOT EXISTS idx_transitions_session ON state_transitions(session_id, sequence_index);
CREATE INDEX IF NOT EXISTS idx_transitions_outcome ON state_transitions(outcome);
CREATE INDEX IF NOT EXISTS idx_transitions_action ON state_transitions(action_tool);
CREATE INDEX IF NOT EXISTS idx_transitions_finding ON state_transitions(finding_id);
```

### fixes

Records of applied corrections (hooks, patches, CLAUDE.md updates, scripts) that were generated in response to findings.

```sql
CREATE TABLE IF NOT EXISTS fixes (
    fix_id            TEXT PRIMARY KEY,       -- UUID
    finding_id        TEXT NOT NULL,          -- FK to findings.finding_id
    fix_type          TEXT NOT NULL,          -- hook | skill_patch | agent_prompt | claude_md | script
    title             TEXT NOT NULL,
    description       TEXT,
    applied_at        TIMESTAMP WITH TIME ZONE,
    status            TEXT DEFAULT 'proposed', -- proposed | applied | verified | ineffective | reverted
    artifact_path     TEXT,                   -- path to the hook/patch/script file
    config_diff       TEXT,                   -- summary of what changed
    ingested_at       TIMESTAMP WITH TIME ZONE DEFAULT now()
);

CREATE INDEX IF NOT EXISTS idx_fixes_finding ON fixes(finding_id);
CREATE INDEX IF NOT EXISTS idx_fixes_status ON fixes(status);
```

### fix_measurements

Before/after metrics measuring whether a fix resolved the underlying finding. Each measurement compares the finding's occurrence rate in a window of sessions before the fix vs. a window after.

```sql
CREATE TABLE IF NOT EXISTS fix_measurements (
    measurement_id    TEXT PRIMARY KEY,       -- UUID
    fix_id            TEXT NOT NULL,          -- FK to fixes.fix_id
    measured_at       TIMESTAMP WITH TIME ZONE,
    window_before     INTEGER NOT NULL,       -- number of sessions in the "before" window
    window_after      INTEGER NOT NULL,       -- number of sessions in the "after" window
    occurrences_before INTEGER NOT NULL,      -- finding count in before-window sessions
    occurrences_after  INTEGER NOT NULL,      -- finding count in after-window sessions
    rate_before       DOUBLE,                -- occurrences_before / window_before
    rate_after        DOUBLE,                -- occurrences_after / window_after
    rate_change_pct   DOUBLE,               -- percentage change (negative = improvement)
    ingested_at       TIMESTAMP WITH TIME ZONE DEFAULT now()
);

CREATE INDEX IF NOT EXISTS idx_measurements_fix ON fix_measurements(fix_id);
```

### sentiment

Migrated from the CSV/existing DuckDB sentiment table. Schema matches the existing `sentiment` table in `kaizen.duckdb` plus project fields.

```sql
-- This table already exists in kaizen.duckdb (created by sentiment-score.py).
-- No migration needed -- the dashboard reads the existing table.
-- Schema shown here for completeness.

CREATE TABLE IF NOT EXISTS sentiment (
    session_id      TEXT    NOT NULL,
    timestamp       TEXT,
    message_index   INTEGER NOT NULL,
    compound        DOUBLE,
    positive        DOUBLE,
    negative        DOUBLE,
    neutral         DOUBLE,
    message_length  INTEGER,
    message_preview TEXT,
    project_path    TEXT,
    project_name    TEXT,
    PRIMARY KEY (session_id, message_index)
);
```

## Query Patterns for Dashboard Views

### Findings Browser Tab

```sql
-- Main findings list with fix status
SELECT
    f.finding_id,
    f.dimension,
    f.severity,
    f.title,
    f.frequency,
    f.first_seen,
    f.last_seen,
    f.recommendation,
    f.status,
    fx.title AS fix_title,
    fx.status AS fix_status
FROM findings f
LEFT JOIN fixes fx ON f.fix_id = fx.fix_id
WHERE (:dimension IS NULL OR f.dimension = :dimension)
  AND (:severity IS NULL OR f.severity = :severity)
  AND (:status IS NULL OR f.status = :status)
ORDER BY
    CASE f.severity
        WHEN 'critical' THEN 1
        WHEN 'warning' THEN 2
        WHEN 'info' THEN 3
    END,
    f.frequency DESC;

-- Dimension summary (sidebar chart)
SELECT
    dimension,
    COUNT(*) AS total,
    COUNT(*) FILTER (WHERE severity = 'critical') AS critical,
    COUNT(*) FILTER (WHERE severity = 'warning') AS warning,
    COUNT(*) FILTER (WHERE status = 'open') AS open
FROM findings
GROUP BY dimension
ORDER BY critical DESC, total DESC;

-- Evidence for a selected finding
SELECT
    e.session_id,
    e.timestamp,
    e.tool_name,
    e.excerpt,
    e.context_summary,
    s.project_name
FROM finding_evidence e
JOIN sessions s ON e.session_id = s.session_id
WHERE e.finding_id = :finding_id
ORDER BY e.timestamp DESC;
```

### Intervention Flow Tab

```sql
-- State transitions for flow graph (cross-session aggregation)
SELECT
    action_tool AS node,
    outcome,
    COUNT(*) AS edge_weight,
    COUNT(*) FILTER (WHERE outcome = 'intervention') AS intervention_count,
    ROUND(100.0 * intervention_count / COUNT(*), 1) AS intervention_pct
FROM state_transitions
WHERE (:session_id IS NULL OR session_id = :session_id)
  AND (:finding_id IS NULL OR finding_id = :finding_id)
GROUP BY action_tool, outcome
HAVING COUNT(*) >= :min_weight
ORDER BY edge_weight DESC;

-- Intervention detail for selected edge
SELECT
    i.intervention_type,
    i.trigger_preview,
    i.preceding_action,
    i.preceding_tool,
    i.outcome,
    st.context_summary,
    st.action_summary
FROM interventions i
JOIN state_transitions st ON i.intervention_id = st.intervention_id
WHERE (:session_id IS NULL OR i.session_id = :session_id)
ORDER BY i.timestamp DESC
LIMIT 50;

-- Intervention rate over time (rolling window)
WITH session_order AS (
    SELECT
        session_id,
        first_timestamp,
        intervention_count,
        ROW_NUMBER() OVER (ORDER BY first_timestamp) AS session_num
    FROM sessions
    WHERE intervention_count IS NOT NULL
)
SELECT
    session_id,
    first_timestamp,
    intervention_count,
    AVG(intervention_count) OVER (
        ORDER BY session_num
        ROWS BETWEEN :window_size PRECEDING AND CURRENT ROW
    ) AS rolling_avg
FROM session_order;
```

### Fix Tracker Tab

```sql
-- Fixes with latest measurement
SELECT
    fx.fix_id,
    fx.title,
    fx.fix_type,
    fx.status,
    fx.applied_at,
    f.title AS finding_title,
    f.dimension,
    f.severity,
    m.rate_before,
    m.rate_after,
    m.rate_change_pct
FROM fixes fx
JOIN findings f ON fx.finding_id = f.finding_id
LEFT JOIN (
    SELECT DISTINCT ON (fix_id)
        fix_id, rate_before, rate_after, rate_change_pct
    FROM fix_measurements
    ORDER BY fix_id, measured_at DESC
) m ON fx.fix_id = m.fix_id
WHERE (:status IS NULL OR fx.status = :status)
ORDER BY fx.applied_at DESC;

-- Fix effectiveness over time (measurement history)
SELECT
    m.measured_at,
    m.rate_before,
    m.rate_after,
    m.rate_change_pct,
    m.window_before,
    m.window_after
FROM fix_measurements m
WHERE m.fix_id = :fix_id
ORDER BY m.measured_at;
```

### Session Explorer Tab

```sql
-- Session list with summary stats
SELECT
    s.session_id,
    s.project_name,
    s.first_timestamp,
    s.last_timestamp,
    s.total_turns,
    s.tool_call_count,
    s.error_count,
    s.intervention_count,
    s.model,
    s.git_branch,
    s.slug,
    COUNT(DISTINCT f.finding_id) AS finding_count
FROM sessions s
LEFT JOIN finding_evidence fe ON s.session_id = fe.session_id
LEFT JOIN findings f ON fe.finding_id = f.finding_id
WHERE (:project_name IS NULL OR s.project_name = :project_name)
  AND (:date_from IS NULL OR s.first_timestamp >= :date_from)
  AND (:date_to IS NULL OR s.first_timestamp <= :date_to)
GROUP BY s.session_id, s.project_name, s.first_timestamp,
         s.last_timestamp, s.total_turns, s.tool_call_count,
         s.error_count, s.intervention_count, s.model,
         s.git_branch, s.slug
ORDER BY s.first_timestamp DESC;

-- Per-session timeline (state transitions in order)
SELECT
    st.sequence_index,
    st.timestamp,
    st.context_type,
    st.context_summary,
    st.action_type,
    st.action_tool,
    st.action_summary,
    st.outcome,
    st.outcome_detail,
    i.intervention_type,
    i.trigger_preview
FROM state_transitions st
LEFT JOIN interventions i ON st.intervention_id = i.intervention_id
WHERE st.session_id = :session_id
ORDER BY st.sequence_index;
```

### Sentiment Tab

```sql
-- Rolling mean trend (replaces CSV-based scatter)
SELECT
    message_index,
    compound,
    session_id,
    timestamp,
    message_preview,
    AVG(compound) OVER (
        ORDER BY message_index
        ROWS BETWEEN 25 PRECEDING AND 25 FOLLOWING
    ) AS rolling_mean
FROM sentiment
WHERE (:session_id IS NULL OR session_id = :session_id)
ORDER BY message_index;

-- Distribution stats
SELECT
    AVG(compound) AS mean_compound,
    MEDIAN(compound) AS median_compound,
    STDDEV(compound) AS stddev_compound,
    MIN(compound) AS min_compound,
    MAX(compound) AS max_compound,
    COUNT(*) AS total_messages
FROM sentiment
WHERE (:session_id IS NULL OR session_id = :session_id);
```

## Migration Path

### Phase 1: Schema Creation

The ingestion script creates all new tables on first run. The existing `sentiment` table is left untouched. No destructive migrations.

### Phase 2: Data Population

1. Run `sentiment-score.py` to ensure sentiment data is current
2. Run `ingest-findings.py` to populate `sessions`, `findings`, `finding_evidence`, `interventions`, `state_transitions`
3. `fixes` and `fix_measurements` start empty and are populated as fixes are applied and measured

### Phase 3: Dashboard Switchover

1. New `dashboard.py` reads from DuckDB (all tables) instead of CSV
2. Old CSV polling code removed
3. Health endpoint updated to report DuckDB connection status instead of CSV status
4. `start_dashboard()` accepts `db_path` parameter instead of `csv_path`

### Backward Compatibility

- The `sentiment` table in DuckDB already exists (created by `sentiment-score.py`). The dashboard reads it directly.
- The CSV file (`sentiment.csv`) is still produced by `sentiment-score.py` for any external consumers. The dashboard no longer reads it.
- The MCP tools in `server.py` are unaffected. They continue to operate on JSONL files directly and via the existing DuckDB connection.
