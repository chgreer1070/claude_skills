### How the Backlog State System Currently Works

The backlog system manages three separate but interconnected state dimensions that frequently drift apart:

1. **Local frontmatter** (`.claude/backlog/*.md` files) — contains a `metadata.status` field that is manually maintained
2. **GitHub labels** (on GitHub Issues) — contain `status:*` labels that are the canonical state machine representation
3. **GitHub issue state** (open/closed) — GitHub's native field, orthogonal to the state machine

The critical bug is that these three dimensions are **never synchronized on read operations**. When `backlog list --from-github` is called, it fetches only `state="open"` issues via `repo_obj.get_issues(state="open", ...)` (backlog.py:878, 755). This causes closed issues to vanish from the GitHub API results but remain in the local `.claude/backlog/` cache files indefinitely. Subsequent `backlog list` commands (without `--from-github`) read the local cache directly, showing closed items as still open.

**The flow in detail:**

1. **Item Creation** (`backlog.py:683`): New items are created with `status: open` (an undefined state not in the state machine) locally, and GitHub issue created with `status:needs-grooming` label.

2. **Grooming** (`backlog.py:1746-1747`): The grooming transition **breaks the state machine**:
   - Removes `status:needs-grooming` label from GitHub
   - **Does NOT** add `status:groomed` label (missing call to `apply_github_transition()`)
   - **Does NOT** update local frontmatter status field
   - Item enters "stateless void" with no label on GitHub and stale status locally

3. **Status Fetching** (`backlog.py:755-783`, `_batch_fetch_statuses()`): When `--with-status` is requested:
   - Queries GitHub for `state="open"` issues only
   - Builds a dict mapping issue number to status label
   - Closed issues are absent from `repo.get_issues(state="open")` result, so dict key is missing
   - Callers get empty status: `status_map.get(num, {"status": "", "milestone": ""})`

4. **Refresh Flow** (`backlog.py:860-886`, `_refresh_local_cache_from_github()`): When `--from-github` flag is used:
   - Fetches only open issues
   - Closed issues never trigger a refresh of their local cache file
   - Next default list shows closed items as still open

5. **Core Transition Pattern** (seen in `backlog.py:1188`, `_close_github_issue()`, `_resolve_github_issue()`):
   - Correct transitions (done, resolved, closed) use `apply_github_transition(repo, issue, from_state, to_state)` (state_handler.py:138-177)
   - This atomically removes old label and adds new label
   - Then call `_update_item_metadata(filepath, {"metadata": {"status": new_state}})` to persist locally
   - But early transitions (creation, grooming) skip this pattern entirely

**The state machine itself is defined in `state_handler.py:54-63`** as `VALID_TRANSITIONS` DAG:
- 8 canonical states: needs-grooming, groomed, blocked, in-milestone, in-progress, done, resolved, closed
- Each state maps to a frozenset of valid target states
- The functions `validate_transition()` (state_handler.py:99-130) checks single-hop validity
- The function `apply_github_transition()` atomically swaps GitHub labels

**Existing synchronization utilities that will be reused:**
- `parse_backlog()` (backlog.py:234-298) — reads and parses all `.claude/backlog/*.md` files
- `_batch_fetch_statuses()` (backlog.py:755-783) — queries GitHub for issue statuses (currently open-only)

#### Key Decision: GitHub as Authoritative Source

When divergence is detected between local frontmatter `status` and GitHub `status:*` label:
- GitHub is always the source of truth
- Local cache is the derived artifact
- Auto-correct by updating local to match GitHub (using _update_item_metadata)
- Use `find_valid_path()` to determine if divergence is DAG-valid (reachable through state transitions)

#### Key Patterns to Preserve

1. **Atomic state transitions** (existing pattern, see backlog.py:1188):
   - Step 1: validate with VALID_TRANSITIONS
   - Step 2: apply_github_transition() for label change
   - Step 3: _update_item_metadata() for local frontmatter
   - Step 4: optional issue.edit(state="closed") for GitHub native state

2. **Graceful GitHub degradation** (existing pattern, _try_get_github):
   - For optional operations: use _try_get_github (returns None on failure, 10-sec timeout)
   - For required operations: use _get_github (fails hard)

3. **Batch operations over single** (existing pattern):
   - Prefer _batch_fetch_statuses() over _fetch_item_status()
   - Single API call, build result dict, iterate items once

4. **Error handling** (existing pattern, catch GithubException):
   - State transitions: fail hard if GitHub unavailable
   - Enrichment/status reading: fail soft, use cache fallback
   - Output reporting: emit warnings to stderr, not stdout
