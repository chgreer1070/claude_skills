### MCP Prompts in FastMCP 3.x

The FastMCP framework (version in worktree: `.claude/worktrees/fastmcp/docs/`) supports prompts via the `@mcp.prompt` decorator (no parentheses). Prompts are distinct from tools:

- **Tools** (`@mcp.tool()`): Functions that perform actions and return structured `dict` results
- **Prompts** (`@mcp.prompt`): Functions that return guidance content as `str` or `list[Message]`, used to orient LLMs before tool invocation

The decorator syntax is important: `@mcp.prompt` (bare decorator, no parentheses) for prompts, `@mcp.tool()` (with parentheses and empty parens) for tools.

**Message objects**: Prompts can return plain `str` or a list of `Message` objects. `Message` is imported from `fastmcp.prompts` and takes a required `content` parameter (the text) and optional `role` parameter (`"user"` or `"assistant"`, defaulting to `"user"`). The `Message` constructor signature is `Message(content, role="user")`.

**Async vs sync**: Prompts can be `async def` if they perform I/O (e.g., calling operations that hit the GitHub API). Pure string-building prompts are `def` (sync). FastMCP automatically runs sync functions in a threadpool to avoid blocking the event loop. The distinction is: use `async def` only when the function body contains `await` expressions.

**Source documentation**: The authoritative reference for prompt decorators is `.claude/worktrees/fastmcp/docs/servers/prompts.mdx` (read 2026-03-06). Dynamic prompts with `Context` parameters are documented at `.claude/worktrees/fastmcp/docs/servers/prompts.mdx` lines 384-400, showing the pattern of async prompts that call operations and return `list[Message]`.

### Guided Tools with Elicitation in FastMCP 3.x

Guided tools are `async def` tools that use `ctx: Context` to call `ctx.elicit()` for interactive field collection. The `Context` parameter is passed by FastMCP to any tool decorated with `@mcp.tool()` that declares it.

**Elicitation API**: `ctx.elicit()` is an `await`-able call that returns an `ElicitationResult` with `action` (one of `"accept"`, `"decline"`, or `"cancel"`) and `data` (structured response matching the `response_type` schema).

**NotSupportedError**: The `NotSupportedError` exception is raised by `ctx.elicit()` when the MCP client does not implement the elicitation protocol. The import path is **critical**: `from fastmcp.server.elicitation import NotSupportedError` (not from `fastmcp` directly). This is verified in `.claude/worktrees/fastmcp/docs/v2/servers/elicitation.mdx` which shows example imports using `from fastmcp.server.elicitation import AcceptedElicitation, ...`.

**Structured schemas**: The `response_type` parameter to `ctx.elicit()` can be a Python `@dataclass` class. FastMCP introspects the dataclass at server registration time and requires it to be defined at module level (not inside functions). Field types can use `Literal` for constrained values, which renders as dropdowns in supporting clients.

**Three outcomes**: All guided tools that use `ctx.elicit()` must handle all three action outcomes explicitly:
- `"accept"`: User confirmed the input; proceed with the operation using `result.data`
- `"decline"`: User declined to provide input; return a friendly message
- `"cancel"`: User cancelled the operation; return a cancellation message

**Source documentation**: `.claude/worktrees/fastmcp/docs/v2/servers/elicitation.mdx` documents the full elicitation API (read 2026-03-06).

### Error Handling Order in backlog_add_guided

The exception handling order is critical and non-obvious:

```python
try:
    result = await ctx.elicit(message="...", response_type=AddItemRequest)
    # handle accept/decline/cancel outcomes
except NotSupportedError:
    # First: catch NotSupportedError (does NOT inherit from BacklogError)
    return {"error": "This MCP client does not support...", "messages": [], "warnings": []}
except BacklogError as e:
    # Second: catch BacklogError (catches errors from operations.add_item)
    return {"error": str(e), **out.to_dict()}
```

If the order is reversed (BacklogError first), it will not catch `NotSupportedError` because `NotSupportedError` does not inherit from `BacklogError`. This is specified in the architect spec Section 2.1 and must be preserved exactly.
