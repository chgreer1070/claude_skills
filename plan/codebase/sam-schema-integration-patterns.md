# SAM Schema Integration Patterns

**Analysis Date:** 2026-03-15
**Focus Areas:** Architecture and conventions for sam_schema module integration across SAM workflow components

---

## 1. Overview: Current Integration State

The SAM workflow has 7 key components that interact with task files. These components currently use three different access patterns:

1. **Direct file parsing** ŌĆö `implementation_manager.py` reads task files directly with custom YAML parsing
2. **Direct file updates** ŌĆö `task_status_hook.py` updates task files in-place via custom field writers
3. **Shell command invocation** ŌĆö Skills invoke `implementation_manager.py` as an external CLI tool

The `sam_schema` module provides a unified schema, query API, and writer interface, but **adoption is incomplete**. Some components use the sam_schema API; others still use legacy direct file access.

## 2. Task File Access Patterns: Current State

### 2.1 Component Access Methods

| Component | Location | Access Method | Current Pattern |
|-----------|----------|----------------|-----------------|
| swarm-task-planner | `plugins/python3-development/agents/swarm-task-planner.md` | Writes task files | Writes YAML frontmatter markdown directly via Write/Edit tools |
| start-task skill | `plugins/python3-development/skills/start-task/SKILL.md` | Reads + claims tasks | Invokes `uv run ... implementation_manager.py claim-task` |
| task_status_hook | `plugins/python3-development/skills/implementation-manager/scripts/task_status_hook.py` | Updates status/timestamps | Uses `update_yaml_field()` from `task_format.py`; also imports `sam_schema.writers.yaml_writer.update_field` |
| implementation_manager | `plugins/python3-development/skills/implementation-manager/scripts/implementation_manager.py` | Queries task state | Imports `sam_schema.core.query.load_plan`, custom parsing via `task_format.py` |
| implement-feature skill | `plugins/python3-development/skills/implement-feature/SKILL.md` | Queries status/ready tasks | Invokes `uv run ... implementation_manager.py status . <slug>`, `ready-tasks . <slug>` |

| context-gathering agent | `plugins/python3-development/agents/context-gathering.md` | Updates Context Manifest | Reads task file, appends/updates markdown Context Manifest section |

### 2.2 File Path Conventions

**Task file locations:**
- Single-file plans: `plan/tasks-{N}-{slug}.md` (where N is task count, slug is feature identifier)
- Directory-based plans: `plan/tasks-{slug}/` (one file per task, each with YAML frontmatter)

**Verification result files** (generated by bookend agents):
- T0 baseline: `plan/T0-baseline-{slug}.yaml` (written by `t0-baseline-capture` agent)
- TN verification: `plan/TN-verification-{slug}.yaml` (written by `tn-verification-gate` agent)

**Context files** (runtime transient):
- Active task context: `.claude/context/active-task-{CLAUDE_SESSION_ID}.json` (written by `/start-task`)

### 2.3 Hook Configuration Patterns

The `start-task` skill declares a `PostToolUse` hook that executes `task_status_hook.py`:

```yaml
hooks:
  PostToolUse:
  - matcher: Write|Edit|Bash
    hooks:
    - type: command
      command: python3 "${CLAUDE_SKILL_DIR}/../../implementation-manager/scripts/task_status_hook.py"
```

**Path resolution:**
- `${CLAUDE_SKILL_DIR}` resolves to the directory containing the active skill's SKILL.md
- `../../implementation-manager/scripts/` navigates from `start-task/` to sibling `implementation-manager/scripts/`

**Hook invocation sequence:**
1. On `PostToolUse` (Write, Edit, or Bash): Hook reads `.claude/context/active-task-{session_id}.json` to know which task is active, updates `LastActivity` timestamp
2. On `SubagentStop` (sub-agent completion): Hook extracts task info from prompt, marks status as `COMPLETE`, adds `Completed` timestamp, syncs to GitHub if applicable

## 3. SAM Schema Module API: Available and Missing

### 3.1 Core Query API (`sam_schema.core.query`)

**Currently imported and used:**

- `load_plan(path: Path) -> Plan` ŌĆö Load a task file or plan directory, return unified Plan object
- `get_task(plan: Plan, task_id: str) -> Task` ŌĆö Retrieve a single task by ID
- `get_ready_tasks(plan: Plan) -> List[Task]` ŌĆö Get tasks with all dependencies resolved
- `get_plan_status(plan: Plan) -> PlanStatus` ŌĆö Aggregate task counts and readiness state
- `update_status(plan: Plan, task_id: str, status: TaskStatus) -> bool` ŌĆö Update task status and write changes

**Location of imports:**
- `implementation_manager.py` line 63: `from sam_schema.core.query import load_plan as sam_load_plan`
- `task_status_hook.py` line 52: `from sam_schema.core.query import update_status as sam_update_status`
- `sam_schema/cli.py` lines 27-29: Imports `get_plan_status`, `get_ready_tasks`, `get_task`, `load_plan`, `update_status`

### 3.2 Addressing API (`sam_schema.core.addressing`)

**Currently available:**

- `parse_address(address_str: str) -> Tuple[str, str]` ŌĆö Parse "P1/T3" style address into plan and task components
- `resolve_plan_address(address_part: str, plan_dir: Path) -> Path` ŌĆö Resolve plan identifier to filesystem path
- `AddressingError` ŌĆö Custom exception for invalid addresses

**Usage:**
- `sam_schema/cli.py` lines 27-28: Used to parse user input addresses in `read`, `state`, `ready`, `status` commands

### 3.3 Writers API (`sam_schema.writers.yaml_writer`)

**Currently available:**

- `update_field(path: Path, task_id: str, field: str, value: Any) -> bool` ŌĆö Update a single YAML field in a task
- `write_plan(path: Path, plan: Plan) -> None` ŌĆö Write an entire Plan back to disk

**Usage:**
- `task_status_hook.py` line 53: `from sam_schema.writers.yaml_writer import update_field as sam_update_field`

### 3.4 CLI Commands (`sam_schema.cli`)

**Available subcommands:**

- `sam read <address>` ŌĆö Read a single task (output: json/yaml/rich)
- `sam state <address> <new-status>` ŌĆö Update task status
- `sam ready <plan>` ŌĆö List ready tasks
- `sam status <plan>` ŌĆö Show plan progress
- `sam migrate <plan>` ŌĆö Migrate legacy markdown tasks to YAML frontmatter

**Example invocations:**
```bash
sam read P1/T3
sam state P1/T3 complete
sam ready P1
sam status P1
sam migrate plan/tasks-1-auth-system.md
```

### 3.5 Known Gaps and Missing Features

**Missing from sam_schema.core.query:**

1. **`create_task(plan: Plan, task_data: dict) -> Task`** ŌĆö No API to create new tasks programmatically (swarm-task-planner currently writes YAML directly)
2. **`claim_task(plan: Plan, task_id: str) -> bool`** ŌĆö No API to mark a task as claimed (prevents duplicate dispatch); `implementation_manager.py` has a standalone `claim-task` command instead
3. **`get_task_context(plan: Plan, task_id: str) -> dict`** ŌĆö No API to retrieve the Context Manifest for a task (needed for recursive follow-up routing)
4. **`get_bookend_tasks(plan: Plan) -> List[Task]`** ŌĆö No API to query T0/TN bookend tasks by `bookend-type` field

**Missing from sam_schema.writers:**

1. **`append_section(path: Path, task_id: str, section_name: str, content: str)`** ŌĆö No API to append sections (like Context Manifest or Divergence Notes); currently done via Bash/Edit tool manipulation in skills

**Missing from sam_schema.cli:**

1. **`sam create`** ŌĆö No command to create new plans or tasks from CLI
2. **`sam update`** ŌĆö No command to update plan metadata (only field-level `state` command exists)
3. **No output for plan context in `sam read`** ŌĆö The `read` command returns task metadata but omits the Context Manifest section

## 5. File Path References in Backlog Items

Backlog items (`.claude/backlog/*.md`) store optional `plan:` fields that reference plan files produced during implementation phases.

**Format observed in backlog sample:**
- Backlog item metadata section (YAML frontmatter) does NOT contain a `plan:` field in current items
- The `plan:` field is added programmatically by `mcp__backlog__backlog_update(selector=..., plan=...)` when a plan is attached

**When plan references need updating:**
When task files are renamed or reorganized (e.g., from `plan/tasks-1-auth.md` to `plan/tasks-auth/`), backlog items that reference them via the `plan:` field must be updated to reflect the new paths.

**Update mechanism:**
Via the MCP tool `mcp__backlog__backlog_update(selector="item_title", plan="path/to/tasks-file.md")` ŌĆö this is the authoritative way to update backlog-plan associations.

## 6. Hook Configuration and Variable Resolution

### 6.1 Variable Resolution in Hook Declarations

**SKILL.md hook declarations use environment variable substitution:**

```yaml
command: python3 "${CLAUDE_SKILL_DIR}/../../implementation-manager/scripts/task_status_hook.py"
```

**Variable behavior:**
- `${CLAUDE_SKILL_DIR}` ŌĆö Resolves at runtime to the directory containing the SKILL.md file
- Path is relative (uses `..` to navigate up directory structure)
- Hook receives full absolute path after variable substitution

**For `/start-task` skill:**
- SKILL.md location: `plugins/python3-development/skills/start-task/SKILL.md`
- `${CLAUDE_SKILL_DIR}` ŌåÆ `plugins/python3-development/skills/start-task/`
- `../../implementation-manager/scripts/` ŌåÆ `plugins/python3-development/skills/implementation-manager/scripts/`

### 6.2 Hook Script Entry Points

**Hook receives JSON on stdin with structure:**
```json
{
  "event": "PostToolUse|SubagentStop",
  "tool": "Write|Edit|Bash",
  "prompt": "...",
  "arguments": "..."
}
```

**Script responsibilities:**
1. Parse JSON from stdin
2. For `PostToolUse` events: Read `.claude/context/active-task-{CLAUDE_SESSION_ID}.json` to learn active task
3. For `SubagentStop` events: Extract task file path and task ID from prompt
4. Update task file YAML with new status/timestamps
5. Optionally sync completion to GitHub (non-fatal on failure)

### 7.2 Current Adoption Pattern

**Heavy users of sam_schema API:**
- `implementation_manager.py` ŌĆö Uses `load_plan`, `get_ready_tasks`, `get_task`, `update_status` (lines 63, 52)
- `task_status_hook.py` ŌĆö Uses `update_status`, `update_field` from sam_schema (lines 52-53)
- `sam_schema/cli.py` ŌĆö Uses all query/addressing APIs (lines 27-31)

**Light/no adoption:**
- `swarm-task-planner.md` ŌĆö Writes YAML directly via Write/Edit tools (no sam_schema calls)
- `start-task/SKILL.md` ŌĆö Delegates task claiming to `implementation_manager.py` CLI (no direct sam_schema calls)
- `implement-feature/SKILL.md` ŌĆö Invokes `implementation_manager.py` as external process (no direct sam_schema calls)
- `complete-implementation/SKILL.md` ŌĆö Reads task files manually, globs for follow-up files (no sam_schema calls)
- `context-gathering.md` ŌĆö Reads task file, appends markdown manually (no sam_schema calls)

### 7.3 Critical Path: Status Updates

When a task must move from `in-progress` ŌåÆ `complete`:

1. **Trigger**: SubagentStop hook fires (sub-agent finishes execution)
2. **Handler**: `task_status_hook.py` line 52 calls `sam_update_status()`
3. **sam_schema.core.query.update_status()** reads task file, parses YAML, updates status field, writes back
4. **GitHub sync**: Hook calls `backlog_core.github.update_task_status()` (non-fatal on failure)
5. **Cleanup**: Hook deletes `.claude/context/active-task-{session_id}.json` context file

**Invariant:** Only `task_status_hook.py` may mutate task file status. Direct edits via Edit tool are prohibited (enforced by `/start-task` skill documentation).

## 8. Conventions: How Components Access Task Files

### 8.1 Task File Format Conventions

**YAML Frontmatter Format** (required for new task files):

```markdown
---
task: T1
title: Implement core module
status: not-started
agent: python-cli-architect
dependencies: []
priority: 1
complexity: medium
accuracy-risk: low
skills: ["python3-development"]
---

## Context
[task context]

## Objective
[one-sentence objective]

## Requirements
1. Requirement 1
2. Requirement 2

## Expected Outputs
- file1.py
- file2.py

## Acceptance Criteria
1. Criterion 1
2. Criterion 2

## Verification Steps
1. Step 1
2. Step 2
```

**Status values** (case-sensitive in YAML):
- `not-started`
- `in-progress`
- `complete`
- `blocked`
- `deferred`
- `skipped`

**Field access convention:**
All YAML fields are lowercase with hyphens (not underscores): `not-started`, `accuracy-risk`, `parallelize-with`

### 8.2 Query Convention: Ready Task Resolution

A task is "ready" when:
1. Status is `not-started` (not already started or complete)
2. All tasks listed in `dependencies:` field have status `complete`

**Entry point:** `implementation_manager.py ready-tasks . <slug>` returns JSON with task list

**Output shape:**
```json
{
  "feature": "auth-system",
  "ready_tasks": [
    {
      "task_id": "T1",
      "title": "Setup project structure",
      "agent": "python-cli-architect",
      "skills": ["python3-development"]
    }
  ],
  "count": 1
}
```

### 8.3 Status Update Convention: Hook-Driven Updates

**Never edit task status directly.** Use the hook mechanism:

1. Agent performs work (Read ŌåÆ Implement ŌåÆ Verify)
2. Agent finishes (SubagentStop event)
3. Hook reads prompt, extracts task info
4. Hook calls `sam_schema.core.query.update_status()`
5. Task status written to file automatically

**Conventions:**
- `Started` timestamp: Added by `claim-task` command (not by agent)
- `LastActivity` timestamp: Updated by PostToolUse hook on every Write/Edit/Bash
- `Completed` timestamp: Added by SubagentStop hook when task marked complete
- All timestamps in ISO 8601 format (UTC)

### 8.4 Plan Discovery Convention

When a skill needs to locate a task file:

1. If user provides explicit path (ends with `.md`): Use that path directly
2. If user provides a slug or identifier: Query `implementation_manager.py status . <slug>` to resolve
3. Script searches for plan files in these locations (in order):
   - `plan/`, `.claude/plan/`, `plans/`, `docs/plan/`, `docs/plans/`
   - Package-level: `*/plan/`, `packages/*/plan/`
   - Recursive search (max depth 3) for any `plan/` directory

## 9. Recommended Integration Refactoring

### 9.1 High Priority: Close the Query API Gaps

**To consolidate task file access around sam_schema:**

1. **Add `claim_task()` API** to `sam_schema.core.query`
   - Currently: `implementation_manager.py` has a standalone `claim-task` command
   - Goal: Move claim logic into sam_schema so agents can call `update_status(..., claim=True)`
   - Impact: Simplifies `start-task` skill ŌĆö no need for external CLI invocation

2. **Add `get_task_context()` API** to `sam_schema.core.query`
   - Currently: `/complete-implementation` reads task file manually to extract Context Manifest
   - Goal: Query API that returns task metadata + Context Manifest in structured form
   - Impact: Enables programmatic access to task context without string manipulation

3. **Add `append_section()` API** to `sam_schema.writers`
   - Currently: Skills append Divergence Notes and Context Manifest sections via Edit tool
   - Goal: Structured API to append markdown sections to task YAML frontmatter body
   - Impact: Eliminates fragile string-based editing of task files

### 9.2 Medium Priority: Unify Agent File Access

**All agents should use sam_schema for task reading:**

1. **swarm-task-planner.md** ŌĆö Currently writes YAML directly
   - Should use `sam_schema.writers.yaml_writer.write_plan()` or new `create_task()` API
   - Impact: Validates all generated tasks against schema; catches invalid field values

2. **context-gathering.md** ŌĆö Currently reads task files, appends Context Manifest manually
   - Should use `sam_schema.core.query.get_task()` + `sam_schema.writers.append_section()`
   - Impact: Eliminates parsing fragility; schema validates section format

3. **complete-implementation skill** ŌĆö Currently globs for follow-up files and reads task files
   - Should use `sam_schema.core.query.get_plan_status()` to determine completion state
   - Should use `sam_schema.core.query.get_task()` to extract task metadata (agent field, etc.)
   - Impact: Reduces brittle file path dependencies; uses schema queries instead

### 9.3 Low Priority: CLI Enhancement

**Extend `sam_schema.cli` for missing commands:**

1. **`sam create`** ŌĆö Create new task files with schema validation
   - Usage: `sam create --plan <dir> --task <id> --title "..." --agent "..."`
   - Output: Validated YAML task file

2. **`sam update`** ŌĆö Update plan metadata (acceptance-criteria-structured, bookend flags)
   - Usage: `sam update <plan> --set field=value`

3. **Enhance `sam read`** ŌĆö Include Context Manifest in output
   - Currently: Returns only metadata and body sections
   - Goal: Optionally include context via `--include-context` flag

## 11. Adoption Path: Phased Migration

### Phase 1: Unblock Low-Risk Components (foundation)

1. Implement `sam_schema.core.query.claim_task()` API
2. Update `start-task/SKILL.md` to use claim API directly (if exposed via new script command)
3. No changes to swarm-task-planner, implement-feature, or other skills

**Outcome:** Task claiming moves into sam_schema; `/start-task` skill code simplifies

### Phase 2: Unify Writer Access (consistency)

1. Implement `sam_schema.writers.append_section()` API for Context Manifest and Divergence Notes
2. Update `context-gathering.md` agent to use append API instead of manual string editing
3. Update any Edit-based section appending in skills to use the sam_schema API

**Outcome:** All task file mutations go through sam_schema writers; consistent format validation

### Phase 3: Unify Query Access (completeness)

1. Implement `sam_schema.core.query.get_task_context()` API

## 12. Backlog Item Plan Field Updates

When task files are renamed or reorganized, backlog items that reference them must be updated.

**Current mechanism:**
```bash
mcp__backlog__backlog_update(selector="Item Title", plan="path/to/new/tasks-file.md")
```

**Rename scenarios requiring updates:**

1. **Single-file to directory split**: `plan/tasks-1-auth.md` ŌåÆ `plan/tasks-auth/` (update all backlog items)
2. **File renaming**: `plan/tasks-1-old-slug.md` ŌåÆ `plan/tasks-1-new-slug.md` (update backlog items linked to old slug)
3. **Follow-up task reorganization**: `plan/tasks-8-auth-followup-1.md` is routed to backlog ŌåÆ plan field added

**Auditing for stale references:**
```bash
grep -r "plan:" .claude/backlog/*.md | grep "tasks-.*-[slug]\.md"
```

Then compare against actual files in `plan/`:
```bash
find plan -name "tasks-*.md" -type f
find plan -type d -name "tasks-*"
```

## 13. Summary: Key Takeaways for Integration

1. **sam_schema module is the canonical data layer** ŌĆö All task file access should flow through it
2. **Three APIs exist: query, writers, CLI** ŌĆö Each layer has gaps that prevent complete adoption
3. **Hook-driven status updates are the pattern** ŌĆö Never edit task status directly via Edit tool
4. **Development-harness replicates core agents** ŌĆö Updates to agent files in python3-development should mirror to development-harness
5. **Backlog items reference plans via `plan:` field** ŌĆö Updates to file paths require backlog synchronization
6. **Path conventions enable plan discovery** ŌĆö Scripts search multiple locations for `plan/` directories
7. **No invented limits** ŌĆö Task file format and schema are extensible; add fields as needed
