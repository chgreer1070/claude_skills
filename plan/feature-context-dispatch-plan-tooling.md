## Original Request

Dispatch plan tooling for milestone orchestration.

The dispatch plan is a YAML file (`plan/milestone-{N}-dispatch.yaml`) generated by `/groom-milestone`
and consumed by `/work-milestone`. It encodes wave assignments, conflict groups, quality gates, and
per-item metadata needed to orchestrate parallel execution of backlog items within a milestone.

The full schema is defined in `.claude/reports/milestone-orchestration-design-20260320.md`.

---

## Core Intent Analysis

### WHO (Target Users)

- **`/groom-milestone` skill** — writes a dispatch plan after analyzing all items in a milestone
- **`/work-milestone` skill** — reads the dispatch plan to execute waves in order
- **Orchestrator agents** — read plan state to decide which items to dispatch next and validate plan integrity before starting

### WHAT (Desired Outcome)

A Python module (`plugins/development-harness/dispatch_schema/`) that provides:

1. Pydantic models for the dispatch plan YAML schema
2. Read and write functions using `ruamel.yaml` (consistent with `sam_schema` patterns)
3. Validation of plan integrity (conflict group consistency, wave dependency ordering, item references)

Additionally, a function `analyze_impact_radius_conflicts()` in
`plugins/development-harness/backlog_core/operations.py` that:

- Accepts a list of groomed backlog items
- Reads each item's Impact Radius (file paths affected)
- Returns conflict groups: sets of items whose Impact Radii overlap

### WHEN (Trigger Conditions)

- `/groom-milestone` completes dependency analysis (Step 6 in the design doc) and needs to persist the result
- `/work-milestone` starts (Step 1: Load Dispatch Plan) and needs to read and validate the file
- `analyze_impact_radius_conflicts()` is called during `/groom-milestone` Step 5 (Dependency Analysis) after all items are groomed

### WHY (Problem Being Solved)

The milestone orchestration workflow (`/groom-milestone` → `/work-milestone`) requires a durable,
structured artifact that captures the results of dependency analysis so that:

- Wave dispatch order is deterministic and reproducible from the same set of items
- `/work-milestone` can validate that items haven't changed since grooming (stale plan detection)
- Conflict groups prevent concurrent modification of overlapping file paths
- Quality gates are stored alongside the plan and executed at merge time

Without this module, `/groom-milestone` has nowhere to persist its analysis, and `/work-milestone`
has no validated input to read.

## Use Scenarios

### Scenario 1: Grooming produces a dispatch plan

**Actor**: `/groom-milestone` skill (orchestrator)
**Trigger**: Step 9 (Build Dispatch Plan) in the groom workflow — all items groomed, waves assigned, conflict groups identified
**Goal**: Persist the structured wave/conflict analysis to a file that `/work-milestone` can read
**Expected Outcome**: `plan/milestone-3-dispatch.yaml` written with correct schema; `dispatch_schema` write function returns success; file is valid YAML readable by the reader

### Scenario 2: Work-milestone loads and validates the plan

**Actor**: `/work-milestone` skill (orchestrator)
**Trigger**: User invokes `/work-milestone 3`
**Goal**: Load the dispatch plan, confirm all referenced items still exist and are open, confirm plan is structurally valid before spawning any team members
**Expected Outcome**: Reader returns a `DispatchPlan` Pydantic model; validation confirms wave dependency ordering is consistent; if items have changed since grooming, returns a stale-plan signal

### Scenario 3: Conflict group analysis during grooming

**Actor**: `/groom-milestone` orchestrator calling `analyze_impact_radius_conflicts()`
**Trigger**: Step 5 (Dependency Analysis) — all items groomed with Impact Radius populated
**Goal**: Determine which items touch overlapping file paths so they cannot run in parallel
**Expected Outcome**: Function returns a list of conflict groups (each group is a list of item titles/issue numbers that share file path prefixes); items with no overlap are ungrouped

### Scenario 4: Stale plan detection

**Actor**: `/work-milestone` skill after loading a plan
**Trigger**: Items in the milestone have changed since the plan was written (new items added, items closed)
**Goal**: Detect that the plan no longer matches the current milestone state
**Expected Outcome**: Validation function returns a stale-plan indicator; `/work-milestone` directs user to re-run `/groom-milestone`

### Q2: Conflict group overlap threshold (depth rule)

- **Category**: Behavior
- **Gap**: The design doc states "depth 2+" for domain overlap detection, but does `analyze_impact_radius_conflicts()` apply the same rule?
- **Question**: Two items both touching `plugins/development-harness/sam_schema/` and `plugins/development-harness/backlog_core/` — do they form a conflict group (both under `plugins/development-harness/`) or not (different depth-3 directories)?
- **Options**:
  - A) Any shared path prefix at depth 2 or more forms a conflict group
  - B) Only items sharing the same depth-3 or deeper prefix conflict
  - C) Only items with overlapping literal file paths (not directories) conflict
- **Why It Matters**: Determines parallelism granularity — option A is most conservative (fewer parallel slots), option C is most permissive (more conflict escapes detection)
- **Resolution**: _pending_

### Q3: dispatch_schema CLI vs library-only
