### How This Currently Works: Session Historian JSONL Querying and Storage

When a user or agent invokes `uv run .claude/skills/session-historian/scripts/session_query.py [command]`, the request hits the Typer CLI application defined in `session_query.py`. The script is a PEP 723 standalone file — no installation required; `uv run` resolves dependencies from the inline metadata block (lines 2-7: `duckdb>=1.0.0`, `typer>=0.21.0`).

Each command follows an orchestration pattern: parse arguments via Typer decorators → resolve session ID (with DuckDB) → read and parse JSONL → render output (Rich or raw) → exit with code 0 or 1.

#### JSONL Storage and Encoding

All session data lives under `~/.claude/projects/<encoded-cwd>/` as JSONL files, one file per session. The encoded-cwd format encodes the project working directory path:

- **Algorithm**: `str(Path.cwd()).replace("/", "-").lstrip("-")`
- **Example**: `/home/alice/repos/my-project` → `home-alice-repos-my-project`
- **File naming**: One session per JSONL file, named by session ID (UUID-like string derived from `CLAUDE_SESSION_ID` env var or the filename stem when indexing)
- **File path pattern**: `~/.claude/projects/<encoded-cwd>/<session-id>.jsonl`

The DuckDB index at `~/.claude/kaizen/session-index.duckdb` maintains two tables:

1. **`sessions` table** (columns: session_id, file_path, project_slug, project_name, first_ts, last_ts, total_records, user_msg_count, assistant_turns, file_size_kb, has_summary, indexed_at) — tracks metadata for each session
2. **`user_messages` table** (columns: session_id, file_path, msg_index, timestamp, content, word_count) — denormalized cache of user messages for quick search

#### JSONL Record Structure and Content Blocks

JSONL records represent discrete events in a session. Each record is a JSON object on a single line, parsed by `_iter_records()` (line 108-130). The function silently skips empty lines and invalid JSON.

**Record types and fields (verified from code):**

- **User-role records** (`record["type"] == "user"`, line 254):
  - `timestamp`: ISO 8601 string
  - `sessionId`: session UUID
  - `message.content`: string OR list of content blocks
  - Content blocks are dicts with `type` field (e.g., `{"type": "text", "text": "..."}`)
  - **Tool result blocks** appear in user records as content blocks:
    - `type == "tool_result"` — identifies a tool result block
    - `is_error`: boolean — `True` if the tool call failed, `False` or absent if successful
    - `tool_use_id`: string — matches the `id` field of a corresponding `tool_use` block from an assistant record
    - `content`: string or list — the error message or result text
  - **Noise guard** (line 259): Records with key `"toolUseResult"` (legacy/alternative format) are also skipped in message extraction; they mark tool result data

- **Assistant-role records** (`record["type"] == "assistant"`, line 254):
  - `timestamp`: ISO 8601 string
  - `message.content`: list of content blocks (required for tool use extraction)
  - **Tool use blocks** appear in assistant records as content blocks:
    - `type == "tool_use"` — identifies a tool use block
    - `id`: string — unique identifier for this tool call; matched by `tool_use_id` in corresponding tool_result blocks
    - `name`: string — tool name (e.g., "Read", "Bash", "Write", "Edit", "Glob", etc.)
    - `input`: dict — tool arguments/parameters (may be nested)

#### Shared Helper Functions (Already Exist)

**`_extract_text(content)`** (line 68-91): Handles three content formats:
- Plain string: return as-is
- `None`: return empty string
- List of blocks: extract from dicts where `type == "text"`, join with newlines

**`_is_noise(text)`** (line 94-105): Returns `True` if stripped text starts with any prefix in `_NOISE_PREFIXES` (line 58-65: `"<local-command-caveat>"`, `"<bash-stdout>"`, `"<tool_use_error>"`, `"<task-notification>"`, `"<command-message>"`, `"<system-reminder>"`). Used to filter system/metadata messages before indexing.

**`_open_db()`** (line 194-206): Opens DuckDB connection at `CACHE_DIR / "session-index.duckdb"`, creates parent dirs, runs DDL schema.

**`_get_table_width(table)`** (line 346-362): Measures Rich table width unconstrained (width=9999) to prevent wrapping.

### For New Feature Implementation: Four Query Commands

The four new commands (`errors`, `tools`, `irritation`, `current-path`) extend the existing CLI with behavioral analysis views not currently available. They all reuse the established patterns from existing commands but focus on different JSONL data:

#### Task T2: `cmd_errors` — Tool Error Enumeration

Extracts and lists tool errors from a session. Uses a two-pass pattern:

1. **Pass 1** over all records: Build a map `dict[tool_use_id: str, tool_name: str]` by scanning assistant records for `tool_use` blocks
2. **Pass 2** over all records: Find user records with `message.content[]` blocks where `type == "tool_result"` AND `is_error is True`; extract timestamp, tool_use_id, error content; resolve tool_name from Pass 1 map

Output: Rich-formatted section with timestamp + tool name + error excerpt, or raw tab-separated lines.

#### Task T3: `cmd_tools` — Tool Usage Statistics

Similar two-pass approach:

1. **Pass 1**: Collect `{id, name, timestamp}` from all `tool_use` blocks in assistant records
2. **Pass 2**: Correlate `tool_result` blocks from user records; classify by `is_error` boolean; aggregate into per-tool-name stats

Output: Rich table (Tool Name, Total Calls, Successes, Failures, Unmatched) or raw tab-separated header + rows.

#### Task T4: `cmd_irritation` — User Frustration Signal Detection

Two separate analyses:

1. **Correction phrase detection**: Scan user records (excluding `"toolUseResult"` key), extract text via `_extract_text()`, skip noise via `_is_noise()`, check each phrase from `_CORRECTION_PHRASES` case-insensitively in text.lower()
2. **Stuck loop detection**: Scan assistant records chronologically, extract `tool_use` blocks, compute identity key `f"{block['name']}:{hashlib.md5(json.dumps(block.get('input', {}), sort_keys=True).encode()).hexdigest()[:8]}"`, track consecutive runs, emit signal when count reaches `_STUCK_LOOP_THRESHOLD`

Output: Two sections (Correction Phrases + Stuck Tool Loops) or raw type-prefixed tab-separated lines.

#### Task T5: `cmd_current_path` — Live Session Path Resolution

**Unique pattern** — no session_id argument, no DuckDB query. Uses environment variable + pure filesystem check:

1. Read `CLAUDE_SESSION_ID` from `os.environ`; fail if empty
2. Compute encoded CWD: `str(Path.cwd()).replace("/", "-").lstrip("-")`
3. Construct path: `PROJECTS_DIR / encoded / f"{session_id}.jsonl"`
4. If exists: print path; if not, print diagnostics to stderr
5. Default output: raw path (agent-facing); `--rich` flag for human display

### Discovered During Implementation

_Session Date: 2026-03-11_

During implementation, we discovered that the architect spec described extraction logic as inline two-pass loops inside each command function, but the implementation agent factored all extraction logic into dedicated private helper functions instead. This wasn't a surprise — the spec explicitly permitted dataclass or TypedDict or NamedTuple as alternatives for data models — but the actual form taken (plain tuples returned from private helpers rather than dataclasses) was not anticipated. The refactor produces cleaner, independently-testable units.

**Key Discoveries:**

1. **Private helper extraction pattern used universally**: Rather than embedding two-pass logic inside each command, the implementation extracted all extraction and aggregation logic into private helpers: `_build_tool_name_map()`, `_collect_errors()`, `_collect_tool_uses()`, `_collect_tool_results()`, `_aggregate_tool_stats()`, `_collect_correction_phrases()`, and `_collect_stuck_loops()`. Future commands in this file should follow this pattern — each command orchestrates helpers, not raw loops.

2. **Dataclass data models from spec not used**: The architect spec defined `ToolStats`, `ToolError`, `CorrectionSignal`, and `StuckLoopSignal` dataclasses (section 5.2). The implementation returned plain tuples from all helpers instead. The `@dataclass` decorator is present in imports (used for `_FileStats` elsewhere in the file) so it was available — the agent chose tuples for their lightweight fit with sorted/iterated patterns.

3. **`usedforsecurity=False` added to `hashlib.md5()`**: The implementation calls `hashlib.md5(..., usedforsecurity=False)` (line 1088-1090). This is the correct pattern for Python 3.9+ when MD5 is used as a fingerprint (not a security primitive) — it suppresses FIPS-mode warnings. The architect spec did not mention this kwarg. Future implementations using `hashlib.md5` for non-security purposes should include `usedforsecurity=False`.

4. **`cmd_errors` Rich output does not truncate at 200 chars**: The architect spec (section 4.3) described Rich mode output as "truncated to ~200 chars with ellipsis if longer". The implementation outputs full error content without truncation. This aligns with the repository's "No Invented Limits" policy — arbitrary content truncation was removed. The raw mode spec (full content with newlines replaced by spaces) was implemented correctly.

5. **`_collect_stuck_loops` emits one signal per additional occurrence beyond threshold**: The spec said "emit a stuck-loop signal" when count reaches `_STUCK_LOOP_THRESHOLD`. The implementation uses `count >= _STUCK_LOOP_THRESHOLD` (checking on each iteration), meaning a loop of depth 5 produces 3 signals (at counts 3, 4, 5). This provides more information than a single signal and was a deliberate choice — the alternative (emit once then stop tracking) would miss worsening loops.

6. **`cmd_current_path` parameter variable name differs from spec**: The spec used `rich_flag` as the Python parameter name; the implementation uses `rich`. The CLI behavior is identical — Typer binds to `--rich` option string regardless of the variable name. No behavioral impact.

#### Updated Technical Details

- Private helper function signatures (all accept `records: list[dict]`, return typed tuples):
  - `_build_tool_name_map(records) -> dict[str, str]`
  - `_collect_errors(records, tool_name_map) -> list[tuple[str, str, str]]` — `(timestamp, tool_name, error_content)`
  - `_collect_tool_uses(records) -> list[tuple[str, str]]` — `(tool_use_id, tool_name)`
  - `_collect_tool_results(records) -> dict[str, bool]` — `{tool_use_id: is_success}`
  - `_aggregate_tool_stats(tool_uses, matched_ids) -> list[tuple[str, int, int, int, int]]` — `(name, total, successes, failures, unmatched)`
  - `_collect_correction_phrases(records) -> list[tuple[str, str, str]]` — `(timestamp, phrase, excerpt)`
  - `_collect_stuck_loops(records) -> list[tuple[str, str, int, str]]` — `(first_timestamp, tool_name, count, input_hash)`
- `hashlib.md5(..., usedforsecurity=False)` is the correct pattern for non-security MD5 in Python 3.9+
- Error content in `cmd_errors` Rich output is not truncated (full content displayed)

#### Gotchas for Future Developers

- When adding new commands to `session_query.py`, follow the helper extraction pattern: commands orchestrate, helpers extract. Do not embed raw JSONL iteration loops inside command functions.
- The `usedforsecurity=False` kwarg on `hashlib.md5()` prevents FIPS-mode failures in restricted environments. Always include it for fingerprinting/equality use cases.
- `_collect_stuck_loops` emits a signal on EVERY iteration where `count >= threshold`, not just once at the boundary. If you need exactly one signal per stuck-loop episode, you need to track whether the signal was already emitted for the current run.
