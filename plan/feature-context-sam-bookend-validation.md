## Original Request

SAM bookend validation -- baseline capture (T0) and verification gate (TN) as mandatory plan tasks. The SAM pipeline's feature-verifier checks structural properties (task count, field presence) but does not execute acceptance criteria commands. A converter that destroyed content passed all validation gates (session 2026-03-15, commits 03a806da and 3da1780e). Fixes #718.

---

## Core Intent Analysis

### WHO (Target Users)

SAM pipeline users -- developers who run `/implement-feature` and `/complete-implementation` to execute plans. The T0/TN tasks are transparent to the user (auto-generated by swarm-task-planner), but the user benefits from behavioral verification that catches defects before completion is declared.

### WHAT (Desired Outcome)

Every SAM plan automatically includes two bookend tasks that bracket implementation work:

1. **T0 (baseline capture)** -- runs acceptance criteria commands against the current codebase before any implementation task starts. Records what passes and what fails.
2. **TN (verification gate)** -- runs the same commands after all implementation tasks complete. Compares results against T0 baseline. Surfaces regressions.

Success: a plan that introduces behavioral regressions (e.g., content destruction) is blocked from completion. The TN verification report shows exactly which checks changed state.

### WHEN (Trigger Conditions)

- T0 triggers when `/implement-feature` starts and the plan contains structured acceptance criteria with `check_command` fields.
- TN triggers after all implementation tasks complete, before `/complete-implementation` quality gates run.
- The feature is relevant whenever a plan has executable acceptance criteria (not just prose descriptions).

### WHY (Problem Being Solved)

The current SAM pipeline validates structural properties only -- file existence, task count, field presence. It does not execute acceptance criteria commands. This means a feature can pass all validation gates while silently destroying data, producing wrong output, or omitting behavior.

Observed instance (2026-03-15): three legacy-format readers stripped markdown body content during YAML conversion. The feature-verifier confirmed structural correctness. Content loss was discovered only by manual human inspection. Commits 03a806da and 3da1780e fixed the data loss, but the pipeline gap remains.

## Use Scenarios

### Scenario 1: New feature plan with executable acceptance criteria

**Actor**: Developer running `/add-new-feature` then `/implement-feature`
**Trigger**: The developer describes a feature with acceptance criteria that include runnable commands (e.g., `uv run pytest tests/test_conversion.py -v`)
**Goal**: The SAM pipeline automatically verifies that the implementation satisfies the acceptance criteria without human spot-checking.
**Expected Outcome**: `swarm-task-planner` generates T0 and TN tasks alongside implementation tasks. T0 runs first and produces `plan/T0-baseline-{slug}.yaml`. Implementation tasks run. TN runs last and produces `plan/TN-verification-{slug}.yaml`. If no regressions, `/complete-implementation` proceeds normally.

### Scenario 2: Implementation introduces a behavioral regression

**Actor**: Developer running `/implement-feature` on a plan where an implementation task introduces a defect
**Trigger**: A task's implementation breaks something that was passing at T0 (e.g., content destruction, wrong output format)
**Goal**: The regression is caught automatically before the feature is declared complete.
**Expected Outcome**: TN detects that a check_command that exited 0 at T0 now exits non-zero. The TN verification report shows `status: regressed` for that criterion. `/complete-implementation` is blocked with the specific failing command and stdout diff.

### Scenario 3: Pre-existing failures that remain after implementation

**Actor**: Developer on a plan where some acceptance criteria were already failing before implementation
**Trigger**: T0 records that certain check_commands exit non-zero (pre-existing failures)
**Goal**: Pre-existing failures do not block completion, but newly introduced failures do.
**Expected Outcome**: TN reports `status: pre-existing-fail` for criteria that failed at both T0 and TN. Only `status: regressed` (passing at T0, failing at TN) blocks completion. The developer sees which checks improved (`newly-passing`) and which were unchanged.

### Scenario 4: Plan without executable acceptance criteria

**Actor**: Developer running `/add-new-feature` for a documentation-only or configuration-only change
**Trigger**: The plan's acceptance criteria are prose descriptions without `check_command` fields
**Goal**: The pipeline works as it does today -- no T0/TN tasks are generated.
**Expected Outcome**: `swarm-task-planner` does not generate T0/TN tasks because no structured `AcceptanceCriterion` entries with `check_command` exist. The existing workflow is unaffected.

## Questions Requiring Resolution

### Q1: Scope of criteria execution -- task-level vs plan-level

- **Category**: Scope
- **Gap**: The Plan model has `acceptance_criteria` and individual Task models also have `acceptance_criteria`. The groomed item mentions "the plan's structured acceptance criteria" but the swarm-task-planner writes per-task criteria.
- **Question**: Should T0/TN execute criteria defined at the plan level, at individual task level, or both? If both, how are they aggregated?
- **Options**:
  - A) Plan-level only -- T0/TN execute criteria from the Plan model's `acceptance_criteria` field
  - B) Task-level only -- T0/TN aggregate all `check_command` entries from all tasks
  - C) Plan-level primarily, with task-level opt-in for tasks that have structured criteria
- **Why It Matters**: Determines where `check_command` fields are authored and how the swarm-task-planner generates them.
- **Resolution**: _pending_

### Q2: TN integration point with /complete-implementation

- **Category**: Integration
- **Gap**: TN is described as running "after all implementation tasks" but before `/complete-implementation`. It is unclear whether TN is a task inside the plan (dispatched by `/implement-feature`) or a phase inside `/complete-implementation`.
- **Question**: Is TN a regular task in the plan (with its own agent, dispatched like any other task) that must complete before `/complete-implementation` is invoked? Or is TN a new phase added to the beginning of `/complete-implementation`?
- **Options**:
  - A) TN is a plan task -- the last task in the plan, with dependencies on all implementation tasks. `/implement-feature` dispatches it like any other task. `/complete-implementation` reads the TN YAML and gates on regressions.
  - B) TN is a new Phase 0 inside `/complete-implementation` -- it runs automatically when `/complete-implementation` is invoked, before Phase 1 (code review).
- **Why It Matters**: Option A keeps TN inside the existing task dispatch loop (simpler). Option B requires modifying `/complete-implementation` to run TN (more complex but ensures TN always runs even if the plan file does not include it).
- **Resolution**: _pending_

### Q3: Task ID for T0 and TN

- **Category**: Integration
- **Gap**: The current task ID pattern is `^[A-Za-z]?\d+(\.\d+)?[A-Za-z]?$`. The names "T0" and "TN" are used conceptually, but "TN" does not match the ID pattern (N is not a digit).
- **Question**: What task IDs should T0 and TN use?
- **Options**:
  - A) Use literal `T0` for baseline. Use a computed ID like `T99` or `T{max+1}` for verification gate.
  - B) Use `T0` and a new special ID that extends the pattern (e.g., `T0` for baseline, `TV` for verification).
  - C) Use regular numeric IDs (e.g., first and last in the sequence) with a special `agent` field value to identify them.
- **Why It Matters**: The ID pattern is validated by Pydantic (`Task.id` field) and used by dependency resolution. Non-conforming IDs will fail validation.
- **Resolution**: _pending_

### Q4: Handling non-existent test files at T0

- **Category**: Behavior
- **Gap**: At T0 time (before implementation), test files for the new feature typically do not exist. A criterion like `uv run pytest tests/test_new_feature.py -v` would fail at T0 with "file not found".
- **Question**: Is a "file not found" failure at T0 a valid pre-existing-fail? Or should criteria that reference not-yet-created files be skipped at T0 and only validated at TN?
- **Options**:
  - A) All criteria run at T0 regardless -- "file not found" is recorded as a pre-existing failure. At TN, if the test now exists and passes, status is `newly-passing`.
  - B) Criteria can be tagged with `expected_baseline: fail` to indicate they are expected to fail at T0, distinguishing them from unexpected pre-existing failures.
- **Why It Matters**: Determines whether T0 baseline is purely observational (record whatever happens) or has expectations built in.
- **Resolution**: _pending_

### Q5: Output capture limits

- **Category**: Behavior
- **Gap**: No specification for how much stdout/stderr to capture per criterion in the T0-baseline and TN-verification YAML files.
- **Question**: Should stdout/stderr be captured in full, truncated to a limit, or stored as file references?
- **Why It Matters**: Verbose test output (e.g., pytest -v on a large test suite) could produce megabytes of output, making the YAML files unwieldy. But truncation loses diagnostic information.
- **Resolution**: _pending_

### Q6: AcceptanceCriterion model -- supplement or replace

- **Category**: Scope
- **Gap**: The existing `acceptance_criteria` field on `Task` is `str` (plain markdown). The new `AcceptanceCriterion` is a structured model. The groomed item says "alongside" but does not specify the field name.
- **Question**: Does the structured `AcceptanceCriterion` list live in a new field (e.g., `structured_acceptance_criteria` or `check_criteria`), or does the existing `acceptance_criteria` field become a union type (`str | list[AcceptanceCriterion]`)?
- **Options**:
  - A) New field alongside existing one -- `acceptance_criteria: str` stays for prose, new `check_criteria: list[AcceptanceCriterion]` for executable checks
  - B) Union type -- `acceptance_criteria` accepts either a string or a list of structured objects
  - C) Replace -- `acceptance_criteria` becomes `list[AcceptanceCriterion]` only, with a `description` sub-field for prose
- **Why It Matters**: Backward compatibility with all existing plans that use `acceptance_criteria` as a plain string. Option B or C require migration. Option A is additive.
- **Resolution**: _pending_
