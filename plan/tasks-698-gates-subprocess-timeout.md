### How This Currently Works: subprocess Timeout in run_quality_gates()

When a caller invokes `run_quality_gates(commands, mode=..., cwd=..., timeout=...)` from `plugins/development-harness/dispatch_schema/gates.py`, the function executes each command string in sequence through this flow:

1. **Tokenization**: Each command string is split via `shlex.split()` into tokens
2. **Executable Resolution**: The first token (binary name) is resolved via the cached `_resolve_executable()` helper, which calls `shutil.which()`. This returns an absolute path or None.
3. **Command-Not-Found Handling**: If the binary is not found (None result), a `CommandResult` with `exit_code=127` and stderr message is created without raising an exception. This follows the established pattern at lines 71-72.
4. **Subprocess Execution**: If the binary is found, the code constructs `resolved_tokens` by concatenating the absolute path with the remaining arguments, then calls `subprocess.run()` at line 75 with `capture_output=True, text=True, cwd=cwd, check=False`. **Currently missing: the `timeout` parameter is not passed.**
5. **Result Capture**: The completed process is converted to a `CommandResult` with `exit_code=completed.returncode`, `stdout=completed.stdout`, `stderr=completed.stderr`
6. **Mode-Based Collection**: The result is appended to the results list. If mode is `FAIL_FAST` and the command failed, the loop breaks; otherwise, it continues to the next command.
7. **Return**: A `GateResult` is returned with `passed=all(r.passed for r in results)`, the collected results list, and the mode.

**The Bug at Line 75**: The `subprocess.run()` call lacks the `timeout` parameter. When a gate command hangs (never completes), the entire `run_quality_gates()` call blocks indefinitely, preventing the caller from enforcing time budgets or failing fast on stuck processes. This breaks dispatch plans in scenarios where a quality gate command (e.g., a linter, test suite, or build tool) misbehaves or enters an infinite loop.

**Current State of Exception Handling**: Lines 71-72 handle the specific case where `shutil.which()` returns None. However, there is no handler for `subprocess.TimeoutExpired`, which is raised by `subprocess.run()` when a timeout is exceeded. The architecture spec defines this should be caught and converted to a `CommandResult` with `exit_code=124` (Unix `timeout(1)` convention), matching the pattern of the command-not-found handler.

### Discovered During Implementation

_Session Date: 2026-03-21_

Implementation aligned with the architecture spec with no surprises. A few details are worth capturing for similar future patches.

**Key Discoveries:**

1. **subprocess.run kills child before raising TimeoutExpired**: ADR-002 was validated in practice. No `Popen` refactoring was needed. `subprocess.run` with `timeout=` sends SIGKILL internally and raises `TimeoutExpired` after the child is gone — the `except` block receives a dead process, not a live one. Future implementors can rely on this: no cleanup code is needed after catching `TimeoutExpired` from `subprocess.run`.

2. **Pre-existing test count was 32, not 8**: AC3 references "8 existing test classes" (accurate), but the actual pre-existing test method count was 32. The new `TestSubprocessTimeoutContract` class added 6 methods, bringing the total to 38. The "8 test classes" framing was correct; "all 38 tests pass" in the implementation summary refers to the final post-implementation count.

3. **shutil.which must be patched in every timeout test**: Every test in `TestSubprocessTimeoutContract` patches `dispatch_schema.gates.shutil.which` to return a non-None path. Without this, the `executable is None` branch fires at line 79 and the `subprocess.run` call (and thus `TimeoutExpired`) is never reached. This is the same requirement as `TestSubprocessInvocationContract` but is not obvious from reading the production code alone.

#### Updated Technical Details

- Final function signature: `(commands: list[str], *, mode: GateRunMode = GateRunMode.FAIL_FAST, cwd: Path | None = None, timeout: float = 300.0) -> GateResult`
- `subprocess.run` call site: `subprocess.run(resolved_tokens, capture_output=True, text=True, cwd=cwd, check=False, timeout=timeout)`
- Timeout result: `CommandResult(command=command, exit_code=124, stdout="", stderr=f"command timed out after {timeout}s")`
- Mock target for timeout tests: `mocker.patch("dispatch_schema.gates.subprocess.run", side_effect=subprocess.TimeoutExpired(cmd=["pytest"], timeout=300.0))`

#### Gotchas for Future Developers

- `shutil.which` must be patched alongside `subprocess.run` in timeout tests — the `executable is None` guard at line 79 short-circuits before `subprocess.run` is ever called
- The `timeout` parameter is keyword-only — callers must use `run_quality_gates(commands, timeout=5.0)`, not positional
- `exit_code=124` follows the Unix `timeout(1)` convention — consistent with how CI tools interpret timed-out subprocess results
