### How This Currently Works: SAM Task Dispatch and Acceptance Criteria

When a developer runs `/implement-feature` on a SAM plan, the orchestrator queries the plan file via `implementation_manager.py ready-tasks` to determine which tasks are ready to dispatch. A task is ready when its status is `not-started` and all tasks in its `dependencies` list have terminal status (`complete`, `deferred`, `skipped`).

The query returns a JSON list of ready tasks sorted by priority (1-5, lower = higher priority) and then by task ID. For each ready task, the orchestrator routes to the agent named in the task's `agent` field, loads any skills listed in the task's `skills` field, and invokes `/start-task` skill to begin execution. The `/start-task` skill:

1. Sets task status to `in-progress` and records a `started` timestamp
2. Reads the task file and linked architecture spec
3. Executes the task against its acceptance criteria
4. Implements the task objective
5. The task completion is marked by the `task_status_hook.py` `SubagentStop` hook, which sets status to `complete` and records `completed` timestamp

The hook system (PreToolUse, PostToolUse, SubagentStop) tracks task lifecycle events automatically. The `task_status_hook.py` script is invoked on every Write, Edit, Bash call (PostToolUse) to update the `last_activity` timestamp, and on SubagentStop to mark the task complete. No manual status updates are needed ŌĆö the hooks handle this.

The existing acceptance criteria in SAM are stored as plain markdown strings (`Task.acceptance_criteria: str` and `Plan.acceptance_criteria: str`). The feature-verifier agent reads these fields during `/complete-implementation` Phase 2 (feature verification) and checks structural properties: goal achievement, task count, file existence, function presence. It does NOT execute the acceptance criteria commands (e.g., running tests, comparing output).

**The Gap**: This means a feature can pass all validation gates (structural checks via feature-verifier) while destroying, corrupting, or omitting behavior. Observed example (session 2026-03-15, commits 03a806da and 3da1780e): three legacy-format readers in `packages/sam_schema/` stripped markdown body content during YAML conversion. The feature-verifier confirmed structural correctness (task count, field presence) but had no way to detect behavioral content loss.

### What T0 and TN Will Do

T0 (baseline capture) and TN (verification gate) are two new mandatory bookend tasks that bracket implementation work:

**T0 Execution** (first task in the plan):
- T0 is generated by `swarm-task-planner` with `dependencies: []`, `priority: 1`, `is_bookend: true`, `bookend_type: t0-baseline`
- When `/implement-feature` queries ready tasks, T0 is the only ready task (no dependencies, highest priority)
- The T0 agent reads `Plan.acceptance_criteria_structured` list (new field containing structured `AcceptanceCriterion` objects)
- For each `AcceptanceCriterion`, the T0 agent runs `check_command` via the Bash tool
- Results (exit code, stdout, stderr, timestamp, duration) are written to `plan/T0-baseline-{slug}.yaml`
- Non-zero exit codes are expected (for pre-existing failures) ŌĆö T0 agent does NOT fail on them
- After T0 completes, all other implementation tasks become ready and dispatch as before

**TN Execution** (last task in the plan, after all implementation tasks complete):
- TN is generated with `dependencies: [all non-bookend task IDs]`, `priority: 5`, `is_bookend: true`, `bookend_type: tn-verification`
- When all implementation tasks complete, TN becomes ready
- The TN agent reads the T0 baseline YAML and the same `Plan.acceptance_criteria_structured`
- For each criterion, TN re-runs the `check_command` and captures results
- TN computes a `CriterionStatus` per criterion using the 4-cell matrix:
  - T0 pass, TN pass = `passed`
  - T0 pass, TN fail = `regressed` (implementation broke something)
  - T0 fail, TN fail = `pre-existing-fail` (was already broken)
  - T0 fail, TN pass = `newly-passing` (implementation fixed something)
- Results are written to `plan/TN-verification-{slug}.yaml` with a top-level `verdict` field:
  - `FAIL` if any criterion has `status: regressed` (blocks completion)
  - `PASS` otherwise (allows completion)

**Integration with `/complete-implementation`**:
- Before Phase 1 (code review) starts, `/complete-implementation` reads `plan/TN-verification-{slug}.yaml`
- If `verdict: FAIL`, the skill blocks completion and displays the regressed criteria with their check commands and stdout diff
- If `verdict: PASS` or no TN file exists (plan has no structured criteria), proceed normally

### Key Integration Points and Assumptions

1. **Task ID Regex**: The existing pattern `^[A-Za-z]?\d+(\.\d+)?[A-Za-z]?$` matches `T0` (letter T, digit 0). For TN, the `swarm-task-planner` assigns the ID `T99` (or `T{max_task_number + 1}` if 99 conflicts). The `bookend_type` field provides semantic identification independent of the ID value.

2. **DependencyGraph Integration**: The existing `get_ready_tasks()` function in `core/query.py` uses `DependencyGraph` to compute readiness. T0 and TN are ordered correctly purely through their dependency declarations ŌĆö T0 has no dependencies (ready first), TN depends on all implementation tasks (ready last). No special-case code in the readiness logic is needed.

3. **Hook Compatibility**: The existing `task_status_hook.py` marks tasks complete via the `SubagentStop` hook. T0 and TN are regular tasks dispatched the same way, so they are auto-tracked without hook changes.

4. **Reader/Writer Backward Compatibility**: All new fields have defaults. Existing plans without structured criteria produce empty lists for `acceptance_criteria_structured` and False/None for the bookend fields. The feature is purely opt-in at the plan level.

5. **YAML Format Convention**: Structured criteria are written as YAML lists with kebab-case field names:
   ```yaml
   acceptance-criteria-structured:
     - criterion-id: AC-1
       check-command: "uv run pytest tests/test_conversion.py::test_body_preserved -v"
       expected-baseline: any
       expected-final: pass
   ```

6. **Content from swarm-task-planner**: The `swarm-task-planner` agent must extract executable acceptance criteria from the architecture spec's acceptance criteria section. If no structured criteria are present in the spec, the planner does not generate T0/TN tasks.

### Why This Architecture Works

The bookend system leverages existing SAM infrastructure without requiring modifications to core dispatch logic:

- **T0 ordering** is automatic via priority/dependency sorting ŌĆö no special case needed
- **TN ordering** is automatic via TN's dependencies on all implementation tasks ŌĆö no special case needed
- **Acceptance Criteria** are stored as Pydantic models with kebab-case aliases ŌĆö readers/writers already handle variant formats
- **Command execution** is delegated to agent-level Bash tool ŌĆö no new execution contexts needed
- **YAML I/O** uses existing `ruamel.yaml` infrastructure ŌĆö no new dependencies
- **Validation** is a helper class (`BookendValidator`) alongside the existing `DependencyGraph` ŌĆö no coupling required

The design is purely additive: existing plans without structured criteria behave exactly as before. T0 and TN are generated only when `acceptance_criteria_structured` is non-empty.
