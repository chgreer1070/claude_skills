### How the Research Freshness System Currently Works

The research system in this repository is a three-layer architecture: an orchestrator skill
(`/research-curator` or `/refresh-research`), a worker agent (`@research-curator`), and a
shared template (`entry-template.md`) that defines what every research entry looks like. These
layers communicate through structured file content — research entries in `./research/` — and
through Mermaid flowcharts that function as executable instruction sets (declared binding at
`.claude/skills/research-curator/SKILL.md` lines 7-10).

**Freshness data** lives entirely in the `## Freshness Tracking` section at the bottom of each
entry file. Three fields: `Last Verified` (ISO 8601 date), `Version at Verification` (verbatim
version string), and `Next Review Recommended` (ISO 8601 date). These are written by the
`@research-curator` agent when it creates or reruns an entry. The template that defines their
format is `.claude/skills/research-curator/references/entry-template.md` lines 168-175.

**The staleness decision** in `/refresh-research` happens in Step 1 (lines 30-40). The
flowchart has two terminal outcomes from the `HasFreshness -->|Yes|` branch: both `STALE` paths
(past review date; or 6 months old) continue to Step 2 target set. The `Fresh[FRESH: skip]`
terminal is a leaf with no outgoing edge — fresh entries are permanently excluded from the
STALE target set under `--stale`. Under `--all`, Step 2 keeps everything (the staleness filter
is bypassed), so fresh entries do reach agents. The `FRESH: skip` terminal is Blocking Site #2.

**The batch duplicate skip** in `/research-curator` Batch Mode happens at line 166: "If found,
skip with info message suggesting `--rerun` instead." This is a one-liner that gives no delta,
shows no freshness data, and requires the user to know the exact file path to bypass. This is
Blocking Site #1.

**The `@research-curator` agent `--rerun` mode** (agent file lines 324-338) is entirely
unchanged by this feature. It already reads the existing entry, re-gathers fresh data from
primary sources, notes what changed vs. confirmed unchanged, and updates the Freshness Tracking
section. The fix in this feature is about making the orchestrator REACH `--rerun` more often
(by removing the batch skip gate), not about changing what happens when `--rerun` is received.

### What This Feature Changes and Why

Issue #444 reframes freshness dates from stop conditions into informational delta indicators.
The resolved design decisions (from `plan/architect-research-freshness-delta.md` section 1):

- **Q1**: `--stale` continues to exclude fresh entries from agent spawning. Only the label and
  per-entry listing change — automation safety is preserved.
- **Q2**: Batch Mode auto-reruns existing entries (submitting a URL = intent to refresh).
- **Q3**: Delta fields are: days since Last Verified (integer), date string, version string, and
  days until next review (integer). Days Old column added to Step 1 inventory table.
- **Q4**: The `@research-curator` agent is NOT modified. Only the three documentation files change.
- **Q5**: `--dry-run` output gains the Days Old column from Step 1 inventory; the Skipped(fresh)
  listing in Step 2 also appears under `--dry-run` since it is emitted before the dry-run stop.

Three Markdown documentation files change. No Python scripts, no agent files, no new flags.

### Delta Computation Rules (For Reference in Implementation Text)

When writing the Duplicate Detection procedure or Step 2 listing, the computation is:

```text
days_old          = today - last_verified_date   (integer, days; render as 0 if negative)
days_until_review = next_review_date - today     (integer, days)
```

Source field: `## Freshness Tracking` table, rows `Last Verified` and `Next Review Recommended`.
Parse as ISO 8601 `YYYY-MM-DD`. If not parseable, treat as absent.

Edge cases from arch spec section 7:
- No Freshness Tracking section → emit Template B (data unavailable), proceed with `--rerun`
- Last Verified absent or malformed → same as no Freshness Tracking (Template B)
- Next Review absent → render as `review date unknown (last: YYYY-MM-DD, vX.Y.Z)`
- Last Verified in the future → render days_old as `0` (not negative)
- Version absent → omit version token from all templates
