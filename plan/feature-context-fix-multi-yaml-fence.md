## Original Request

The swarm-task-planner agent generates task files with YAML inside fenced code blocks (` ```yaml ... ``` `) which the parser cannot read, causing null agents and NOT_STARTED defaults. `has_yaml_frontmatter()` in `task_format.py` requires `content.startswith("---\n")` -- fenced blocks fail this check.

Three-part fix:

1. Agent template fix: Remove ` ```yaml ` wrapper from task structure template in both swarm-task-planner agent files
2. Format validator: Add validation to `parse_task_content()` that detects fenced YAML blocks and raises actionable errors
3. Script-only writes policy: Document in `TASK_FILE_FORMAT.md` that task data files are written only by designated scripts

Fixes #320.

## Core Intent Analysis

### WHO (Target Users)

1. **The SAM pipeline orchestrator** (`/implement-feature` skill) -- consumes task files to route tasks to agents.
2. **The swarm-task-planner agent** -- generates task files that must be parseable by `implementation_manager.py`.
3. **Repository maintainers** -- need reliable task file generation and actionable error messages when parsing fails.

### WHAT (Desired Outcome)

- Task files generated by `swarm-task-planner` are parseable by `implementation_manager.py` without manual intervention.
- When a task file contains fenced YAML blocks instead of raw YAML frontmatter, the parser produces an actionable error message instead of silently defaulting to `agent=None` and `status=NOT_STARTED`.
- The format spec (`TASK_FILE_FORMAT.md`) explicitly states that task data files must be written by designated scripts, preventing agents from generating malformed files.

### WHEN (Trigger Conditions)

This bug manifests when:

1. The swarm-task-planner agent generates a new task file based on an architecture spec.
2. The agent follows the template in its own agent file (lines 249-265 in both planner agents), which wraps YAML frontmatter inside a ` ```yaml ` code fence.
3. The orchestrator then calls `implementation_manager.py status` or `ready-tasks`, which calls `parse_task_content()` on the generated file.
4. The parser fails to extract metadata, defaults all tasks to `agent=None`, `status=NOT_STARTED`, and the orchestrator cannot route tasks to the correct agent.

### WHY (Problem Being Solved)

Silent parsing failure. The parser does not error on fenced YAML -- it silently falls through to legacy parsing, which may also fail silently by producing tasks with default (wrong) values. This causes:

- Tasks assigned to `None` agent (orchestrator cannot delegate)
- Tasks stuck at `NOT_STARTED` even when they have real status values in the fenced YAML
- Debugging requires manual inspection of generated files to discover the format mismatch

## Codebase Research

### Similar Patterns Found

#### Pattern 1: YAML Frontmatter Detection (`has_yaml_frontmatter`)

- **Location**: `plugins/python3-development/skills/implementation-manager/scripts/task_format.py:66-92`
- **Relevance**: This is the root cause function. It checks `content.startswith("---\n")` and looks for a closing `\n---\n` delimiter. Content wrapped in ` ```yaml\n---\n...\n---\n``` ` will fail the `startswith` check because the content starts with ` ```yaml\n `, not `---\n`.
- **Reusable**: The detection logic is correct for properly formatted frontmatter. The gap is that it does not detect and warn about near-miss formats (fenced YAML blocks).

#### Pattern 2: `parse_task_content()` Silent Fallthrough

- **Location**: `plugins/python3-development/skills/implementation-manager/scripts/implementation_manager.py:645-693`
- **Relevance**: When `has_yaml_frontmatter()` returns `True` (for plan-level frontmatter) but `parse_task_from_frontmatter()` raises `ValueError` (because the plan-level YAML lacks `task`/`title`/`status` fields), the function silently falls through to legacy markdown parsing (line 666-667: `except (ValueError, TypeError): pass`). If legacy parsing also finds nothing (because headers use `### T1:` instead of `### Task T1:`), the function returns an empty list with no error.
- **Reusable**: The dual-format detection is a good pattern but needs a third path for the fenced-YAML case and better error reporting on empty results.

#### Pattern 3: Swarm Task Planner Template (python3-development)

- **Location**: `plugins/python3-development/agents/swarm-task-planner.md:249-265`
- **Relevance**: The template uses ` ````markdown\n```yaml\n---\n ` nesting. The agent treats this as a documentation example showing proper YAML frontmatter format. But when the agent generates task files, it reproduces this fenced structure literally, wrapping the YAML in code fences.
- **Reusable**: The template content (field names, section headers) is correct. Only the code fence wrapping needs removal.

#### Pattern 4: Swarm Task Planner Template (development-harness)

- **Location**: `plugins/development-harness/agents/swarm-task-planner.md:248-264`
- **Relevance**: Same template with same fencing issue. Uses `role:` instead of `agent:` (development-harness uses role resolution via language manifests). Same fix needed.
- **Reusable**: Same as Pattern 3.

#### Pattern 5: Existing Affected Task Files

- **Location**: `plan/tasks-4-validate-orchestrator-discipline.md:265-279`
- **Relevance**: Real-world example of the bug. Individual tasks in this file have YAML frontmatter wrapped in ` ```yaml ` code fences. Additionally, these tasks use `### T1:` headers (without the word "Task"), so legacy parsing also fails to match them. The file-level YAML frontmatter (lines 1-12) is a plan descriptor, not a task.
- **Reusable**: Serves as a test case for validation.

#### Pattern 6: `_create_empty_task_data()` Default Values

- **Location**: `plugins/python3-development/skills/implementation-manager/scripts/implementation_manager.py:605-626`
- **Relevance**: When the parser falls through to legacy format and matches a header, it creates a task with `agent=None`, `status=TaskStatus.NOT_STARTED`, and `priority=TaskPriority.CRITICAL`. These defaults mask the real values that exist inside the fenced YAML blocks, creating a hard-to-diagnose discrepancy.
- **Reusable**: The defaults are appropriate for truly missing fields. The problem is that the parser reaches this code path when the real data is present but unreadable.

### Existing Infrastructure

1. **`parse_yaml_frontmatter()`** in `task_format.py:95-134` -- correctly parses `---` delimited YAML. Could be extended to strip fenced wrappers before parsing, or to detect fenced wrappers and raise a descriptive error.

2. **`split_task_file.py`** at `plugins/python3-development/scripts/split_task_file.py` -- already imports `has_yaml_frontmatter` from `task_format` and uses `parse_task_file` from `implementation_manager`. This script would also be affected by fenced YAML blocks.

### Q2: Should the validator auto-strip fenced YAML or raise an error?

- **Category**: Behavior
- **Gap**: The request says "raises actionable errors" but an alternative is to auto-strip the fences and parse the contained YAML.
- **Question**: When `parse_task_content()` detects fenced YAML blocks, should it (A) raise a ValueError with an actionable message, or (B) strip the fences and parse the YAML successfully, with a warning logged?
- **Options**:

- **Why It Matters**: Strict errors force the template to be fixed but may break in-flight work. Auto-strip is pragmatic but may delay fixing the root cause.
- **Resolution**: _pending_

### Q3: Which scripts are "designated" for the script-only writes policy?
