### How the SAM Task System Currently Works

The SAM (Structured Agent-Managed) feature implementation pipeline uses task/plan files to track feature decomposition and execution. When a user invokes `/add-new-feature` with a feature description, the system runs a multi-stage planning workflow that culminates in writing a task plan file (typically `plan/tasks-{N}-{slug}.md`). The plan file contains structured metadata (feature name, version, description) followed by individual task sections with YAML frontmatter.

Currently, **five independent Python scripts parse task files**, each with its own format assumptions and parsing logic:

1. **implementation_manager.py** (`plugins/python3-development/skills/implementation-manager/scripts/`) — Queries task status and readiness. Defines `Task` dataclass (lines 117-143) and `TaskStatus` enum. Implements `parse_task_file()` (lines 547-575) which dispatches to `parse_task_content()` (lines 479-544) for multi-task files or `parse_task_from_frontmatter()` (lines 403-471) for single tasks.

2. **task_format.py** (`plugins/python3-development/skills/implementation-manager/scripts/`) — Shared YAML parsing utilities. Provides `parse_yaml_frontmatter()` (lines 132-171) for extracting YAML blocks, `normalize_status()` (lines 275-320) for status value normalization with `STATUS_MAP` dict (lines 28-45), and `update_yaml_field()` (lines 179-241) for in-place field updates that preserve comments and formatting.

3. **task_status_hook.py** (`plugins/python3-development/skills/implementation-manager/scripts/`) — Updates task status during execution. Uses **regex-based section finding** (`find_task_section()`, lines 245-296) independently from the other parsers. Called via `SubagentStop` hook to set status to `complete` and `PostToolUse` hook to update `last_activity` timestamp.

4. **split_task_file.py** (`plugins/python3-development/scripts/`) — Converts single-file task plans to directory format (one task per `.md` file). Has its own parsing logic.

5. **migrate_task_format.py** (`plugins/python3-development/scripts/`) — Migrates legacy markdown task files to YAML frontmatter format. Has its own parsing logic.

**The core problem**: These five parsers operate **independently with no shared contract**. When `swarm-task-planner` (the agent that writes plan files) produces a format variant that one parser doesn't anticipate, that parser silently returns zero tasks instead of reporting an error. As observed with `plan/tasks-1-backlog-state-reconciliation.md`, a task file using global-manifest + prose-body format returned `total_tasks: 0` because the implementation_manager parser didn't handle that variant.

### Existing Parser Patterns (What Gets Reused)

All current parsers follow **YAML frontmatter format**: files starting with `---\n` delimiter, followed by key-value YAML metadata, closed with another `---\n`. The YAML frontmatter contains required fields (`task`, `title`, `status`) and optional fields (agent, dependencies, priority, complexity, skills, started, completed, etc.). Markdown body content (objectives, requirements, context) follows after the closing delimiter.

**Status normalization** is implemented in `task_format.py:275-320` with `STATUS_MAP` dict that handles:
- Space-separated variants: "NOT STARTED" → "not-started"
- Emoji markers: ":white_check_mark:" → "complete"
- Default fallback: unknown values → "not-started"

**Field update strategy** (task_format.py:179-241) uses regex-based line scanning to update individual YAML fields **without re-serializing the entire file**. This preserves comments, field order, and body content. The function splits on `---\n`, finds the target field line (or inserts before closing delimiter), and returns updated content.

**Task ID validation** uses regex pattern `^[A-Za-z]?\d+(\.\d+)?$` (task_format.py:47) supporting: `1`, `1.1`, `T1`, `T2.3`, etc.

**Timestamp handling** coerces ISO 8601 strings to `datetime` objects during YAML parsing, then converts back to strings for storage (implementation_manager.py:323-337). This handles ruamel.yaml's automatic type inference.

**Dependency parsing** (implementation_manager.py:273-300) normalizes multiple input formats: "Task 1, Task 2" (legacy), "T1, T2" (current), or "1.1, 2.1" (dotted). Uses regex to extract task IDs and validate them.

### For New Module Implementation: What Needs to Connect

The new `sam_schema` module replaces all five ad-hoc parsers with a **single shared Python package** (`packages/sam_schema/`) that becomes the sole interface for reading, writing, and querying task files. Existing consumers will import from this module instead of duplicating parsing logic.

**Integration points:**

1. **Models** — Pydantic `Task`, `Plan`, `TaskStatus`, etc. defined in `core/models.py`. Task ID validation, status enum, complexity enum all captured in type hints and validators. Replaces the stdlib `dataclass` in implementation_manager.py.

2. **Readers** — Format-specific readers (`readers/yaml_reader.py`, `readers/frontmatter_reader.py`, `readers/legacy_reader.py`, `readers/manifest_reader.py`) all normalize to the same Pydantic model. Format detection (`readers/detect.py`) routes based on file extension and content inspection. Replaces the multi-format parsing in implementation_manager.py and task_status_hook.py.

3. **Writers** — Pure YAML output (`writers/yaml_writer.py`) using ruamel.yaml for round-trip formatting preservation. Handles both single-file (under 500 lines) and directory (over 500 lines) formats. Replaces regex-based field updates in task_format.py.

4. **Query layer** (`core/query.py`) — `load_plan()`, `get_task()`, `list_tasks()`, `get_ready_tasks()`, `update_status()`, `get_plan_status()` functions that encapsulate all business logic. Returns structured data (Pydantic models, not raw dicts).

5. **Dependency resolution** (`core/dependencies.py`) — `DependencyGraph` class implementing cycle detection, readiness checks, terminal status definition. Replaces the ad-hoc dependency checking in implementation_manager.py.

6. **CLI and MCP servers** — Expose the same query operations via Typer CLI (`cli.py`: `sam read P1/T3`, `sam state P1/T3 complete`, `sam ready P1`, `sam status P1`) and FastMCP server (`server.py`: `sam_read`, `sam_state`, `sam_ready`, `sam_status`). Both delegate to the shared query layer.

**Patterns to follow:**

- **Status map**: Reuse the `STATUS_MAP` from task_format.py (lines 28-45) — normalize incoming status values to canonical enum values
- **Field aliases**: YAML uses kebab-case (`blocked-by`, `parallelize-with`, `analysis-method`); Python uses snake_case (`blocked_by`, `parallelize_with`, `analysis_method`). Pydantic `Field(alias="kebab-case")` handles the mapping.
- **Timestamp format**: Always ISO 8601 with timezone (e.g., "2026-03-14T15:30:00Z"). Pydantic `datetime` fields auto-parse; serialization via `use_enum_values=True` keeps strings.
- **Task ID pattern**: Validate with `pattern=r"^[A-Za-z]?\d+(\.\d+)?$"` in Pydantic field definition. Reuse this in `addressing.py` for CLI address parsing (`P1/T3`).

### Critical Dependencies and Constraints

**No breaking changes to consumers during migration** — Hook script (task_status_hook.py) must continue working with both old and new code paths initially. The module provides a `claim_task()` function (new to sam_schema) that guards against double-claiming via status check.

**File format versions** — The module supports three known format variants (per-task YAML frontmatter, global-manifest + embedded per-task blocks, legacy markdown with `## Task N:` headings). Additional variants discovered during codebase survey (e.g., global-manifest + prose body) must be handled. Schema gap detection reports missing optional fields when reading legacy formats.

**Address scheme** — CLI uses `P{N}/T{M}` notation (e.g., `sam read P1/T3`). Addressing module (`core/addressing.py`) resolves `P1` to a plan file path by matching filename pattern `tasks-1-*` (sequence number from filename) or `tasks-{slug}/*` (slug match). Full address parsing: `parse_address()` returns `(plan_ref, task_ref)` tuple.

**File write atomicity** — Use `tempfile.NamedTemporaryFile` with atomic rename for writes to prevent partial files on crash. This is standard practice but not currently used in task_format.py.

**Backward compatibility** — Legacy files (markdown with `## Task N:` headers, old YAML frontmatter formats) must be readable but not auto-converted on read. Conversion is explicit via `sam migrate P1` command. This preserves git history and file references.

### Discovered During Implementation

_Session Date: 2026-03-15_

During implementation, several patterns emerged that were not documented in the original context. These represent design choices made during execution that future implementers working with or extending `sam_schema` should know.

**Key Discoveries:**

1. **Shared YAML utility module emerged organically**: A `readers/_yaml_utils.py` module was created to hold `make_yaml()` and `load_yaml()` helpers shared by all four readers. This was not in the original plan, which specified each reader would use `ruamel.yaml` independently. The shared utility prevents duplicate `YAML(typ="rt")` instance creation and centralizes the `preserve_quotes=True` configuration. Future reader implementations must import from `_yaml_utils` rather than instantiating `YAML()` directly.

2. **Custom exceptions replaced by stdlib exceptions**: The architecture spec (T7) specified creating custom exception classes `TaskNotFoundError` and `ClaimError`. The implementation uses `KeyError` for not-found conditions and `ValueError` for invalid-state conditions (e.g., claiming a task not in `not-started` status). `AddressingError` was defined in `core/addressing.py` (not `core/query.py` as the spec suggested). The CLI catches these stdlib exceptions and maps them to user-friendly error messages. Future code should raise `KeyError` for missing tasks and `ValueError` for state violations — not import a custom exception class from `query.py`.

3. **`update_fields()` bulk API added alongside `update_field()`**: The YAML writer exposes both `update_field(file_path, task_id, field, value)` (single field) and `update_fields(file_path, task_id, fields: dict)` (multiple fields in one read-write cycle). The query layer (`query.py`) imports and uses `update_fields` (plural), not `update_field`. When updating more than one field at a time (e.g., status + last-activity timestamp), callers must use `update_fields` to avoid multiple file reads. The architecture spec only described `update_field`.

4. **Sync MCP tool handlers (see DN-1)**: T9 assumed `async def` was required by FastMCP. FastMCP 3.0.2 accepts synchronous handlers. All four MCP tools use `def` not `async def` to avoid ruff RUF029 lint errors. See the existing `## Divergence Notes / DN-1` section for the full record.

5. **`STATUS_MAP` includes "WONT FIX" mapping**: The context manifest listed six status values in the `TaskStatus` enum (`not-started`, `in-progress`, `complete`, `blocked`, `deferred`, `skipped`). The `STATUS_MAP` in `models.py` includes an additional entry: `"WONT FIX": "wont-fix"`. This mapping is in `STATUS_MAP` but there is no corresponding `WONT_FIX` enum member in `TaskStatus` — it appears to be a legacy normalization alias for edge cases. Future code should not treat `wont-fix` as a valid `TaskStatus` enum value.

#### Updated Technical Details

- `readers/_yaml_utils.py` provides `make_yaml() -> YAML` and `load_yaml(content: str) -> Any` — all four format readers import from here
- `AddressingError` is defined in `sam_schema.core.addressing`, not in `sam_schema.core.query`
- `yaml_writer.update_fields(file_path, task_id, fields: dict)` is the preferred multi-field update API
- `query.py` imports `update_fields` (not `update_field`) from `writers.yaml_writer`
- T10b (Consumer Migration) status is `in-progress` at session end — consumers not yet migrated

#### Gotchas for Future Developers

- When the writer spec says `update_field`, the actual callable in the implementation is `update_fields` (plural). The singular `update_field` also exists but the query layer uses the plural form.
- Do not import `TaskNotFoundError` or `ClaimError` from `sam_schema.core.query` — these classes do not exist. Catch `KeyError` and `ValueError` respectively.
- The `_yaml_utils.py` module is private (underscore prefix) but is a required import for any new reader that uses ruamel.yaml. Do not bypass it with a local `YAML()` instance.
