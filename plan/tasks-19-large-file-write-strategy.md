### How the Agent Large File Write Problem Works: Current State

The root failure mode was observed in commit 8beddfb, where the `swarm-task-planner` sub-agent
attempted to write `plan/tasks-14-console-forwarding-mcp-server-plugin.md` as a single Write tool
call with 57,800 characters (821 lines). The Write stalled or timed out, blocking the SAM planning
phase and requiring manual intervention. No failure was observed below 30K characters in session
history.

The Write tool schema (from `plugins/plugin-creator/skills/hooks-guide/references/claude-code.md`
line 279-285) accepts only two fields — `file_path: string` and `content: string` — with no
documented size constraint. There is no maximum content size anywhere in the tool documentation,
and no guardrail exists at the tool level to prevent oversized calls.

Sub-agents in the SAM workflow write output files directly using the Write tool. The orchestrator
passes file paths to agents rather than file contents (per CLAUDE.md: "Pass file paths to agents —
transcribing file contents into prompts bypasses agent verification"). Each phase agent produces its
artifact independently using Write, then the next phase receives the path. The current agent-to-file
write chain is:

```text
/add-new-feature orchestrator
  └─> feature-researcher agent     → Write plan/feature-context-{slug}.md
  └─> codebase-analyzer agent      → Write plan/codebase/{FOCUS}.md
```

### How Rules Files Are Loaded in This Repository

Rules files live in `.claude/rules/` and have two loading mechanisms:

1. **Auto-load via `paths:` frontmatter**: If a rule file has a `paths:` key with glob patterns,
   Claude Code auto-loads it whenever a matching file is opened. Example:
   `skill-content-optimization.md` has `paths: ["**/SKILL.md", "**/references/*.md"]` and fires
   whenever those file types are opened.

2. **Manual reference via CLAUDE.md**: Rules files without `paths:` frontmatter must be
   explicitly loaded. CLAUDE.md uses the pattern `- Rule Name: \`.claude/rules/rule-file.md\``
   to reference them as inline-readable links. Since CLAUDE.md is loaded at every session start,
   any rule file referenced there is discoverable by the orchestrator.

The new rule file `.claude/rules/large-file-write-strategy.md` must use method 2 (CLAUDE.md
reference, no `paths:` frontmatter). Method 1 with a `paths:` glob targeting agent files would
only auto-load when the agent file itself is being read/opened by Claude Code — not when the
agent is executing and about to write. The rule would be invisible to the agent at the exact
moment it matters.

The CLAUDE.md "Tool Usage" section (lines 57-61) currently lists:

```text
- Files: `Read`, `Write`, `Edit` — not `cat`, `sed`, `echo >`
- Search: `Grep`, `Glob` — not `find`, `ls -R`
- Python: `Bash(uv run script.py)`
```

The new bullet-link goes after this block (after `Python: \`Bash(uv run script.py)\``) and
before the `---` separator that follows. The section separator at approximately line 69 must
be preserved; the new line uses backtick-enclosed path (not a markdown hyperlink) to match
the style of other rule references in CLAUDE.md.

### How PreToolUse Hooks Work (for Phase 2 reference)

The `plugins/orchestrator-discipline/hooks.json` currently has two PreToolUse hook entries:

```json
{
  "PreToolUse": [
    { "matcher": "Read|Grep", "command": "pre-tool-orchestrator-read-warning.cjs" },
    { "matcher": "Bash", "command": "pre-tool-diagnostic-command-gate.cjs" }
  ]
}
```

For Phase 2, a third entry would be added matching `"Write"` pointing to
`pre-tool-large-write-warning.cjs`. Hook language convention (from `.claude/rules/language-conventions.md`)
requires Node.js CommonJS `.cjs` files for all Claude Code hooks.

A PreToolUse hook receives stdin JSON with `tool_input` containing the full Write call parameters:
`{ "file_path": string, "content": string }`. The hook reads `tool_input.content.length`, and if
> 25000, emits `additionalContext` in its response JSON:

```json
{
  "hookSpecificOutput": {
    "hookEventName": "PreToolUse",
    "additionalContext": "WRITE SIZE WARNING: ..."
  }
}
```

The hook exits 0 (non-blocking, soft warning). Omitting `permissionDecision` means the Write
proceeds normally. The `additionalContext` is injected into the agent's context before the Write
executes — the agent sees the warning and may adapt. The existing `pre-tool-orchestrator-read-warning.cjs`
follows this exact pattern and serves as the reference implementation.

### Relationship to Existing Constraints

**"No Invented Limits" (CLAUDE.md)**: The 25K threshold is NOT a content truncation limit. The
rule file must explicitly state this. The full document is written — incrementally instead of
atomically. The threshold governs write mechanics, not content completeness. This is analogous to
the Read tool's `offset`/`limit` parameters — they control the read window without hiding content.

**"Document Structure Policy" in swarm-task-planner (line 168)**: The existing 500-line split
threshold is subsumed by Strategy A. A task plan >= 500 lines is almost certainly > 25K characters,
so both thresholds align. The swarm-task-planner's existing PLAN/ directory split implementation
is already a correct example of Strategy A and needs no changes.

**orchestrator-discipline plugin**: The hook pattern (soft warning, exit 0, additionalContext)
follows the existing `pre-tool-orchestrator-read-warning.cjs` implementation. No changes to
hook infrastructure are needed in Phase 1. Phase 2 adds one new hook file and one hooks.json
entry to orchestrator-discipline.
