### How This Currently Works: FastMCP Creator Plugin (v2.4.3)

The `fastmcp-creator` plugin lives at `plugins/fastmcp-creator/` and exposes three Claude Code
skills: `fastmcp-creator` (the main skill), `fastmcp-python-tests`, and `fastmcp-client-cli`.
The plugin is registered in `.claude-plugin/marketplace.json`. When a user loads
`Skill(skill="fastmcp-creator:fastmcp-creator")`, Claude Code reads
`plugins/fastmcp-creator/skills/fastmcp-creator/SKILL.md` (730 lines, version 2.4.3) and
optionally any linked `./references/*.md` files the SKILL.md points to.

The current plugin has NO `mcpServers` field in `plugin.json` — it is a documentation-only
plugin. The `plugin.json` at `plugins/fastmcp-creator/.claude-plugin/plugin.json` currently
contains only `name`, `description`, `version`, and `author` fields.

The references directory at `plugins/fastmcp-creator/skills/fastmcp-creator/references/`
currently holds 7 files (note: the glob shows 6 `.md` files in references plus evaluation-guide):
`development-guidelines.md`, `community-practices.md`, `mcp-best-practices.md`,
`prompts-and-templates.md`, `example-projects.md`, `typescript-mcp-server.md`,
`claude-code-mcp-integration.md`, and `evaluation-guide.md`.

**Current SKILL.md has 6 confirmed errors (from feature-context doc):**

1. `@mcp.tool()` with parentheses — v3 canonical is `@mcp.tool` (no parentheses), per `quickstart.mdx`
2. `task=TaskConfig(mode="required")` — v3 API is `task=True`, per `tasks.mdx`
3. `require_auth` documented as valid — removed in v3; correct pattern is `require_scopes`
4. Provider names wrong — skill names 5 but actual v3 built-ins are: `LocalProvider`,
   `FastMCPProvider` (aka `ProxyProvider`), `FileSystemProvider`, `SkillsProvider`, `OpenAPIProvider`
5. Transform names wrong — actual v3 built-ins are: `Namespace`, `ToolTransform`, `Enabled`
   (Visibility), `ResourcesAsTools`, `PromptsAsTools`
6. `ctx.get_state()` / `ctx.set_state()` described as async — unverified against v3 `context.mdx`

**Sibling skills that are correct for v3 and must NOT be touched:**

- `plugins/fastmcp-creator/skills/fastmcp-python-tests/SKILL.md` — pytest patterns using
  in-memory transport `Client(mcp)` which matches v3 patterns correctly
- `plugins/fastmcp-creator/skills/fastmcp-client-cli/SKILL.md` — `fastmcp list` / `fastmcp call`
  / `fastmcp discover` CLI; accurate for v3 Beta 2+ including `generate-cli` and MCPConfig

**Deliverable 3 — MCP server** (Task T1.4, parallel with T1.1–T1.3):

files as MCP resources via `FileSystemProvider`. Path resolution uses `Path(__file__)` at
runtime — NOT `${CLAUDE_PLUGIN_ROOT}` inside Python code.


`plugins/agentskill-kaizen/.claude-plugin/plugin.json` — read this before writing the

### Discovered During Implementation

_Session Date: 2026-03-05_

During implementation, five structural discoveries were made that were not captured in the original context.

**Key Discoveries:**

1. **`mcp.add_provider()` is the correct v3 API; `mcp.include()` is a bug in the scaffold template**: The actual FastMCP v3 API method for attaching a provider to a server is `mcp.add_provider()`, confirmed in `server.py` line 40. However, the `scaffold_server()` tool's generated template (server.py line 116) still emits `mcp.include(FileSystemProvider(...))` — this is incorrect and will cause errors for users who run scaffolded code. Future implementations of any FastMCP v3 server must use `mcp.add_provider()`, not `mcp.include()`. The scaffold bug is tracked for fix.

2. **`accessing_online_resources.md` was never present in the repo**: Despite being listed as a "file to preserve" in the architecture spec and task file, this file does not exist in the references directory. Any future plan that lists it as a preserve target is carrying a phantom constraint. It can be dropped from preserve lists without risk.

3. **`fastmcp-client-cli/SKILL.md` was a staged new file at overhaul start**: The git status at session start shows `A plugins/fastmcp-creator/skills/fastmcp-client-cli/SKILL.md` — this file was staged as new, not pre-existing. The context manifest's assertion that it was a "sibling skill to preserve" was technically correct (it should not be modified) but the framing implied it was a long-standing file. Future plans involving this skill should treat it as a recently added file, not a legacy artifact.

4. **Five advanced.md source `.mdx` files are absent from the worktree**: The architecture spec listed `servers/storage-backends.mdx`, `servers/middleware.mdx`, `servers/dependency-injection.mdx`, `servers/versioning.mdx`, and `servers/visibility.mdx` as sources for `advanced.md`. None of these files were present in `.claude/worktrees/fastmcp/docs/`. This means the FastMCP v3 docs worktree is incomplete relative to what the architecture spec assumed. Any future task targeting these topics must first verify the worktree has been updated or sourced from a more complete docs snapshot.

5. **apps.md required a dual-layer approach (stable + unreleased warning)**: The `apps/overview.mdx` file contains content describing both the available v3.0 low-level HTML API and the unreleased v3.1 Python-native app framework, without clear separation. The implementation correctly isolates what is stable by adding a prominent WARNING block, but this required judgment to determine which sections of `apps/overview.mdx` were v3.0 vs v3.1. Future implementations should treat `apps/overview.mdx` as a mixed-stability source.

#### Updated Technical Details

- `mcp.add_provider(provider_instance)` — correct method to attach a FileSystemProvider or other provider in FastMCP v3
- `mcp.include(...)` — incorrect; this was the old API. Do not use. The scaffold_server() tool in server.py currently generates this incorrectly (bug DN-1)
- `.claude/worktrees/fastmcp/docs/servers/` is missing at least 5 files: `storage-backends.mdx`, `middleware.mdx`, `dependency-injection.mdx`, `versioning.mdx`, `visibility.mdx`
- `accessing_online_resources.md` — confirmed non-existent; remove from all future preserve lists

- The `scaffold_server()` tool in `plugins/fastmcp-creator/server/server.py` generates `mcp.include()` (line 116) — this is wrong. Until that bug is fixed, any test of scaffolded server output will fail at runtime. Do not use scaffolded output as a correctness reference for the `add_provider` pattern.
