### How This Currently Works: SAM Workflow Architecture

The SAM (Structured Agent-Managed) workflow is a multi-phase system that converts feature ideas into executable task files, implements them via agent delegation, and validates results through quality gates. The system integrates five components:

**Component 1: Backlog System (backlog_core MCP server)**
Manages items in .claude/backlog/ directory, syncs with GitHub Issues (source of truth). The backlog_add MCP tool creates both a local .md file and a GitHub issue with issue_number. That issue_number becomes the parent_issue_number for all downstream task tracking. Fields: title, priority (P0-P2 or Ideas), type, description, status, issue_number, plan (file path when linked to a SAM plan).

**Component 2: SAM Task Files (plan/ directory + sam CLI)**
The sam CLI reads/writes plan YAML files. Two formats supported: single file (plan/tasks-{N}-{slug}.md) or directory (plan/tasks-{slug}/ with one file per task). Each task has: task ID, title, status (not-started | in-progress | complete | blocked | deferred | skipped), agent (who implements it), dependencies, priority, complexity, skills (list), started timestamp, completed timestamp, last-activity timestamp. Plan-level fields: plan_number, slug, goal, issue (GitHub issue number), status.

**Component 3: Ready Task Query (sam_ready CLI + backlog_get_ready_sam_tasks MCP)**
When a task's status is not-started AND all dependencies have status complete, it is ready. sam_ready returns ReadyTasksResult: {feature: slug, ready_tasks: [{id, name, agent, skills, issue_number, ...}], count: N}. The MCP version (backlog_get_ready_sam_tasks) adds skills from task YAML, used by /implement-feature to load agent skills.

**Component 4: Task Execution (/start-task skill → agent delegation)**
The orchestrator calls Skill(skill='start-task', args='{task_file_path} --task {task_id}'). The skill runs inside a sub-agent delegation context. It reads the task YAML, calls sam_claim to mark the task in-progress (and acquire it), writes .claude/context/active-task-{CLAUDE_SESSION_ID}.json containing {task_file_path, task_id, parent_issue_number}, loads skills from the task's skills: field, then delegates implementation to the specified agent. The context file is the bridge between the skill layer and the hook layer.

**Component 5: Hook-Based Status Updates (task_status_hook.py)**
Two hook events:
- SubagentStop (fired when /implement-feature finishes a sub-agent): The hook reads the prompt for /start-task invocation, extracts task_file_path and task_id, calls sam_state to set status: complete and add Completed: {ISO timestamp}. Then calls sync_completion_to_github() to sync the completion to the GitHub sub-issue (github_issue field in task YAML, non-fatal if absent or fails).
- PostToolUse (Write|Edit|Bash during task execution): Hook reads .claude/context/active-task-{session_id}.json, updates the LastActivity: {ISO timestamp} field in the task YAML. This tracks real-time activity.

**Data Flow from GitHub Issue to Completion:**
1. backlog_add creates GitHub Issue #{N} → stored as issue_number in BacklogItem
2. sam create writes plan with issue: {N} field → becomes plan_number for file naming P{N}-{slug}.yaml
3. /start-task writes active-task-{sid}.json with parent_issue_number: {N}
4. Task execution happens with context file present
5. SubagentStop hook reads active-task-{sid}.json, uses parent_issue_number to find GitHub issue
6. Hook syncs task.github_issue (sub-issue) completion to GitHub
7. Active task context file is deleted

**Quality Gates (/complete-implementation)**
After all tasks complete (sam status shows all COMPLETE), /complete-implementation runs six validation phases: code-reviewer → feature-verifier → integration-checker → doc-drift-auditor → service-docs-maintainer → context-refinement. Each produces findings or durable artifacts. Then follow-up task files (if any from code-reviewer) are linked to backlog items and evaluated for recursion. Finally all modified files are staged and committed.

**Bookend Tasks (T0 and TN when acceptance-criteria-structured is present)**
When a plan has acceptance-criteria-structured entries, swarm-task-planner generates:
- T0 (t0-baseline-capture, Priority 1, no dependencies): Runs first (highest priority, no deps). Reads acceptance-criteria-structured, executes each check_command, records exit codes and stdout/stderr to plan/T0-baseline-{slug}.yaml.
- TN (tn-verification-gate, Priority 5, depends on all non-bookend tasks): Runs last. Reads T0 baseline, re-runs each check_command, compares results, classifies per criterion as: passed (T0 pass + TN pass), regressed (T0 pass + TN fail - blocks further execution), pre-existing-fail (T0 fail + TN fail), newly-passing (T0 fail + TN pass). Writes plan/TN-verification-{slug}.yaml as array of per-criterion records with status field (no top-level verdict field).
