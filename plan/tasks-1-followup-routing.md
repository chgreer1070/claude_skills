# Task Plan: Follow-up Task File Routing to Backlog

## Context

**Backlog item**: complete-implementation: route follow-up task files to backlog instead of orphaning
**GitHub Issue**: #381
**Priority**: P1
**Feature slug**: followup-routing
**Architecture spec**: [plan/architect-followup-routing.md](./architect-followup-routing.md)
**Codebase analysis**: [plan/codebase/followup-routing-patterns.md](./codebase/followup-routing-patterns.md)
**Feature context**: [plan/feature-context-followup-routing.md](./feature-context-followup-routing.md)

**Problem statement**: The `complete-implementation` skill's "Recursive Follow-up Handling" section
unconditionally recurses into `/implement-feature` when the `code-reviewer` creates follow-up task
files. When the orchestrator skips recursion (session timeout, deferred scope, context compaction),
follow-up files are orphaned at `plan/` with no backlog item and no rediscovery path. This has
occurred 3+ times.

**Resolution**: Replace the unconditional recursion with a 4-step routing workflow:
1. Detect follow-up files from code-reviewer ARTIFACTS output (with glob fallback)
2. For each file, search backlog by title keywords derived from the filename
3. Link to existing backlog item or create a new one — BEFORE the recursion decision
4. Recurse only if follow-up priority is High AND feature slug matches parent; otherwise defer

**Architecture decisions** (from architect spec, all accepted):
- ADR-1: Routing logic lives in SKILL.md as orchestrator instructions (no new Python script)
- ADR-2: "Same priority" = follow-up file `## Priority` field is `High`
- ADR-3: "Same session scope" = same feature slug extracted from filename
- ADR-4: Backlog item always created/linked BEFORE recursion decision
- ADR-5: Follow-up detection uses ARTIFACTS as primary + glob as fallback

### How the Code Reviewer Creates Follow-up Files

The code-reviewer agent (`plugins/python3-development/agents/code-reviewer.md`) creates follow-up task files in Step 6 of its SOP (line 122-125). It uses the `Write` tool directly — it does NOT invoke `backlog.py` or create any backlog item. Follow-up file creation happens entirely outside the backlog system.

**Naming convention** (`plugins/python3-development/agents/code-reviewer.md:175-192`):

```text
{project_path}/plan/tasks-{N}-{feature-slug}-followup-{issue-number}.md
```

Derivation: the agent reads the original task file path (e.g., `tasks-4-data-validation.md`), extracts the feature slug (`data-validation`), globs existing plan files to find next available `N`, then creates `tasks-{N}-{feature-slug}-followup-{issue-number}.md`.

Example for `tasks-4-data-validation.md` with two issues:

- `plan/tasks-5-data-validation-followup-1.md`
- `plan/tasks-6-data-validation-followup-2.md`

**Follow-up file internal structure** (`plugins/python3-development/agents/code-reviewer.md:195-238`):

The follow-up file is a hybrid YAML frontmatter + markdown body. The YAML frontmatter contains a `tasks:` list with `task`, `status`, and `parent_task` fields. The markdown body has these sections: `# Task: {Brief Title}`, `## Parent Task`, `## Status`, `## Priority` (values: `High`, `Medium`, or `Low` — this is what the routing gate reads), `## Description`, `## Acceptance Criteria`, `## Files to Modify`, `## Verification Steps`, `## References`. Note: this format differs from the primary SAM task file format (`tasks-{N}-{slug}.md`) which uses YAML frontmatter with feature-level metadata and `## Task N:` sections.

**Code reviewer output format** (`plugins/python3-development/agents/code-reviewer.md:240-254`):

```text
STATUS: DONE
SUMMARY: {one_paragraph_summary_of_review_findings}
ARTIFACTS:
  - Files reviewed: {count}
  - Issues found: {count}
  - Tasks created: {count}
  - Task files: {list of task file paths}
RISKS:
  - {critical_issues_requiring_attention}
NOTES:
  - {recommendations_for_improvement}
```

The `Task files:` line in `ARTIFACTS` is the primary source for the routing step to discover what follow-up files were created (ADR-5). The list is comma-separated file paths. If the ARTIFACTS block is absent or `Task files:` is empty, the routing step falls back to globbing `plan/tasks-*-{slug}-followup-*.md`.

### How the Backlog CLI Works (backlog.py)

All backlog operations go through `uv run .claude/skills/backlog/scripts/backlog.py`. Direct writes to `.claude/backlog/*.md` or `gh issue` CRUD bypass sync logic and are forbidden by `CLAUDE.md` (Backlog Operations section).

**`find_item()` function** (`.claude/skills/backlog/scripts/backlog.py:310-333`):

```python
def find_item(items: list[dict], selector: str) -> dict | None:
    # Supports: URL, #N, bare number, or title substring
    # Title substring: case-insensitive, first match wins
    selector_lower = selector.lower()
    matches = [it for it in items if selector_lower in it.get("_title", "").lower()]
    return matches[0] if len(matches) == 1 else (matches[0] if matches else None)
```

Returns `None` if no match. Returns first match if multiple exist (no error raised on multi-match). This is called by the `update` command to locate items — the routing step relies on this behavior via `backlog.py update "{selector}"`.

**`update` command signature** (`.claude/skills/backlog/scripts/backlog.py:1768-1802`):

```bash
uv run .claude/skills/backlog/scripts/backlog.py update \
  "<title substring, #N, bare number, or URL>" \
  [--plan PATH] \
  [--status in-progress] \
  [--create-issue] \
  [-R repo]
```

When `--plan PATH` is passed, writes `metadata.plan = PATH` to the per-item frontmatter file. **Exit code 1 if no item found** (`raise typer.Exit(1)` at line 1786). This is the behavior that triggers the title-mismatch retry described in Step 3 of the routing.

**`list --format json` command** (`.claude/skills/backlog/scripts/backlog.py:1038-1056` and JSON shape at 963-979`):

```bash
uv run .claude/skills/backlog/scripts/backlog.py list --format json -R Jamie-BitFlight/claude_skills
```

JSON output shape:

```json
[
  {
    "section": "P1",
    "title": "Item title",
    "issue": "#42",
    "plan": "plan/tasks-N-slug.md",
    "file_path": ".claude/backlog/p1-item-slug.md",

```

Items with `_skip=True` (done/resolved status, or in `Completed` section) are excluded from the output. The routing step must do a case-insensitive substring match on the `title` field of each returned item.

**`add` command** (`.claude/skills/backlog/scripts/backlog.py:856-868`):

### How create-backlog-item --auto Works

The `create-backlog-item` skill (`.claude/skills/create-backlog-item/SKILL.md`) in `--auto` mode is designed for agent-to-agent invocation without human interaction.

**Invocation from within another skill:**

```text
Skill(skill: "create-backlog-item", args: "--auto {derived_title}")
```

This is the pattern used in `work-backlog-item` (`.claude/skills/work-backlog-item/SKILL.md:59`) when `AUTO_MODE=true` and no item is found. The routing step reuses this exact pattern.

**`--auto` mode behavior** (`.claude/skills/create-backlog-item/SKILL.md:31-53`):

1. Title comes from `$1` onward — no `AskUserQuestion` calls.
2. All fields derived from research files and context.
3. Every decision logged as `[AUTO] ...` text.
4. GitHub issue creation skipped unless `--create-issue` explicitly passed — always adds `--no-create-issue` to the `backlog add` call.
5. **On duplicate detection**: logs `[AUTO] STOP — duplicate detected: "{existing title}" in {section}` and stops without writing. This is the "duplicate detection = match found" case for the routing step's error handling.

**Script call generated in `--auto` mode** (`.claude/skills/create-backlog-item/SKILL.md:163-169`):

```bash
uv run .claude/skills/backlog/scripts/backlog.py add \
  --title "{title}" \
  --priority "{priority}" \
  --description "{description}" \
  --source "Agent task — auto-derived from {filename}" \
  --type "{type}" \
  --no-create-issue \
  -R Jamie-BitFlight/claude_skills
```

The `--no-create-issue` flag is always present in `--auto` mode. Follow-up backlog items created by the routing step will be local-only unless the caller adds `--create-issue`. This is by design (ADR-4 in architect spec).

### The 4-Step Routing Workflow to Implement

The architecture spec (`plan/architect-followup-routing.md`) defines the routing replacement as four sequential steps expressed as orchestrator instructions in SKILL.md. No new Python script is created (ADR-1).

**Step 1 — Detect follow-up files:**

Extract file paths from the `Task files:` list in the code-reviewer's `ARTIFACTS` output. If the list is empty or absent, run a confirmatory glob using the pattern `plan/tasks-*-{slug}-followup-*.md` where `{slug}` is extracted from the parent task file path (`plan/tasks-{N}-{slug}.md` → strip `tasks-{N}-` prefix and `.md` suffix). If both ARTIFACTS and glob return empty: skip routing entirely (no follow-ups to route).

**Step 2 — Search backlog by derived title keywords:**

For each follow-up file, derive a search title using the 5-step algorithm:

```text
Input:  plan/tasks-8-data-validation-followup-1.md
Step 1: Strip directory prefix     --> tasks-8-data-validation-followup-1.md
Step 2: Strip .md extension        --> tasks-8-data-validation-followup-1
Step 3: Strip tasks-{N}- prefix    --> data-validation-followup-1
Step 4: Strip -followup-{k} suffix --> data-validation
Step 5: Replace hyphens with spaces --> data validation
Output: "data validation"
```

Search backlog: `uv run .claude/skills/backlog/scripts/backlog.py list --format json -R Jamie-BitFlight/claude_skills`. Parse JSON and check each item's `title` for case-insensitive substring match against derived keywords.

**Step 3 — Link or create backlog item:**

Branch on Step 2 result:

- **Match found**: `uv run .claude/skills/backlog/scripts/backlog.py update "{matched_item_title}" --plan "{followup_file_path}" -R Jamie-BitFlight/claude_skills`
- **No match found**: Invoke `Skill(skill: "create-backlog-item", args: "--auto {derived_title}")`, then run `backlog.py update "{derived_title}" --plan "{followup_file_path}" -R Jamie-BitFlight/claude_skills`.
- **Title mismatch retry**: If `backlog.py update` exits code 1 after creation (title mismatch between what `create-backlog-item` produced and what `update` searched), re-run `backlog.py list --format json`, find the most recently added item, retry `update` with its exact title. If retry also fails, log the error and continue to next follow-up.
- **Duplicate detection on create**: When `create-backlog-item --auto` logs `[AUTO] STOP — duplicate detected`, treat this as "match found" — run `backlog.py update` on the duplicate's title to attach the plan.

**Step 4 — Recursion gate (two conditions both must be true):**

1. Follow-up file slug matches parent task file slug (same session scope, ADR-3): extract slug from follow-up filename by stripping `tasks-{N}-` prefix and `-followup-{k}.md` suffix; compare against parent slug.
2. Follow-up file `## Priority` section contains `High` (ADR-2): read the follow-up file content and find the `## Priority` section. Only `High` qualifies for immediate recursion.

If BOTH conditions are met: `Skill(skill="implement-feature", args="{followup_task_file_path}")` then re-run `complete-implementation` on the follow-up.

If EITHER condition is NOT met: log `Follow-up {followup_path} linked to backlog item "{title}" -- deferred (priority: {priority}, scope: {same|different})`. Do not recurse.

**Error handling table** (from `plan/architect-followup-routing.md`):

| Condition | Behavior |
|-----------|----------|
| ARTIFACTS has no `Task files:` line | Glob for follow-ups. If glob empty, skip routing. |
| `backlog.py list` fails | Log error. Skip search. Create new item for each follow-up. |
| `backlog.py update` exits code 1 after creation | Re-list, find most recent item, retry with exact title. If retry fails, log and continue. |
| `create-backlog-item --auto` detects duplicate | Treat as match found — run `backlog.py update` on duplicate's title. |
| Follow-up has no `## Priority` section | Default to `Medium` (defer). Log: `No priority found in {path}, defaulting to Medium (deferred).` |
| Follow-up filename does not match expected pattern | Log warning. Create new backlog item with full filename as title. |
| Glob returns files from different feature slug | Filter to only include files matching parent task file's slug. |
