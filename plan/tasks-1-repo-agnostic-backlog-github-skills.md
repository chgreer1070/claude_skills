  Generated by context-gathering agent on 2026-03-20

  ## How This Currently Works: Repo-Agnostic Discovery

  The feature addresses a critical architectural gap: all GitHub-integrated skills (gh, backlog, milestone, daily-releases) currently hardcode the repository as Jamie-BitFlight/claude_skills across approximately 500 references in ~25 files. This prevents marketplace adoption — the plugin is unusable in other repositories without manual bulk edits.

  **Current Architecture:**
  The codebase already demonstrates runtime discovery patterns via _resolve_repo_root() in plugins/development-harness/backlog_core/models.py (lines 20-35). This function accepts an optional project_dir parameter (passed via --project-dir CLI argument for plugin cache installations) and falls back to Path.cwd() during in-repo development. The init(project_dir) function (lines 54-57) mutates module globals to re-initialize paths when called, enabling plugin installations to override the repo root dynamically.

  **Consumers of DEFAULT_REPO:**
  - plugins/development-harness/backlog_core/github.py — all functions use repo: str = DEFAULT_REPO default parameter
  - plugins/development-harness/backlog_core/operations.py — all functions accept repo parameter with same default
  - .claude/skills/gh/scripts/github_project_setup.py (line 55) — defines DEFAULT_REPO constant; already accepts --repo OWNER/REPO CLI parameter
  - .claude/skills/daily-releases/scripts/*.py — multiple PEP 723 scripts with inline DEFAULT_REPO declarations

  **SKILL.md Environment Variable Issue:**
  SKILL.md files undergo variable substitution at load time: $N, ${N}, and $ARGUMENTS are all replaced with user arguments. Importantly, environment variables like GITHUB_REPO return empty strings in project-level skills (not plugin cache installs), making env var substitution unreliable. However, inline command substitution (via backticks with !) works — commands execute at skill load time and their output is injected as literal text.

  ## For New Feature Implementation: What Needs to Connect

  **Discovery Function Architecture:**
  The implementation must add discover_repo() -> str to plugins/development-harness/backlog_core/models.py, following an explicit priority chain:
  1. GITHUB_REPO environment variable (if set and non-empty)
  2. gh repo view --json nameWithOwner -q .nameWithOwner via subprocess (requires gh CLI)
  3. Git remote URL parsing from git config --get remote.origin.url via subprocess (handles HTTPS, SSH, and proxy formats)
  4. Raise RepoDiscoveryError with actionable message (no fallback to hardcoded value)

  The function must validate the resolved owner/repo against a regex pattern and cache results via @lru_cache(maxsize=1) (per ADR-002) to avoid repeated discovery calls within a session.

  **Module-Level Integration:**
  DEFAULT_REPO becomes a module global initialized by discover_repo() at import time. Python evaluates default parameter values once at function definition time. Since DEFAULT_REPO is assigned before any function definitions, all consumers automatically receive the discovered value without signature changes.

  **SKILL.md Pattern (ADR-003):**
  SKILL.md files cannot reliably use env vars. Instead, they must include a discovery preamble that executes gh repo view at skill load time, using inline command substitution. The command output is inlined as literal text, replacing the backtick block.

  **Standalone Scripts (PEP 723):**
  Scripts that cannot import backlog_core must implement inline _discover_repo() function with the same priority chain as discover_repo() but without GitPython. Uses subprocess for gh and git commands.
