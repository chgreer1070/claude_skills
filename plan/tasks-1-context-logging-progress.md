### How the MCP Backlog Tool System Currently Works

The backlog skill provides an MCP server that exposes 10 tools for managing GitHub issue-backed development backlog items. The server is defined in `.claude/skills/backlog/backlog_core/server.py`, which acts as the orchestration layer that receives tool calls from MCP clients (Claude Code, agents, IDEs), validates inputs, delegates to business logic in `operations.py`, collects output, and returns structured results.

**Request Flow for Long-Running Tools** (the 4 tools being modified):

When an MCP client invokes `backlog_sync`, `backlog_pull`, `backlog_groom`, or `backlog_normalize`:

1. FastMCP framework routes the tool call to the corresponding function in `server.py`
2. The function creates an `Output()` instance to accumulate messages and warnings
3. The function calls the corresponding operation helper from `operations.py` (e.g., `operations.sync_items(dry_run=dry_run, output=out)`)
4. The operation helper processes the request â€” iterating GitHub issues, calling GitHub API endpoints, reading/writing local files â€” and populates the `output` object with messages and warnings
5. The server function receives the result dict from the operation helper, merges it with `out.to_dict()` (which includes warnings and messages), and returns the combined dict
6. The MCP client receives this dict â€” by the time the client sees anything, the entire operation is complete

**Current Behavior**: The client is silent during the operation. For a `backlog_sync` over 30 items taking 15â€“30 seconds, the client sees no feedback during execution and cannot distinguish between "still working" and "hung silently".

**What This Task Changes**: By adding `ctx: Context` parameter to the 4 target tools and inserting `await ctx.info(...)` / `await ctx.warning(...)` calls at phase-level milestones, the MCP framework streams real-time feedback to the client during execution. The client can now see "Starting backlog sync", "Sync complete: 5 issue(s) created, 8 item(s) pushed" and any warnings that occurred, as they happen, without waiting for the tool to finish.

### FastMCP Context Auto-Injection Mechanism

FastMCP's legacy type-hint injection (documented in `.claude/worktrees/fastmcp/docs/servers/context.mdx` lines 77â€“93) automatically injects a `Context` object into any tool function that declares `ctx: Context` as a parameter. This injection:

- Does NOT appear in the MCP schema â€” clients never know the `ctx` parameter exists
- Does NOT require any special import beyond `from fastmcp import Context, FastMCP`
- IS excluded from tool signature validation â€” FastMCP auto-provides it
- IS available as an async coroutine object â€” all `ctx.*` methods (`ctx.info()`, `ctx.warning()`) are async and must be `await`'ed

**Critical Prerequisite**: The 4 target functions must be `async def` (not synchronous `def`) because `await ctx.info(...)` is a coroutine and cannot be called from synchronous code. Issue #472 converts these 4 tools to `async def`. This task assumes #472 is merged before T1 starts.

### Why ctx Calls Stay in server.py, Not operations.py

The architect spec explicitly places all `ctx` calls in `server.py` and keeps `operations.py` unchanged. This boundary preserves:

- **Concern separation**: `operations.py` contains pure business logic; `server.py` is the MCP server orchestration layer. Logging and progress are MCP concerns, not business logic.
- **Caller independence**: `operations.py` functions remain callable from unit tests, scripts, and other non-MCP contexts without needing a mock Context object.
- **Limited scope**: Per-item progress requires refactoring operations loop internals (e.g., injecting a progress callback into the loop at `operations.py:1014`, `operations.py:1055`, `operations.py:1532`, `operations.py:1382`). This is deferred to a future ticket.

The 4 server.py tools each follow the same insertion pattern: `await ctx.info(...)` before the operation, `for w in out.warnings: await ctx.warning(w)` loop after the operation returns, and `await ctx.info(...)` completion summary before the return statement.

### Output Handling: Both ctx and out.warnings

The architect spec specifies that both `ctx.warning(...)` calls AND the existing `out.warnings` list are preserved. This creates two channels for warnings:

- **Streaming channel** (`ctx.warning(w)`): Messages flow to the MCP client in real time, visible in logs or progress UI
- **Structured channel** (`out.warnings` in return dict): Same warnings appear in the tool's final return dict, preserving backward compatibility for callers that parse the dict structure

This duplication is intentional. Code that currently extracts warnings from the return dict still works; code updated to display live warnings via `ctx.warning` streams gets feedback in real time.

### The 4 Target Tools and Their ctx Call Patterns

**backlog_sync** (`server.py:122â€“138`):
- Delegates to `operations.sync_items` which handles both "create missing issues" and "push groomed content" in a single call
- ctx calls: start log "Starting backlog sync" (or with " (dry-run)" suffix if dry_run), warning surfacing loop, completion log "Sync complete: N issue(s) created, M item(s) pushed"

**backlog_pull** (`server.py:347â€“381`):
- Has two branches: single-item path (when `selector` is provided) calls `operations.pull_by_selector`; bulk path (when `selector` is None) calls `operations.pull_items`
- Single-item: ctx.info start, warning loop, ctx.info result
- Bulk: ctx.info start "Starting bulk pull from GitHub", warning loop, ctx.info completion "Pull complete: N item(s) pulled"

**backlog_groom** (`server.py:292â€“323`):
- Processes a single item (one selector); calls `operations.groom_item` once
- ctx calls: start log "Grooming item: {selector}", warning loop, completion log "Groomed: {title}"
- No `ctx.report_progress` â€” this is single-item, not batch

**backlog_normalize** (`server.py:326â€“344`):
- Processes all matching `.md` files in the backlog directory; calls `operations.normalize_items`
- ctx calls: start log "Starting normalize" (or with " (dry-run)" suffix), warning loop, completion log "Normalized N file(s)" (with dry-run suffix if applicable)

### The 6 Excluded Tools

`backlog_add`, `backlog_list`, `backlog_view`, `backlog_close`, `backlog_resolve`, `backlog_update` are NOT modified. These are short-running synchronous functions:

- `backlog_add`: Creates a single backlog item file (no GitHub I/O)
- `backlog_list`: Queries local files (no GitHub I/O)
- `backlog_view`: Reads and displays a single item (no GitHub I/O)
- `backlog_close`: Updates a single item and creates/updates one GitHub issue (synchronous, already handles its own output)
- `backlog_resolve`: Updates a single item and creates/updates one GitHub issue (synchronous, already handles its own output)
- `backlog_update`: Updates a single item's properties and groomed content (synchronous, already handles its own output)

These tools do not iterate over collections or make multiple sequential API calls, so the feedback benefit of live logging is minimal. Keeping them unchanged reduces risk and avoids the async conversion for functions that don't need it.

### Discovered During Implementation

_Session Date: 2026-03-06_

During implementation, we discovered that Python syntax prevents placing `ctx: Context` as the final parameter when preceding parameters have defaults. The architect spec prescribed appending `ctx: Context` after all existing parameters (e.g., after `dry_run=False`), but Python forbids a non-default parameter following a default parameter. The implementing agent placed `ctx: Context` first in each signature instead. FastMCP auto-injects `ctx` regardless of position, so the runtime behavior and MCP schema are identical to what the spec intended.

The test infrastructure discovery is also noteworthy: T2 used FastMCP's built-in `Client` test harness rather than `unittest.mock.AsyncMock`. The test file `tests/test_backlog_core_server.py` contains 21 new ctx-specific tests (covering all 4 tools: start log, completion log, dry-run variants, warning surfacing) using FastMCP's client pattern, not direct function calls with mocked ctx. This means the context manifest's note about "mock or test Context object" was technically correct in intent but the actual mechanism was the FastMCP Client â€” future test authors should search the test file for the client setup pattern before writing new ctx tests.

Two pre-existing test files received minor parameter-name fixes for API drift unrelated to this feature: `test_live_validation.py` and `test_scenarios.py` had stale parameter names (`checklist_pass` â†’ `reason`, `reason` â†’ `summary`). These were fixed as part of T2 to restore a passing baseline, not as part of the ctx feature itself.

**Key Discoveries:**

1. **ctx parameter position constraint**: Python syntax requires `ctx: Context` (a non-default parameter) to appear BEFORE any defaulted parameters. The architect spec's placement (after `dry_run=False`, etc.) is syntactically invalid. Actual position: first parameter in each function. FastMCP injection is position-independent, so behavior is unchanged.

2. **FastMCP Client test pattern**: The recommended test mechanism for ctx calls is FastMCP's `Client` harness, not `AsyncMock`. The test file `tests/test_backlog_core_server.py` demonstrates this pattern for all 4 tools. Use it as the reference for future ctx test additions.

3. **Pre-existing test API drift**: `test_live_validation.py` and `test_scenarios.py` had stale parameter names (`checklist_pass`, `reason`) that were silently wrong before this session. Fixed in T2 as baseline restoration. Future test sessions should run the full suite before adding new tests to catch similar drift early.

4. **report_progress omitted entirely**: The original feature request (and feature-context Goals section) specified `ctx.report_progress(i, total)` for batch operations. The architect spec determined this is not achievable without refactoring `operations.py` (item totals are computed inside operation helpers, not accessible from `server.py`). The implementation delivers only `ctx.info` start/completion logs and `ctx.warning` surfacing â€” no progress fractions. This is the resolved design-refinement documented in the architect spec's `report_progress Decision` section.

5. **Documentation approach â€” prose over parameter tables**: SKILL.md and README.md describe progress behavior via prose paragraphs and inline `# Progress:` comments in code examples. The `ctx` parameter is framed as framework-injected, not caller-supplied, so it does not appear in parameter tables. Future doc authors should follow this convention: framework-injected parameters belong in a "How it works" prose section, not in the parameter reference table.

#### Updated Technical Details

- Correct `ctx: Context` position: FIRST parameter in each async function signature, before all defaulted parameters
- FastMCP test pattern: use `from fastmcp import Client` and `async with Client(mcp) as client: result = await client.call_tool(...)` â€” ctx is auto-injected by the test client, no manual mock needed
- `report_progress` is deferred to a future ticket requiring `operations.py` refactoring; the future insertion targets are documented in `plan/architect-context-logging-progress.md` under "report_progress Decision"
- Pre-existing parameter drift in test files: `checklist_pass` â†’ `reason`, `reason` â†’ `summary` (fixed in T2 baseline pass)

#### Gotchas for Future Developers

- Never place `ctx: Context` after any defaulted parameter in a FastMCP tool signature â€” Python will raise `SyntaxError: parameter without a default cannot follow parameter with a default`. Always put it first.
- FastMCP excludes `ctx` from the MCP schema regardless of its position â€” callers never supply it, position only matters for Python syntax.
- Run the full test suite (`uv run pytest tests/ -k backlog -v`) before adding new ctx tests to catch pre-existing drift. The baseline was restored in T2 but future drift is possible.
- `ctx.report_progress` requires item totals computed before iteration begins. For `backlog_sync`, `backlog_pull`, and `backlog_normalize`, totals are only known inside `operations.py` helpers. Server-layer progress fractions require either a progress-callback parameter in each operation helper or moving iteration into `server.py`.
