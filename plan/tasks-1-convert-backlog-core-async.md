### How the backlog_core MCP Server Currently Works

The backlog_core system implements a FastMCP 3.x server exposing 10 backlog management tools via the MCP protocol. When an MCP client invokes any tool (e.g., `backlog_add`, `backlog_sync`), the request flows through three distinct architectural layers:

**Layer 1: MCP Tool Dispatch (server.py, lines 16-381)**

All 10 tools are currently synchronous `def` functions decorated with `@mcp.tool()`. FastMCP automatically registers these as callable MCP tools. When a client invokes a tool, FastMCP dispatches it by:
1. Parsing the client's JSON-RPC request into tool name and parameters
2. Locating the matching `@mcp.tool()` function in `server.py`
3. Calling the function synchronously (since FastMCP 3.x thread-pools sync functions automatically)
4. Returning the result dict back to the client

The tool function body follows an identical pattern across all 10 tools:
- Create an `Output()` instance to collect logging messages
- Execute a try/except block
- Call a single `operations.*` function with tool parameters plus the output instance
- Return `{**result, **out.to_dict()}` on success or `{"error": str(e), **out.to_dict()}` on `BacklogError`

This pattern is guaranteed identical across all 10 tools — no tool does branching beyond the single-call pattern, except `backlog_pull` which conditionally calls `operations.pull_by_selector()` OR `operations.pull_items()` based on whether `selector` is provided.

**Layer 2: Business Logic (operations.py, all functions synchronous)**

Each tool calls exactly one function from `operations.py`. The operations module provides 10 public functions (`add_item`, `list_items`, `view_item`, `sync_items`, `close_item`, `resolve_item`, `update_item`, `groom_item`, `normalize_items`, `pull_items`/`pull_by_selector`) that implement high-level backlog CRUD logic. These functions are 100% synchronous — they perform file system operations (pathlib Path.read_text/write_text) and call into `github.py` functions. They do not use `async`/`await` anywhere.

**Layer 3: GitHub API Boundary (github.py, all functions synchronous + PyGithub blocking)**

The `github.py` module wraps all interactions with the GitHub API via PyGithub 2.8.1+. PyGithub is a **synchronous blocking HTTP library** — its methods like `repo.create_issue()`, `repo.get_issue()`, `issue.edit()`, and `gh.search_issues()` all block the calling thread until the HTTP request completes. There is no async alternative to PyGithub; the library is blocking by design.

**The Constraint: No `await` in Sync Tools**

FastMCP's Context object provides methods for logging and interactive features:
- `ctx.info()` — logs info-level messages (async coroutine)
- `ctx.report_progress()` — sends progress updates to the client (async coroutine)
- `ctx.warning()` — logs warnings (async coroutine)
- `ctx.elicit()` — opens interactive dialog with the client (async coroutine)

**These are all async methods.** They require `await` to call. However, because the current tools are synchronous `def` functions, the Python async constraint forbids `await` in a sync context — any attempt to `await` inside a sync function raises `RuntimeError: cannot use await outside an async function`.

This is precisely what blocks the four dependent issues (#465, #469, #470, #473). They all need to call `await ctx.info()`, `await ctx.report_progress()`, or `await ctx.elicit()` inside tool bodies.

### For This Conversion Task: What Needs to Connect

Since this task converts all 10 tools to `async def`, the conversion must satisfy these requirements:

1. **Preserve tool signatures identically** — MCP client contracts depend on tool names, parameter names, types, defaults, and return shapes remaining unchanged
2. **Preserve the try/except structure** — `BacklogError` exception handling must remain in place and functional
3. **Wrap all blocking operations** — `operations.*` calls must use `await asyncio.to_thread(operations.X, ...)` so the blocking I/O does not freeze the event loop
4. **Handle the dual-branch pattern in backlog_pull** — two separate `asyncio.to_thread()` calls, one in each branch of the if/else
5. **Add `import asyncio`** — stdlib import at the top of the file
6. **Verify no other files change** — operations.py, github.py, models.py are read-only for this task

**Why this approach works**: FastMCP 3.x (>=3.0.2) natively supports async tools. When a tool is `async def`, FastMCP awaits its coroutine directly in the event loop. The `asyncio.to_thread()` call inside each tool offloads the blocking `operations.*` call (and by extension all its PyGithub calls) to the thread pool, preventing the event loop from blocking. This maintains the same threading behavior as sync tools (via FastMCP's internal thread pool), but allows the tool body to be async so `await` works for future `ctx.*` calls.

#### Conversion Pattern (identical for all 10 tools except backlog_pull)

**Before:**

```python
@mcp.tool()
def backlog_TOOLNAME(...) -> dict:
    out = Output()
    try:
        result = operations.OPERATIONNAME(arg1=val1, arg2=val2, ..., output=out)
        return {**result, **out.to_dict()}
    except BacklogError as e:
        return {"error": str(e), **out.to_dict()}
```

**After:**

```python
@mcp.tool()
async def backlog_TOOLNAME(...) -> dict:
    out = Output()
    try:
        result = await asyncio.to_thread(operations.OPERATIONNAME, arg1=val1, arg2=val2, ..., output=out)
        return {**result, **out.to_dict()}
    except BacklogError as e:
        return {"error": str(e), **out.to_dict()}
```

Two mechanical changes:
1. `def` → `async def`
2. `operations.OPERATION(...)` → `await asyncio.to_thread(operations.OPERATION, ...)`

All keyword arguments are passed through exactly as written. The `asyncio.to_thread(fn, *args, **kwargs)` signature passes all `**kwargs` through to the target function.

### Discovered During Implementation

_Session Date: 2026-03-06_

During implementation, we discovered that `operations.pull_by_selector()` does not accept `dry_run` or `force` parameters. The architect spec's Tool 10 specification incorrectly listed those two keyword arguments in the `pull_by_selector` branch of `backlog_pull`. The actual `operations.pull_by_selector` signature accepts only `selector` and `output`. The implementing agent read the live `server.py` and `operations.py` files, identified the mismatch, and applied the correct argument list. The `dry_run` and `force` parameters are correctly passed only to `pull_items` in the else branch.

Additionally, `selector` was passed positionally (not as a keyword) to `pull_by_selector` — `pull_by_selector(selector, output=out)` — which is also a deviation from the keyword-only style shown in the spec. This is acceptable because both forms are equivalent for a positional parameter.

2. **Import placement at line 5 (alphabetically correct)**: `import asyncio` was inserted between `from __future__ import annotations` and `from typing import Annotated` at line 5, satisfying both the alphabetical-ordering convention and the spec's requirement to place it before third-party imports.
