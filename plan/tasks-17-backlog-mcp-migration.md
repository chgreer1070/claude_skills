### How the Backlog Tooling Currently Works

The backlog system manages backlog items through two separate interfaces that share a common Python operations layer. Understanding this layered architecture is essential for every task in this migration plan.

**Architecture layers** (bottom to top):

1. **`operations.py`** — the single source of business logic at `.claude/skills/backlog/backlog_core/operations.py`. All CRUD functions live here: `add_item()`, `list_items()`, `view_item()`, `sync_items()`, `close_item()`, `resolve_item()`, `update_item()`, `groom_item()`, `normalize_items()`, `pull_items()`.
2. **`backlog.py`** — the CLI script at `.claude/skills/backlog/scripts/backlog.py`. A thin wrapper calling `operations.py` functions and exposing them as `add|list|view|sync|close|resolve|update|groom|normalize|pull` subcommands.
3. **`server.py`** — the FastMCP server at `.claude/skills/backlog/backlog_core/server.py`. A second thin wrapper calling `operations.py` functions and exposing them as 10 MCP tools: `backlog_add`, `backlog_list`, `backlog_view`, `backlog_sync`, `backlog_close`, `backlog_resolve`, `backlog_update`, `backlog_groom`, `backlog_normalize`, `backlog_pull`.

**Current flow when a skill runs a backlog operation:**

When the orchestrator reads a skill like `/work-backlog-item` and encounters a Step instruction such as "list backlog items," it currently finds a bash code fence:

```bash
uv run .claude/skills/backlog/scripts/backlog.py list --format json --with-status -R Jamie-BitFlight/claude_skills
```

The orchestrator interprets this literal command as "invoke via Bash tool." It spawns a subprocess, captures stdout (a JSON array), and the surrounding prose says "Parse the JSON output." Every operation follows this pattern: prose context, bash fence, prose about parsing. There are 19 such invocations in `work-backlog-item/SKILL.md` alone.

**Target state after migration:**

After migration, that same step will say in prose (no bash fence):

```text
Call the `mcp__backlog__backlog_list` tool with `with_status=true`.
Parse the returned dict. Each entry in `items` has `title`, `priority`, `issue`, `plan`, `status`,
`milestone`, `file_path`, `groomed`.
```

The orchestrator reads "Call the `mcp__backlog__backlog_list` tool" and invokes the MCP tool directly — no subprocess, no JSON parsing, a native structured dict. The `-R Jamie-BitFlight/claude_skills` flag is dropped because the MCP server uses the `DEFAULT_REPO` constant internally.

**Why MCP tools are available to the orchestrator:**

The MCP server is registered in `.mcp.json` (lines 25-28) and the `mcp__backlog__*` permission pattern is already in `.claude/settings.json` line 13. No infrastructure changes are needed — only instruction file content changes. The MCP server starts via `uv run python -m backlog_core.server` from the working directory where `.claude/skills/backlog/` is reachable.

**Why sub-agents also have MCP access:**

Sub-agents spawned via the Agent tool do not inherit the orchestrator's MCP connections by default. However, the two agent files that need backlog access (`backlog-item-groomer.md` and `backlog-mcp-validator.md`) both declare their own `mcpServers:` block in frontmatter (with `cwd: .claude/skills/backlog`), giving them independent MCP access. All user-invocable backlog skills (`/work-backlog-item`, `/create-backlog-item`, `/groom-backlog-item`, `/group-items-to-milestone`, `/backlog`, `/backlog-tools-administrator`) are loaded by the orchestrator, not sub-agents, so MCP-only instructions are safe for all migration targets (ADR-3 confirms this).

### The Core Format Problem Being Solved

Current skill files instruct the orchestrator using **bash code fences**. The orchestrator reads these as "run via Bash tool." The migration changes that instruction format to **prose MCP tool call descriptions**, which the orchestrator reads as "invoke this MCP tool."

ADR-1 from the architecture spec defines three prose instruction formats:

**Simple calls (3 or fewer parameters):**

```text
Call the `mcp__backlog__backlog_list` tool with `with_status=true`.

Parse the returned dict. Each entry in `items` has `title`, `priority`, `issue`, `plan`.
```

**Complex calls (4+ parameters or conditional construction):**

```text
Call the `mcp__backlog__backlog_add` tool:

| Parameter | Value |
|-----------|-------|
| `title` | `"{title}"` |
| `priority` | `"{priority}"` |
| `description` | `"{description}"` |
| `source` | `"{source}"` |
| `type` | `"{type}"` |
| `create_issue` | `true` if P0/P1 and user confirmed; `false` if P2/Ideas or user declined |

Check the returned dict for `error` key.
```

**Shorthand in annotation blocks (non-executable):**

```text
mcp__backlog__backlog_groom(selector="{title}", section="Fact-Check", content="{summary}")
```

The `mcp__backlog__` prefix is the established naming convention. It appears in `.claude/settings.json` permissions, in agent `tools` frontmatter, and in the already-migrated `backlog-item-groomer.md` line 88. This prefix must be preserved exactly on all tool references.

### Critical Semantic Correction: `close --reason` vs `backlog_resolve` (ADR-8)

This is the highest-risk correctness issue in the migration. Several skill files use:

```bash
uv run .claude/skills/backlog/scripts/backlog.py close "{title}" --reason "..."
```

This is a semantic bug in the CLI invocation. The CLI `close` subcommand does NOT accept `--reason` — only `resolve` does (confirmed in `backlog.py`). The migration must fix this: every `close ... --reason` call becomes `mcp__backlog__backlog_resolve(selector=..., reason=...)`, not `mcp__backlog__backlog_close(...)`.

Affected locations (all four must be corrected):

- `work-backlog-item/SKILL.md` line 195: `close "{title}" --reason "..."` → `backlog_resolve`
- `work-backlog-item/SKILL.md` line 301: `close "{title}" --reason "..."` → `backlog_resolve`
- `groom-backlog-item/SKILL.md` line 65: `close "{title}" --reason "..."` → `backlog_resolve`
- `groom-backlog-item/SKILL.md` line 94: `close "{title}" --reason "..."` → `backlog_resolve`

The distinction: `backlog_close` requires `checklist_pass=true` and marks the item DONE (work completed). `backlog_resolve` requires `reason=...` and marks the item RESOLVED (no longer applicable, duplicate, already done, etc.). These have different semantics in `operations.py`.

### Dropped Parameters — What Must NOT Appear in MCP Calls

Four CLI flags have no MCP equivalents and must be omitted:

1. **`-R Jamie-BitFlight/claude_skills`** — The MCP server uses `DEFAULT_REPO` constant. No `repo` parameter exists on any MCP tool. Remove from every invocation.
2. **`--format json`** — MCP tools always return structured dicts. No `format` parameter exists. Remove from every invocation.
3. **`--groomed-file <path>` and `groomed: bool`** — These exist on `operations.update_item()` and `operations.groom_item()` but are NOT exposed via MCP (DOCUMENTATION_DRIFT_AUDIT.md FIND-10, FIND-11). MCP callers must provide content as inline strings.
4. **`--research-first`** — This CLI flag for `backlog add` has NO MCP equivalent. The `research_first` field does not exist on `backlog_add`. Embed research questions in the `description` parameter instead.

### The `--help` Discovery Pattern Replacement (ADR-5)

`groom-backlog-item/SKILL.md` has a safety guard at line 276 that instructs the orchestrator to run `backlog.py <subcommand> --help` before calling an unfamiliar subcommand. MCP tools return structured `{"error": "..."}` responses for invalid parameters, making this guard unnecessary.

ADR-5 replacement text (verbatim):

```text
**MCP tool parameters are schema-enforced.** Unlike CLI subcommands, MCP tools reject invalid parameters
with a structured error. There is no need to verify signatures before calling. If unsure which tool to
use, check the tool name and parameters:

- `mcp__backlog__backlog_update` — updates an existing item (selector required)
- `mcp__backlog__backlog_groom` — writes groomed content (selector required)
- `mcp__backlog__backlog_sync` — syncs ALL items to GitHub (no selector — operates on entire backlog)
```

### Return Value Contract for All MCP Tools

Every MCP tool returns a `dict`. The contract is:

- On error: `{"error": str(e), ...}` — always check for `"error"` key first
- On success: result data keys vary per tool, plus `messages: list[str]` and `warnings: list[str]` (always present, may be empty)

Surrounding prose in skill files currently says "Parse the JSON output." After migration, change to "Parse the returned dict." or "Check the returned dict for `error` key."

### The `backlog_list` Missing Parameters in Validator Agent

The `backlog_list` tool exposes six optional parameters: `with_status`, `from_github`, `label`, `section`, `status`, `title`. The `backlog-mcp-validator.md` agent only lists the first three. Task 8 must add `section`, `status`, and `title` to the validator agent's tool reference for `backlog_list`.

### Complete CLI-to-MCP Parameter Mapping

| CLI Pattern | MCP Tool Call |
|---|---|
| `backlog list --format json --with-status` | `mcp__backlog__backlog_list(with_status=true)` |
| `backlog list --format json` | `mcp__backlog__backlog_list()` |
| `backlog list --format json --status resolved` | `mcp__backlog__backlog_list(status="resolved")` |
| `backlog view "{selector}" --format json` | `mcp__backlog__backlog_view(selector="{selector}")` |
| `backlog add --title T --priority P --description D` | `mcp__backlog__backlog_add(title=T, priority=P, description=D)` |
| `backlog add ... --no-create-issue` | `mcp__backlog__backlog_add(..., create_issue=false)` |
| `backlog close "{sel}" --plan P --checklist-pass` | `mcp__backlog__backlog_close(selector="{sel}", plan=P, checklist_pass=true)` |
| `backlog close "{sel}" --reason "..."` | `mcp__backlog__backlog_resolve(selector="{sel}", reason="...")` (ADR-8) |
| `backlog resolve "{sel}" --reason "..."` | `mcp__backlog__backlog_resolve(selector="{sel}", reason="...")` |
| `backlog update "{sel}" --plan P` | `mcp__backlog__backlog_update(selector="{sel}", plan=P)` |
| `backlog update "{sel}" --status in-progress` | `mcp__backlog__backlog_update(selector="{sel}", status="in-progress")` |
| `backlog update "{sel}" --create-issue` | `mcp__backlog__backlog_update(selector="{sel}", create_issue=true)` |
| `backlog update "{sel}" --section S --content C` | `mcp__backlog__backlog_update(selector="{sel}", section=S, content=C)` |
| `backlog groom "{sel}" --section S --content C` | `mcp__backlog__backlog_groom(selector="{sel}", section=S, content=C)` |
| `backlog groom "{sel}" --groomed-content BODY` | `mcp__backlog__backlog_groom(selector="{sel}", groomed_content=BODY)` |
| `backlog sync` | `mcp__backlog__backlog_sync()` |
| `backlog normalize --dry-run` | `mcp__backlog__backlog_normalize(dry_run=true)` |
| `backlog pull --dry-run` | `mcp__backlog__backlog_pull(dry_run=true)` |
| `-R Jamie-BitFlight/claude_skills` (any command) | omit — MCP uses `DEFAULT_REPO` constant |
| `--format json` (any command) | omit — MCP always returns dicts |

### MCP Tool Signatures (authoritative source: server.py)

```python
# backlog_add — returns {file_path, title, priority, issue?, messages, warnings}
# CRITICAL: parameter is called "type" (alias), not "type_"
backlog_add(title, priority, description, source="Not specified",
            type="Feature", create_issue=True, force=False)

# backlog_list — returns {items: [{title, priority, issue, plan}], messages, warnings}
# section/status/title are filter params NOT in backlog-mcp-validator.md (Task 8 fix needed)
backlog_list(with_status=False, from_github=False, label=None,
             section=None, status=None, title=None)

# backlog_view — returns {title, priority, issue, plan, file_path, body, groomed, messages, warnings}
backlog_view(selector, offset=0, limit=0)

# backlog_sync — returns {created, pushed, messages, warnings}
backlog_sync(dry_run=False)

# backlog_close — returns {title, issue?, messages, warnings}
# NO "reason" parameter — use backlog_resolve for reason-based closes
backlog_close(selector, plan, checklist_pass=False, cleanup=False, force=False)

# backlog_resolve — returns {title, reason, issue?, messages, warnings}
backlog_resolve(selector, reason, cleanup=False, force=False)

# backlog_update — returns {title, changes, messages, warnings}
backlog_update(selector, plan=None, status=None, create_issue=False,
               groomed_content=None, section=None, content=None,
               title=None, description=None)

# backlog_groom — returns {title, synced, messages, warnings}
backlog_groom(selector, groomed_content=None, section=None, content=None)

# backlog_normalize — returns {normalized, messages, warnings}
backlog_normalize(dry_run=False)

# backlog_pull — returns {pulled, messages, warnings}
backlog_pull(dry_run=False, force=False)
```
