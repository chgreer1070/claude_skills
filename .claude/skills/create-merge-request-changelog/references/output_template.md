# MR Description Template

This template defines the structure for merge request descriptions generated by this skill.

## Template Structure

```markdown
# {MR_TITLE}

## Summary

{AI_GENERATED_OVERVIEW}

## Statistics

- **Commits**: {commit_count}
- **Files Changed**: {files_changed}
- **Lines Added**: {lines_added}
- **Lines Deleted**: {lines_deleted}

## Changes by Category

### 🐛 Bug Fixes

{bug_fixes_section}

### ✨ Enhancements

{enhancements_section}

### 🏗️ Technical Debt

{tech_debt_section}

### 📚 Documentation

{documentation_section}

### 🧪 Testing

{testing_section}

### 🔧 Build & CI

{build_ci_section}

### 🧹 Non-Functional Changes

{non_functional_section}

## Components Affected

{components_list}

## Breaking Changes ⚠️

{breaking_changes_section}

## Deployment Notes

{deployment_considerations}

---

*Generated by Claude Code `/create-merge-request-changelog` skill*
```

## Section Templates

### Bug Fixes Section

```markdown
- **{title}**
  - **Description:** {what_was_broken_and_fix}
  - **Impact:** {who_affected}
  - **Files:** `{file1}`, `{file2}`
  - **Technical Notes:** {implementation_details}
```

### Enhancements Section

```markdown
- **{title}**
  - **Feature:** {what_was_added}
  - **Benefits:** {value_provided}
  - **Usage:** {how_to_use}
  - **Files:** `{file1}`, `{file2}`
  - **Technical Details:** {api_changes_config}
```

### Technical Debt Section

```markdown
- **{title}**
  - **Purpose:** {why_needed}
  - **Changes:** {what_modified}
  - **Impact:** {how_improves_system}
  - **Files:** `{file1}`, `{file2}`
```

### Documentation Section

```markdown
- **{title}**: {changes_description}
  - **Location:** {file_paths_or_urls}
  - **Importance:** {why_it_matters}
```

### Testing Section

```markdown
- **{title}**: {coverage_details}
  - **Type:** {unit|integration|e2e}
  - **Files:** `{test_files}`
```

### Build & CI Section

```markdown
- **{title}**
  - **Changes:** {what_modified}
  - **Impact:** {workflow_improvement}
  - **Files:** `{config_files}`
```

### Non-Functional Changes

```markdown
Minor repository improvements:
- **{category}**: {concise_description} ({files})
```

### Breaking Changes

```markdown
- **{change_description}**
  - **Migration:** {how_to_migrate}
  - **Affected:** `{files_or_apis}`
  - **Commits:** {commit_hashes}
```

## Component List Format

```markdown
- **{component_name}**: {brief_description_of_changes}
```

## Deployment Notes Format

```markdown
### Prerequisites
- {prerequisite_1}
- {prerequisite_2}

### Migration Steps
1. {step_1}
2. {step_2}

### Rollback Plan
{how_to_rollback_if_needed}
```

## Empty Section Handling

When a category has no changes, omit the entire section from the output. Don't show empty sections with "None" or similar placeholders.

## Markdown Formatting Guidelines

- Use **bold** for field labels and emphasis
- Use `code ticks` for file paths, code references, and technical terms
- Use bullet lists (-) for items within sections
- Use numbered lists (1. 2. 3.) for sequential steps
- Keep line length reasonable for readability (wrap around 100-120 characters)
- Add blank lines between major sections for visual separation

## Tone and Style

- **Professional but approachable**: Write for technical audiences but avoid unnecessary jargon
- **Action-oriented**: Use active voice and present/past tense consistently
- **Specific and concrete**: Provide details, not vague descriptions
- **Value-focused**: Explain why changes matter, not just what changed
- **Concise**: Be thorough but respect reviewer time
