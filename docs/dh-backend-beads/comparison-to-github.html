1. Executive Summary

Beads (bd CLI) is a Dolt-powered distributed graph issue tracker with 70+ commands, native JSON output on virtually every command, dependency-aware DAG execution, a molecule/formula workflow system, multi-assignee claim semantics, a persistent memory system, and bidirectional sync with GitHub, Linear, Jira, GitLab, ADO, and Notion. Its data model is significantly richer than GitHub Issues: first-class dependency typing (19 built-in edge types), custom statuses and types, structured metadata JSON blobs on every issue, async gate primitives, and Dolt-based version-controlled storage with federation.

Development-Harness (DH) is a Claude Code plugin that manages software project backlogs via GitHub Issues. It exposes a FastMCP server with three logical subsystems: backlog CRUD tools (backlog_*), artifact manifest management (artifact_*), and dispatch/wave orchestration (dispatch_*). The SAM (Structured Agent-Managed) system adds a fourth layer: YAML-based task plans with dependency graphs, atomic claiming, T0/TN bookend verification, and session-scoped active-task parking. GitHub Issues is currently the authoritative source of truth.

Why integrate? Teams using beads for issue tracking should not need to maintain a parallel GitHub Issues mirror. A beads backend eliminates the round-trip to GitHub (with its rate limits, authentication overhead, and network dependency), enables offline-first workflows, and lets agents operate on a richer native data model (typed dependencies, metadata blobs, DAG visualization, atomic claiming) without shimming those concepts through GitHub labels and issue-body conventions.

Chosen architecture — Vision C (Universal Shim): BeadsBackend implements the existing BacklogBackend Python Protocol via bd CLI subprocess calls. Agents continue using the same backlog_* MCP tools or the new dh-backlog CLI wrapper. The bd binary is an implementation detail inside the backend — no agent instruction rewrite is needed. Backend selection follows existing DH precedence: BACKLOG_BACKEND env var → backend.toml → .beads/ auto-detect (new) → GitHub default.

What is immediately buildable: The core BeadsBackend class (Phase 1) can implement 18 of 38 MCP tools with direct or adapted mappings. Backlog CRUD (backlog_add, list, view, update, close, resolve), comments, labels, and read-only GitHub tools all map cleanly to bd subcommands with JSON output. What needs bridging: Artifact management (no beads equivalent — needs filesystem + bd remember), dispatch/wave orchestration (needs a BeadsDispatchAdapter using beads molecules), SAM active-task parking (needs bd remember convention), and structured acceptance criteria with executable check commands (needs beads notes JSON blob storage).

2. Open Decisions Required

⚠️

Architectural Forks — Human Decision Required BEFORE Implementation Begins

The following decisions have multiple valid approaches. Each fork affects the implementation design of Phases 2–4. Resolve these before writing any code beyond Phase 1.

DECISION 1 Plan ↔ Beads Representation

How should SAM plans map to beads issues?

Option A: Epic + Sub-issues

Plan = beads epic (-t epic). Each task = child bead via --parent. Native hierarchical IDs (bd-abc.1, bd-abc.2). DAG via bd dep add. No YAML files.

Option B: Plans Stay YAML + Leaf Tasks as Beads

Plan = YAML on disk (existing format). Only leaf implementation tasks become beads issues (bd create --type=task). Metadata bead links YAML plan to beads. Less migration risk.

DECISION 2 Artifact Storage Backend
Option A: Beads notes JSON blob

Store artifact metadata in bd update bd-NNN --metadata '{"artifacts":[...]}'. Content stored in issue notes field. Native to beads, but large content is awkward in notes.

Option B: Filesystem + bd remember

Artifacts stay in .dh/artifacts/{slug}/. Path recorded via bd remember "artifact:{name}:{path}" --key artifact-{id}. Works offline, survives beads re-inits.

DECISION 3 sam_active_task Parking
Option A: bd remember per-session

bd remember "active-task: bd-NNN" --key active-task-{session_id}. Persists across session boundaries. Cleared by bd forget on SubagentStop hook.

Option B: Local JSON file (existing pattern)

Keep existing ~/.dh/projects/{slug}/context/active-task-{session_id}.json format. No change to SubagentStop hook. Beads backend stores same JSON, different bead ID format.

DECISION 4 T0/TN Bookend Result Storage

Beads has no verification artifact type. Options: (A) filesystem files at .dh/bookends/{slug}/ with path recorded in bd remember, or (B) full YAML stored in beads issue notes JSON blob — searchable but size-limited.

DECISION 5 SAM Dispatch/Wave State

Options: (A) Dedicated beads formulas as wave containers using bd mol bond for wave member linking — native beads concept, requires formula authoring. (B) Keep existing YAML dispatch plan files + beads tracking issues for wave status — minimal change, familiar pattern.

DECISION 6 CLI Wrapper Scope

Options: (A) Thin passthrough — dh-backlog reads backend config and delegates 1:1 to MCP tool parameters, returning raw tool output. (B) Full routing shim with backend selection logic embedded in the wrapper, aware of both bd commands and GitHub API calls.

7. Gaps Requiring Bridges

📦
BRIDGE 1 Artifact Management

Beads has no artifact concept. DH stores artifact manifests in GitHub Issue body HTML comments and content in GitHub Issue comments.

Bridge: Create BeadsArtifactManager outside the BacklogBackend Protocol. Artifacts stored in .dh/artifacts/{slug}/{type}/ on the filesystem. Manifest entries stored as bd remember memories with key pattern artifact:{issue_id}:{type}:{artifact_id}. MCP server routes artifact_* calls to BeadsArtifactManager when backend == beads. Cross-worktree access: paths must be absolute or relative to a shared mount point.

bd remember "artifact:{type}:{path}:{status}" --key "art-{issue}-{type}"
bd recall art-bd123-architect    # retrieve artifact metadata
# File content at: .dh/artifacts/{slug}/{type}/{filename}
🌊
BRIDGE 2 Dispatch / Wave Orchestration

DH dispatch is GitHub-milestone-centric: plans reference milestone numbers, staleness check queries GitHub. Beads has no milestone container object — milestones are issues.

Bridge: Implement BeadsDispatchAdapter. Wave containers use beads molecule epics (bd create -t epic --label sam-wave-N). Wave membership via bd mol bond A B --type parallel. Dispatch plan YAML remains on disk unchanged (format compatible). dispatch_stale_check queries bd list --parent bd-MILESTONE --status open --json instead of GitHub milestone API. Wave state stays in existing SQLite dispatch-state.db (backend-independent).

🎯
BRIDGE 3 Active Task Parking

Beads has no per-session active-task concept. DH's SubagentStop hook reads active-task-{session_id}.json directly from the filesystem.

Bridge: Keep existing filesystem JSON pattern (~/.dh/projects/{slug}/context/active-task-{session_id}.json). BeadsContextBackend implements the same ContextBackend Protocol but stores a beads bead ID (bd-NNN) in the task_id field instead of a SAM task ID (T3). SubagentStop hook unchanged — it reads the JSON file, then calls the SAM MCP which delegates to BeadsTaskBackend to update status via bd update --status closed. Alternatively use bd remember for cross-machine session continuity.

✅
BRIDGE 4 Structured Acceptance Criteria with check_command

Beads acceptance_criteria is free-form text. DH SAM has structured AC with criterion-id, description, check-command, expected-baseline, expected-final fields.

Bridge: Store structured AC as JSON in the beads metadata blob: bd update bd-NNN --metadata '{"acceptance_criteria_structured":[...]}'. Free-form AC stays in --acceptance field for human readability. T0/TN agents read structured AC from bd show bd-NNN --json | jq '.metadata.acceptance_criteria_structured'.

📸
BRIDGE 5 T0/TN Bookend Lifecycle

Beads has no verification artifact type. T0 captures pre-implementation baseline; TN proves acceptance criteria were met.

Bridge: T0 and TN tasks become beads issues with labels sam-bookend-t0 / sam-bookend-tn. T0 task: dependencies:[], priority:0 (maps to beads P0). TN task: depends on all non-bookend issues. Result YAML files stored at .dh/bookends/{slug}/, path registered via bd remember "bookend:t0:{slug}:{path}". complete-implementation skill reads path via bd recall.

👥
BRIDGE 6 Multi-Assignee

GitHub Issues supports assignees[] (multiple users). Beads has a single assignee string field.

Bridge: Set primary assignee as bd update bd-NNN --assignee alice. Additional assignees stored in metadata.co_assignees: ["bob","carol"]. DH currently uses single assignee in practice — this gap is low priority.

🏷️
BRIDGE 7 Label / Tag Taxonomy

GitHub has colored labels with descriptions. DH uses label conventions for priority (priority:p1), type (type:bug), status (status:in-progress). Beads has bare-string labels with no color/description.

Bridge: Map DH label conventions directly to beads labels as strings. Priority is native in beads (numeric field), so priority:p1 label becomes --priority 1 flag. Type and status are also native beads fields. Labels in beads are used for custom DH metadata (sam-groomed, sam-ready, sam-bookend-t0). Response objects fill color="#888888" and description="" as defaults.

🔢
BRIDGE 8 GitHub Issue Numbers vs Beads IDs

DH uses #NNN GitHub integer issue numbers as canonical IDs throughout (selectors, artifact references, dispatch plans). Beads uses bd-NNN nanoid format.

Bridge: When backend=beads, use bd-NNN as the canonical ID. Store bd_id in local backlog cache frontmatter (metadata.bd_id: "bd-a1b2"). MCP tool selectors that accept #N or integer also accept bd-NNN via BeadsBackend.resolve_selector(). Dispatch plan YAML uses bd-NNN in the issue field instead of integer. Agent prompts generated by dispatch_spawn reference bd-NNN instead of #N.

9. CLI Wrapper Specification

The dh-backlog and dh-sam CLIs are thin Python PEP 723 scripts in plugins/development-harness/scripts/. They accept the same parameters as the MCP tools, route to the correct backend, and return JSON or formatted text. When BACKLOG_BACKEND=beads (or auto-detected), dh-backlog backlog_add calls bd create internally.

dh-backlog — Backlog Operations

# Add a backlog item
dh-backlog backlog_add \
  --title "Fix login bug" \
  --description "OAuth token refresh fails" \
  --priority P1 \
  --type Bug

# List open items
dh-backlog backlog_list \
  --status open \
  --section P1

The router is transparent to callers. The same dh-backlog backlog_add command works regardless of backend — output shape is identical. Only the underlying storage and ID format differ (bd-NNN vs #NNN).

Script Header (PEP 723)

#!/usr/bin/env -S uv run --quiet
# /// script
# requires-python = ">=3.11"
# dependencies = [
#   "typer>=0.12",
#   "fastmcp>=3.1",
#   "rich>=13.0",
# ]
# ///
"""dh-backlog — Thin CLI wrapper for DH backlog MCP tools.

Routes to BeadsBackend or GitHubBackend based on backend config.
Usage: uv run scripts/dh_backlog_router.py backlog_add --title "..." --priority P1
"""
import json
import os
import subprocess
import typer
from typing import Optional

app = typer.Typer(name="dh-backlog", help="DH Backlog CLI")

def _mcp_call(tool: str, **kwargs) -> dict:
    """Call a backlog MCP tool via fastmcp."""
    args = json.dumps({k: v for k, v in kwargs.items() if v is not None})
    result = subprocess.run(
        ["uv", "run", "fastmcp", "call",
         "--command", "uv run --script plugins/development-harness/scripts/run_backlog_server.py",
         tool, args],
        capture_output=True, text=True, check=True
    )
    return json.loads(result.stdout)

@app.command()
def backlog_add(
    title: str = typer.Option(..., help="Item title"),
    description: str = typer.Option(..., help="Item description"),
    priority: str = typer.Option("P2", help="P0|P1|P2|Ideas"),
    type_: str = typer.Option("Feature", "--type", help="Feature|Bug|Refactor|Docs|Chore"),
) -> None:
    result = _mcp_call("backlog_add",
        title=title, description=description,
        priority=priority, type=type_,
        gate_token="problems-not-solutions"
    )
    typer.echo(json.dumps(result))
