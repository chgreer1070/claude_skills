01 · Executive Summary

What's being integrated and why

Beads (bd CLI) is a Dolt-powered distributed graph issue tracker with 70+ commands, native JSON output on virtually every command, dependency-aware DAG execution, a molecule/formula workflow system, multi-assignee claim semantics, a persistent memory system, and bidirectional sync with GitHub, Linear, Jira, GitLab, ADO, and Notion. Its data model is significantly richer than GitHub Issues: first-class dependency typing (19 built-in edge types), custom statuses and types, structured metadata JSON blobs on every issue, async gate primitives, and Dolt-based version-controlled storage with federation.

Development-Harness (DH) is a Claude Code plugin that manages software project backlogs via GitHub Issues. It exposes a FastMCP server with three logical subsystems: backlog CRUD tools (backlog_*), artifact manifest management (artifact_*), and dispatch/wave orchestration (dispatch_*). The SAM (Structured Agent-Managed) system adds a fourth layer: YAML-based task plans with dependency graphs, atomic claiming, T0/TN bookend verification, and session-scoped active-task parking. GitHub Issues is currently the authoritative source of truth.

Why integrate? Teams using beads for issue tracking should not need to maintain a parallel GitHub Issues mirror. A beads backend eliminates the round-trip to GitHub (with its rate limits, authentication overhead, and network dependency), enables offline-first workflows, and lets agents operate on a richer native data model (typed dependencies, metadata blobs, DAG visualization, atomic claiming) without shimming those concepts through GitHub labels and issue-body conventions.

Chosen architecture: Vision C (Universal Shim). BeadsBackend implements the existing BacklogBackend Python Protocol via bd CLI subprocess calls. Agents continue using the same backlog_* MCP tools or the new dh-backlog CLI wrapper. The bd binary is an implementation detail inside the backend; no agent instruction rewrite is needed. Backend selection follows existing DH precedence: BACKLOG_BACKEND env var → backend.toml → .beads/ auto-detect (new) → GitHub default.

What is immediately buildable: The core BeadsBackend class (Phase 1) can implement 18 of 38 MCP tools with direct or adapted mappings. Backlog CRUD (backlog_add, list, view, update, close, resolve), comments, labels, and read-only GitHub tools all map cleanly to bd subcommands with JSON output. What needs bridging: Artifact management (no beads equivalent; needs filesystem + bd remember), dispatch/wave orchestration (needs a BeadsDispatchAdapter using beads molecules), SAM active-task parking (needs bd remember convention), and structured acceptance criteria with executable check commands (needs beads notes JSON blob storage).

02 · Open Decisions Required

Architectural Forks: Human Decision Required Before Implementation

The following decisions have multiple valid approaches. Each fork affects the implementation design of Phases 2–4. Resolve these before writing any code beyond Phase 1.

Decision 1Large

Plan ↔ Beads Representation

How should SAM plans map to beads issues?

Option A: Epic + Sub-issues

Plan = beads epic (-t epic). Each task = child bead via --parent. Native hierarchical IDs (bd-abc.1, bd-abc.2). DAG via bd dep add. No YAML files.

Option B: Plans Stay YAML + Leaf Tasks as Beads

Plan = YAML on disk (existing format). Only leaf implementation tasks become beads issues (bd create --type=task). Metadata bead links YAML plan to beads. Less migration risk.

Decision 2Large

Artifact Storage Backend

Option A: Beads notes JSON blob

Store artifact metadata in bd update bd-NNN --metadata '{"artifacts":[...]}'. Content stored in issue notes field. Native to beads, but large content is awkward in notes.

Option B: Filesystem + bd remember

Artifacts stay in .dh/artifacts/{slug}/. Path recorded via bd remember "artifact:{name}:{path}" --key artifact-{id}. Works offline, survives beads re-inits.

Decision 3Medium

sam_active_task Parking

Option A: bd remember per-session

bd remember "active-task: bd-NNN" --key active-task-{session_id}. Persists across session boundaries. Cleared by bd forget on SubagentStop hook.

Option B: Local JSON file (existing pattern)

Keep existing ~/.dh/projects/{slug}/context/active-task-{session_id}.json format. No change to SubagentStop hook. Beads backend stores same JSON, different bead ID format.

Decision 4Medium

T0/TN Bookend Result Storage

Beads has no verification artifact type.

Option A: Filesystem files

Files at .dh/bookends/{slug}/ with path recorded in bd remember.

Option B: YAML in beads notes

Full YAML stored in beads issue notes JSON blob — searchable but size-limited.

Decision 5Medium

SAM Dispatch/Wave State

Option A: Beads formula wave containers

Dedicated beads formulas as wave containers using bd mol bond for wave member linking — native beads concept, requires formula authoring.

Option B: YAML dispatch plan files

Keep existing YAML dispatch plan files + beads tracking issues for wave status — minimal change, familiar pattern.

Decision 6Small

CLI Wrapper Scope

Option A: Thin passthrough

dh-backlog reads backend config and delegates 1:1 to MCP tool parameters, returning raw tool output.

Option B: Full routing shim

Backend selection logic embedded in the wrapper, aware of both bd commands and GitHub API calls.

SAM Gap Analysis: What Beads Lacks Natively

Not in Beads (Needs Bridge)

  • Plan drafting state — use label convention
  • Session-scoped active task — keep JSON file or use bd remember
  • Structured AC with check-command — store in metadata JSON
  • T0/TN bookend lifecycle — filesystem + memory pointer bridge
  • Downstream cascade on failure — implement in BeadsTaskBackend
  • Autonomy mode gating — epic metadata + orchestrator logic

Better in Beads Than GitHub

  • Atomic task claiming — bd update --claim is natively atomic
  • DAG readiness — bd ready computes this natively
  • Dependency graph — 19 typed edge types vs GitHub convention hacks
  • Priority system — native P0-P4 numeric vs label convention
  • Offline-first — Dolt embedded, no network required
  • Metadata JSON — arbitrary structured data on every issue

07 · Gaps

Gaps Requiring Bridges

Eight capabilities DH relies on have no direct beads equivalent. Layer chips show which plugin layers each bridge affects.

Bridge 1Large

Artifact Management

apiMCP Scripts terminalCLI

Beads has no artifact concept. DH stores artifact manifests in GitHub Issue body HTML comments and content in GitHub Issue comments.

Bridge: Create BeadsArtifactProvider implementing the ArtifactBackend Protocol defined in backlog_core/artifact_provider.py (the same Protocol used by GitHubGistArtifactProvider, LinearArtifactProvider, GitLabArtifactProvider). Register via create_artifact_provider() factory — no MCP server routing change needed. Storage approach depends on Decision 2: (A) beads metadata blob or (B) filesystem at .dh/artifacts/{slug}/{type}/ + bd remember manifest index. Cross-worktree access: paths must be absolute or relative to a shared mount point.

09 · CLI Wrapper

CLI Wrapper Specification

The dh-backlog and dh-sam CLIs are thin Python PEP 723 scripts in plugins/development-harness/scripts/. They accept the same parameters as the MCP tools, route to the correct backend, and return JSON or formatted text. When BACKLOG_BACKEND=beads (or auto-detected), dh-backlog backlog_add calls bd create internally.

dh-backlog — Backlog Operations

# Add a backlog item
dh-backlog backlog_add \
  --title "Fix login bug" \
  --description "OAuth token refresh fails" \
  --priority P1 \
  --type Bug

# List open items
dh-backlog backlog_list \
  --status open \
  --section P1

# View a specific item (by title, bd-ID, or #N)
dh-backlog backlog_view \
  --selector "Fix login bug"

dh-backlog backlog_view \
  --selector bd-a1b2

# Update item status
dh-backlog backlog_update \
  --selector "Fix login bug" \
  --status in-progress

# Close (dismiss) an item
dh-backlog backlog_close \
  --selector "Fix login bug" \
  --reason wontfix

# Resolve (complete) an item
dh-backlog backlog_resolve \
  --selector "Fix login bug" \
  --summary "Fixed OAuth token refresh timing issue"

# Groom an item section
dh-backlog backlog_groom \
  --selector "Fix login bug" \
  --section "Acceptance Criteria" \
  --content "- Login completes within 2s"

# Add comment
dh-backlog backlog_comment_issue \
  --issue-number bd-a1b2 \
  --body "Investigating the token refresh race condition"

Script Scaffold

# /// script
# requires-python = ">=3.11"
# dependencies = [
#   "fastmcp>=3.1",
#   "rich>=13.0",
# ]
# ///
"""dh-backlog: Thin CLI wrapper for DH backlog MCP tools.

Routes to BeadsBackend or GitHubBackend based on backend config.
Usage: uv run scripts/dh_backlog_router.py backlog_add --title "..." --priority P1
"""
import json
import os
import subprocess
import typer
from typing import Optional

app = typer.Typer(name="dh-backlog", help="DH Backlog CLI")

def _mcp_call(tool: str, **kwargs) -> dict:
    """Call a backlog MCP tool via fastmcp."""
    args = json.dumps({k: v for k, v in kwargs.items() if v is not None})
    result = subprocess.run(
        ["fastmcp", "call", tool, "--args", args],
        capture_output=True, text=True, check=True
    )
    return json.loads(result.stdout)

@app.command()
def backlog_add(
    title: str = typer.Option(..., help="Item title"),
    description: str = typer.Option(..., help="Item description"),
    priority: str = typer.Option("P2", help="Priority: P0-P4 or P0-P4"),
    type: Optional[str] = typer.Option(None, help="Item type: Feature|Bug|Chore"),
):
    """Create a new backlog item."""
    result = _mcp_call("backlog_add", title=title, description=description,
                       priority=priority, type=type)
    print(json.dumps(result, indent=2))

if __name__ == "__main__":
    app()
